笔记：K8S

一、理论

1. 导图：Map：K8S

2. K8S 概述

K8S 是 Kubernetes 的缩写（中间的 “8” 代表 “ubernete” 这 8 个字母），它是目前最流行的容器编排系统，用于自动化部署、扩展和管理容器化应用（如 Docker 容器）

3. K8S 最佳实践

3.1. 最佳实践综述

  1. ==主从架构设计==
    1. Kubernetes 采用主从架构，Master 节点负责全局调度与决策，而 Worker 节点则执行具体任务。
    2. Master 节点：
      1. API Server：
        1. 处理 kubectl 命令：
          1. 接收并处理所有 REST 请求（如 kubectl apply、kubectl get 等一系列 kubectl 命令），提供统一的 HTTP/HTTPS 接口
          2. 对于 REST 请求，管理员可以直接在终端使用 kubectl 命令操作；
          3. 对于 HTTP/HTTPS 请求，也可以在 Pod 内或通过浏览器调用 kubectl，在这种情况下，请求必须携带 Token。
        2. 进行认证：（☆☆☆）
          1. 根据执行 kubectl 命令时传递的 Service Account（Pod 内部执行命令）或 kubeconfig 自动附加的用户证书（终端执行命令），进行用户身份认证。
          2. 如果认证通过，则继续执行后续操作（如鉴权、操作 etcd、推送变更等）；
          3. 如果认证失败，则返回 401 错误。
        3. 进行鉴权：（☆☆☆）
          1. 基于我们为 Service Account 或用户证书，所使用 RBAC 所绑定的用户权限，判断他是否有权限进行这个 kubectl 操作，如创建 Pod，获取 Pod 信息等
          2. 如果没有该权限，返回 403 错误
        4. 与 etcd 交互：
          1. API Server 对 etcd 进行增删改查操作，例如读取 Deployment 配置或更新 Service 状态
          2. 在进行写入或修改操作（例如更新 Pod、Service 或 Controller 配置）时，API Server 会先将配置转换成 JSON 格式，然后写入 etcd
        5. 推送变更：
          1. 当 etcd 中的数据发生变更时，API Server 通过 Watch 机制向 Controller Manager、Scheduler 和 Kubelet 推送事件。
          2. 需要注意的是，API Server 不会主动”通知组件”，而是组件主动监听 API Server 的资源变更。
      2. Scheduler：
        1. 监听 API Server，检测是否有待调度的 Pod。
        2. 如果存在待调度 Pod，Scheduler 根据资源需求（如 CPU/内存）、亲和性（nodeAffinity）、污点（Taint）等策略选择目标 Worker 节点，更新 Pod 的 nodeName 字段，并将变更回写至 etcd。
      3. Controller Managment：
        1. 负责运行和管理各类 Controller，确保集群实际状态与用户预期保持一致
      4. etcd：
        1. etcd 是一个分布式键值数据库，存储集群中所有的配置和状态数据，并在 Master 节点间进行数据同步
      5. 注意事项：
        1. 虽然每个 Master 节点上都有 API Server、Scheduler、Controller Manager 和 etcd 组件，但集群中仅由一个选举出的 Leader Master 负责实际的写入和调度操作，以确保操作的唯一性和正确性。
        2. 与 etcd 交互并不一定需要通过 Leader Master，因为所有 Master 节点上的 API Server 组件都可以与 etcd 进行交互。
    3. Worker 节点：
      1. Kubelet：
        1. Kubelet 会监听 API Server，当 Scheduler 选定某个 Worker 节点运行 Pod 后，该 Worker 节点上的 Kubelet 会从 API Server 接收 Pod 定义或更新信息，并负责创建容器
        2. Kubelet 持续监控容器状态，并将状态上报给 API Server。
        3. 同时，Kubelet 会执行存活探针和就绪探针，以确保容器的健康状态和可用性。
        4. 需要注意的是，如果我们使用 kubeadm 部署 K8S 集群，Kubelet 会自动配置 API Server 的地址，无需担心 Kubelet 找不到 API Server
      2. Kube-proxy：
        1. 负责维护节点网络规则，实现 Service 的 IP 和端口映射，确保 Pod 间以及 Service 与 Pod 之间的网络互通
  2. ==多主节点之间的联系==：
    1. 多个主节点之间通过 etcd 实现数据同步，保证每个主节点都能及时获取到集群的最新状态信息。
    2. 如果我们使用 kubeam 部署 K8S 集群，他会自动在主节点部署 etcd，这个我们无需担心
  3. ==负载均衡策略==：
    1. Kubernetes 本身是会为部署的 Pod 提供负载均衡功能，因此不需要在外部额外为 Kubernetes 集群配置负载均衡。
  4. ==高可用性实现==：
    1. 主节点之间能实现故障转移，在多个主节点中，会通过 Raft 算法选举出一个 Leader，只有 Leader 的各项组件（如 API Server、Scheduler 等）真正起作用，其他主节点的组件不起作用。这主要是为了避免多个主节点同时操作 etcd，重复调度 Pod，导致冲突。
    2. 如果当前 Leader 挂掉，其它主节点会自动发起选举，选出新的 Leader 来接管工作。
  5. ==最佳实战架构==：
    1. 在 Kubernetes 中，实战架构主要分为三类：用户访问应用、管理员执行命令（通过终端） 和 管理员执行命令（Pod 内部），具体说明如下：
    2. 用户访问应用（通过浏览器）：
      1. 用户请求：
        1. 用户在浏览器中输入请求地址，例如 blog.wagnjia.xin。
      2. 公共 DNS 服务器处理：
        1. 请求首先到达公共 DNS 服务器，DNS 将域名解析为由 MetalLB 分配给 Ingress Controller Service 的 VIP 地址
      3. 请求到达 Ingress Controller Service：
        1. 用户请求通过解析得到的 VIP 地址被路由到 Ingress Controller Service。
      4. 请求到达 Ingress Controller Pod：
        1. Ingress Controller Service 根据负载均衡策略，将请求分发到某个 Ingress Controller Pod 实例
      5. 请求到达 应用 Service：
        1. Ingress Controller Pod 会按照 Ingress 资源配置的域名规则，将请求发送到相应的 应用 Service
      6. 请求到达 应用 Pod：
        1. 应用 Service 接受到请求后，会把请求负载均衡到一个其管理的应用 Pod 实例
      7. 应用 Pod 完成业务处理：
        1. 然后，将由应用 Pod 完成真正的业务处理
    3. 管理员执行命令（通过终端）：
      1. 管理员执行命令：
        1. 管理员在任意一个 Master 节点的终端执行 kubectl 命令，kubectl 根据 kubeconfig 配置文件中的 API server 字段，定位 Kubernetes 集群的 API Server 地址（此地址通常配置为 Keepalived VIP，该 VIP 绑定在 HAProxy 服务器上）。
        2. 同时 kubeconfig 还会自动将认证信息（用户证书）随命令发送过去用于身份验证（☆☆☆）
        3. 注意事项： kubeconfig 为集群级配置，无需为每个 Master 节点单独修改。
      2. 命令到达 HAProxy：
        1. 命令通过配置的 VIP 到达 HAProxy 服务器。
      3. 命令到达 Master 节点：
        1. HAProxy 通过负载均衡将请求转发至任一 Master 节点，避免单节点过载。
      4. Master 节点的 API Server 处理命令：
        1. 目标 Master 节点的 API Server 验证用户证书并检查权限。
        2. 若验证成功且具有此权限，API Server 会处理 kubectl 命令。
        3. 注意事项：所有 Master 节点的 API Server 均为无状态服务，都可以独立处理请求，无需依赖 Leader Master
      5. API Server 与 etcd 交互：
        1. API Server 将请求的操作（如资源创建、更新）通过 etcd 客户端库写入到 etcd 分布式数据库集群
      6. 组件响应 etcd 变更：
        1. Leader Master 上的 Scheduler、Controller Manager 等组件监听 etcd 变更并执行相应操作
        2. 同时，Worker 节点上的 Kubelet 也会监听 etcd 变更并执行对应操作
    4. 管理员执行命令（Pod 内部）
      1. 管理员执行命令：
        1. 管理员在 Pod 内部执行 kubectl 命令，默认通过 https://kubernetes.default.svc:443 访问随机 Master 节点的 API Server
        2. 同时，还需手动将 Pod 挂载的 Service Account 随命令发送过去用于身份验证（☆☆☆）
        3. 注意事项：通过 Kubernetes Service kubernetes.default.svc 访问 API Server，该 Service 由 kube-apiserver 的 Endpoints 维护，天然支持多 Master 负载均衡。
      2. 命令到达 Master 节点：
        1. 命令通过 Kubernetes DNS URL 访问 Master 节点。
      3. Master 节点的 API Server 处理命令：
        1. 目标 Master 节点的 API Server 验证 Service Account 并检查权限
        2. 若验证成功且具有此权限，API Server 会处理 kubectl 命令
      4. API Server 与 etcd 交互：
        1. API Server 将请求的操作（如资源创建、更新）通过 etcd 客户端库写入到 etcd 分布式数据库集群
      5. 组件响应 etcd 变更：
        1. Leader Master 上的 Scheduler、Controller Manager 等组件监听 etcd 变更并执行相应操作
        2. 同时，Worker 节点上的 Kubelet 也会监听 etcd 变更并执行对应操作

4. K8S 核心概念：Pod

5. K8S 核心概念：Controller

5.1. Controller 引入的原因

在前面的 Pod 实战步骤中，我们虽然成功启动了 Pod，但整个过程是手动管理的。如果某个 Pod 发生故障，我们需要手动检测并重新启动，这不仅繁琐，还影响效率。而 Controller 的作用正是自动管理 Pod，确保它们的运行状态符合预期，从而减少人为干预。

需要注意的是，Scheduler 负责将 Pod 分配到具体的 Worker 节点，而 Controller 负责管理 Pod 的生命周期：在 Pod 故障时自动重启，在节点故障时触发迁移。Ingress 负责将其转发到 Service，而 Service 负责将请求转发到 Pod。

5.2. Controller 的类型

Controller 主要分为以下几类：

  1. ==ReplicationController（过时）==
    1. ReplicationController 可确保指定数量的 Pod 始终保持运行，即使某些 Pod 发生故障，也会自动补充。同时，它支持手动调整 Pod 副本数量，方便进行扩容或缩容
    2. 然而，该过滤器已过时，不建议在新项目中使用。原因在于它仅支持基于 等值匹配（equality-based） 的标签选择器（如 app=my-nginx），而无法兼容集合匹配（set-based） 选择器（如 app in (nginx, web)）
  2. ==ReplicaSet==
    1. ReplicaSet 是 ReplicationController 的升级版，提供了更强的标签匹配功能（如支持基于集合的 Label Selector），比 RC 更灵活
    2. ReplicaSet 很少单独使用，通常是作为 Deployment 的一部分，由 Deployment 来管理
  3. ==Deployment==
    1. Deployment 通过管理 ReplicaSet，负责 Pod 的创建、删除和更新。当你创建一个 Deployment 时，实际上他会自动创建并管理一个 ReplicaSet ，Deployment 通过控制 ReplicaSet 来间接管理 Pod 的创建、删除和更新等操作。
    2. 此外，Deployment 还支持滚动更新（平稳升级，避免服务中断）、回滚、暂停/恢复更新等功能，并能与 HPA 和 Service 配合使用，实现弹性扩缩容和流量管理
  4. ==HPA==
    1. Horizontal Pod Autoscaler（HPA）用于根据 CPU 使用率、内存使用率或自定义指标自动调整 Deployment 等资源的 Pod 副本数量，以实现资源的弹性伸缩。
    2. 举例来说，若让 HPA 监控 CPU 使用率，当 CPU 使用率超出设定的阈值（如 50%）时，HPA 会自动增加 Pod 的数量，以应对更高的负载；而当 CPU 使用率低于设定阈值时，HPA 则会减少 Pod 的数量，从而节省资源。
  5. ==Deployment + HPA==
    1. 当前主流，强烈推荐，其口诀是，先创 Deployment 再创 HPA

6. K8S 核心概念：Labels

6.1. Labels 引入的原因

假设你有一个复杂的系统，包含多个微服务，每个服务有多个 Pod 实例。例如，你的系统有以下服务：web、api、database，每个服务对应多个实例：

  • web-1、web-2、web-3
  • api-1、api-2、api-3
  • db-1、db-2、db-3
    在传统的管理方式中，你可能只能通过 Pod 的名称或 ID 来定位它们，甚至可能需要进入每个 Pod 查看详细信息，这样既麻烦又低效。
    但如果你在创建 Pod 时 为它们打上标签（例如 app=web、app=api、app=db），你就能通过 标签快速筛选 出相关的资源，无需逐个查找或进入 Pod，极大地提高了效率。

[!NOTE] 注意事项：一般用到 Label 的地方

  1. HPA 通过 Selector 来筛选对应的 Controller
  2. Controller、Service 通过 Selector 来筛选对应的 Pod
  3. Pod 通过 nodeSelector 来筛选对应的 Node 节点，并将 Pod 部署到该节点上

6.2. Labels 常见的约定

尽管标签可以自定义，但我们通常将其与 Namespace 结合使用，并遵循一些常见的约定，具体如下：

  1. ==Namespace==：
    1. Namespace 用于隔离不同的环境、团队或应用，每个资源只能属于一个 Namespace。常见的约定包括：
    2. 隔离环境：
      1. namespace: env_dev：开发环境
      2. namespace: env_test：测试环境
      3. namespace: env_staging：预生产环境
      4. namespace: env_prod：生产环境
    3. 开发团队：
      1. namespace: team_tencent：腾讯团队
    4. 应用名称：
      1. namespace: app_qqrobot：QQ 机器人应用
  2. ==Labels==：
    1. Labels 用于标识资源的具体业务、版本、角色等，一个资源可以具有多个标签。常见的约定如下：
    2. 业务：
      1. bussiness: nginx：负载均衡业务
      2. bussiness: payment：支付业务
    3. 版本：
      1. version: v1.0：版本 v1.0
    4. 角色：
      1. role: batian：开发人员”霸天”

8. K8S 核心概念：K8S 调度器

8.1. 调度器 引入的原因

上面的 Namespace 仅实现了逻辑分区，但不同的 Pod 仍然可能运行在同一台主机上。而调度器的作用在于实现物理隔离，即决定 Pod 应该运行在哪台主机上，以及主机如何主动将 Pod 驱逐。

8.2. 调度器 的调度因素

调度器主要受以下三部分因素的影响：

  1. ==节点亲和度==：
    1. 调度器会主动将 Pod 分配到满足特定条件的节点上，即根据节点的特性来选择合适的运行环境。
  2. ==Pod 亲和度==：
    1. 调度器会将彼此具有亲和关系的 Pod 调度到同一台服务器上，这样可以缩短它们之间的通信延迟，提高整体性能
  3. ==污点和容忍性==：
    1. 节点不断被赋予污点，如果 Pod 没有相应的容忍度，就无法适应这种环境，最终会被节点主动驱逐。
    2. 污点（Taints）
      1. 定义在节点上，为节点“打上污点”，让它显示出不欢迎普通 Pod 的特性。
    3. 容忍度（Tolerations）
      1. 定义在 Pod 上，用来说明该 Pod 能接受节点上的特殊污点，从而允许其在带有污点的节点上运行。

8.3. K8S 调度工作流程

Kubernetes 调度器（kube-scheduler）就好比集群的“大脑”，它不断观察 API Server 的“呼救信号”，将那些还没有座位（即 PodSpec.NodeName 为空）的 Pod 逐一分配到合适的节点上。整个调度过程大致分为三个阶段：

  1. ==预选阶段（Predicates）==
    1. 在这个阶段，调度器先像严格的门卫一样过滤掉那些“不符合要求”的节点。比如：
    2. PodFitsResources：
      1. 检查节点是否有足够的资源（CPU、内存、存储等）来满足 Pod 的需求。
    3. PodFitsHostPorts：
      1. 检查节点是否存在与 Pod 端口冲突的情况，确保所请求的端口没有被其他 Pod 使用。
    4. NoDiskConflict：
      1. 检查节点上的磁盘是否存在冲突，确保没有磁盘资源冲突。
    5. CheckNodeDiskPressure：
      1. 检查节点是否因为磁盘压力（磁盘满了等问题）而不能调度 Pod。
    6. CheckNodeMemoryPressure：
      1. 检查节点是否因为内存压力而不能调度 Pod。
    7. TaintToleration：
      1. 检查节点是否有污点（taint）。如果节点上存在污点，调度器会检查 Pod 是否具有与该污点匹配的容忍度（toleration）。如果 Pod 没有匹配的容忍度，则该 Pod 将无法调度到该节点（☆☆☆）
    8. 如果所有节点都不合适，Pod 就只能站在门口等候（Pending 状态），直到有节点“空出座位”。
  2. ==优选阶段（Priorities）==
    1. 通过预选后，调度器会为剩下的节点打分，就像给各个餐桌打分：哪个桌子环境好、资源充足、服务棒？打分高的节点将更有机会获得这个 Pod 的“订单”。常见的优选策略有：
    2. LeastRequestedPriority：
      1. 资源使用越低，得分越高，就像优先安排那些桌子空闲的区域。
    3. SelectorSpreadPriority：
      1. 为了高可用，尽量把同一组 Pod 分散在不同节点上。
    4. ImageLocalityPriority：
      1. 如果某个节点上已经下载了需要的镜像，那么“就餐速度”更快，得分自然上升。
    5. NodeAffinityPriority：
      1. 根据节点亲和性打分（☆☆☆）
  3. ==Binding 阶段==
    1. 最后，调度器会选择打分最高的节点，并通过绑定操作（binding）把 Pod 安排到这个节点上，最终记录到 etcd 中。随后，该节点上的 kubelet 会监听到 etcd 的变更，并创建 Pod。

[!NOTE] 注意事项：taint-effect

  1. taint-effect 表示排斥等级，主要有三种：
    • NoSchedule：
      • 调度器不会将不具备相应容忍度的 Pod 调度到该节点，但已经在该节点运行的 Pod 不受影响。
    • PreferNoSchedule：

      • 调度器不会将不具备相应容忍度的 Pod 调度到该节点，并且还会将已在该节点运行但不满足容忍度的 Pod 驱逐出去。

9. K8S 核心概念：K8S 网络

9.1. K8S 网络 引入的原因

K8s 网络的核心作用就是实现 容器之间的通信 和 外部访问容器中的服务。更具体一点，它主要负责以下两件事：

  1. 容器间通信：K8s 让不同的容器，即便它们可能跑在不同的机器（节点）上，也能通过一个统一的网络协议来互相联系。这就像是在一个大公司的不同部门之间，大家都能打电话联系上彼此。
  2. 外部访问容器服务：通过 Service 和 Ingress，K8s 为外部流量提供了一个入口，就像是有一个大门，外部请求可以通过这个门进入到集群内部的特定容器服务。你只需要知道 Service 的地址，而不用关心具体容器在哪个节点上，K8s 会帮你自动找到并路由到对应的容器。

9.2. K8S 网络 的组成

K8S 网络主要由以下几部分组成：

  1. ==hostNetWork==：
    1. 在 Pod 中，我们可以配置 hostNetwork，该选项表示 Pod 直接使用宿主机的网络栈，而不是默认的 Kubernetes CNI 网络。这种方式可以减少网络转发的开销，从而提升访问速度
    2. 它的原理相对复杂，这里暂且不深入探讨。只需记住：虽然 hostNetwork: true 可以提升网络性能，适用于高并发的数据库等场景，但在大多数情况下，尤其是 Spring Boot 和 Spring Cloud 项目中，通常会保持默认（false），以确保网络隔离性和可移植性。
  2. ==ContainerPort 和 hostPort==：
    1. 这两种端口既可以在 Pod 内进行配置，也可以通过 Service 的 port 和 targetport 来统一配置。但为了实现统一管理和更方便的维护，强烈建议通过 Service 来管理 Pod 的端口。
    2. ContainerPort：
      1. 指容器内部应用监听的端口。
    3. hostPort：
      1. 将容器内的某个端口映射到 Pod 所在宿主机的指定端口，使外部可以通过 hostIP + hostPort 直接访问容器内的应用
  3. ==Service==：
    1. Service 提供了一个固定的入口（serviceIP + servicePort）用于访问 Pod 资源。由于 Pod 所在宿主机的 IP 可能会发生变化（例如 Pod 调度或迁移），如果直接使用 hostIP + hostPort 方式，就无法确保稳定的访问。
    2. 此外，Service 内置了负载均衡机制，自动将请求均匀分发到所有可用的 Pod。这不仅解决了只能连接到单个 Pod 的问题，还能更高效地利用集群中所有 Pod 的资源。
    3. 需要注意的是，Service IP 并不是某个具体主机的物理 IP，而是 K8S 提供的虚拟 IP（☆☆☆）
  4. ==Ingress==：
    1. 由于不同的 Service 会暴露不同的端口，若直接通过外部客户端访问 Pod，往往需要手动指定端口。而 Ingress 作为外部流量的统一入口，可以根据请求中的域名信息，将流量路由到对应的 Service，再由 Service 定位到具体的 Pod，从而免去了手动输入端口号的麻烦，相当于 Kubernetes 内部的反向代理或 API 网关。
    2. 借助 Ingress，用户访问应用时无需指定端口号。例如，访问 blog.example.com 时，无需写成 blog.example.com:9111；其他服务也可以直接通过域名访问，从而简化了访问路径并提升了用户体验。

9.5. Service 实战步骤

9.5.1. Service 的分类

Service 由以下四种类型组成：

  1. ==ClusterIP 类型==：
    1. 该类型是 Service 的默认类型，仅在集群内部暴露服务，集群内的客户端可以访问，而外部客户端无法直接访问。
    2. 内部访问：
      1. 访问 Service IP + Service Port：http://<service-ip>:<service-port>
      2. 访问 Service 内部域名：http://<service-name>.<service-namespace>.svc.cluster.local
    3. 注意事项：
      1. Service IP 即是我们调用 kubectl get service <service-name> -n <service-namespace> 获取的 Cluster IP
      2. 由于 Service 的 IP 是固定的，所以为访问 Pod 提供了稳定的入口
      3. 内部访问时，Service 会采用轮询（Round Robin）的方式，将请求均匀分发到所有功能相同的 Pod。
      4. Service IP 并不是某个具体主机的物理 IP，而是 K8S 提供的虚拟 IP（☆☆☆）
  2. ==NodePort 类型==：
    1. NodePort 类型的 Service 不仅在集群内部暴露服务，也允许集群外部客户端访问 Service 管理的 Pod。
    2. 内部访问：
      1. 访问 Service IP + Service Port：http://10.96.12.34:80
      2. 访问 Service 内部域名：http://<service-name>.<service-namespace>.svc.cluster.local
      3. 注意事项：
        1. 由于 Service 的 IP 是固定的，所以为访问 Pod 提供了稳定的入口
        2. 同样，内部访问时 Service 会自动采用轮询（Round Robin）的方式，将请求均匀分发到所有对应的 Pod。
        3. Service IP 并不是某个具体主机的物理 IP，而是 K8S 提供的虚拟 IP（☆☆☆）
    3. 外部访问：
      1. 访问 Node IP + Node Port：http://192.168.100.210:32600
      2. 在每台运行 Pod 的宿主机上，Kubernetes 会分配一个介于 30000~32767 之间的 NodePort，外部客户端通过 NodeIP + NodePort 访问服务时，请求会经过 NodeIP:NodePort -> ServiceIP:ServicePort -> Pod:PodPort
      3. 注意事项：
        1. 由于 Pod 所在 Node 的 IP 可能变化，因此直接依赖 Node IP 无法提供完全稳定的访问入口（☆☆☆）
        2. 尽管外部访问看似连接到固定的 Pod，Service 依旧会以轮询（Round Robin）的方式分发请求到所有具备相同功能的 Pod，而非固定转发到某个特定 Node
        3. 如果一台服务器上运行多个 Pod，则可能会分配多个不同的 NodePort，只要不发生 Service Port 和 NodePort 端口冲突即可。
  3. ==LoadBalancer 类型==：
    1. 此类型的 Service 特点在于，一旦创建 LoadBalancer 类型的 Service，系统会自动为其分配一个固定的 External IP 。我们可以使用此 IP 在外部访问该 Service，从而访问其下的 Pod。
    2. 该类型主要用于云环境（如阿里云、AWS、Google Cloud 等），在内网或裸机部署的 Kubernetes 集群中，可以借助 MetalLB 来模拟云环境下的负载均衡器。
    3. 内部访问：
      1. 访问 Service IP + Service Port：http://10.96.12.34:80
      2. 访问 Service 内部域名：http://<service-name>.<service-namespace>.svc.cluster.local
    4. 外部访问：
      1. 访问 External IP + Service Port：http://192.168.100.210:80
  4. ==ExternalName 类型==
    1. 该类型的 Service 相对特殊，主要用于访问 集群外部 的服务，而 不是 让外部访问集群内部的服务。
    2. 当前阶段可暂不深入学习，日后有时间再了解。

10. K8S 核心概念：K8S 存储

10.1. K8S 存储 的组成

K8S 存储由以下几部分组成：

  1. ==EmptyDir==：
    1. EmptyDir 是当 Pod 创建时，Kubernetes 会自动在 Pod 所在宿主机上为其分配一个临时目录（ /var/lib/kubelet/pods/<pod-id>/volumes/kubernetes.io~empty-dir/ ）该 Pod 内的所有容器都可以读写这个目录。
    2. 然而，一旦 Pod 挂掉或迁移，数据将不可被访问，直到 Kubelet 定期清理 /var/lib/kubelet 下的无用数据时才会被删除。
  2. ==HostPath==：
    1. HostPath 允许 Pod 内的容器访问 Pod 所在宿主机的特定目录，如果多个 Pod 在同一个 Node 上，可以共享该数据，即便 Pod 挂掉或迁移，数据仍然在宿主机上
    2. 然而，由于数据存储在宿主机上，当 Pod 迁移到其他节点时，就无法访问原节点上的数据，无法访问原宿主机上的 HostPath 存储卷，只能访问当前节点的 HostPath 路径下的数据，每个节点上的 HostPath 存储卷数据可能不同，这可能导致数据不一致。
    3. 需要特别注意的是，HostPath 存储卷不会自动创建目录，我们必须手动在宿主机上预先创建目录。如果目录不存在，可能会导致问题。
  3. ==NFS==：
    1. NFS 可以让 Pod 从远程 NFS 服务器中读写数据，可以跨多个 Pod 和 Node 共享数据
    2. 需要特别注意的是，通常情况下，NFS 不需要由 Kubernetes 管理或配置，它是一个独立的外部服务，负责数据存储和共享。
  4. ==PV/PVC==：
    1. 传统存储方式的缺陷：
      1. 传统的存储方式（HostPath、EmptyDir、NFS）需要在 Pod 配置文件中直接指定数据存储的位置，这种紧耦合的设计存在几个主要问题：
      2. 迁移困难：
        1. 当你需要切换存储方式（比如从本地存储迁移到云存储或更换存储供应商）时，就得逐个修改 Pod 的配置文件
      3. 缺乏统一管理：
        1. 这种方式无法让 Kubernetes 对存储资源进行统一调度、监控和管理，存储资源不能自动分配、回收或动态调整。存储管理和业务应用紧密绑定，一旦需求变化，维护成本会大幅上升
      4. 难满足多样化需求：
        1. 不同业务对存储性能、容量和安全性的要求各不相同。直接将存储配置硬编码在 Pod 中，就难以灵活选择最合适的存储方案
    2. PV/PVC 的优势
      1. PV/PVC 是 Kubernetes 对存储资源的统一抽象和管理机制，其优势在于实现了存储层与业务层的解耦，即便底层存储发生变化（例如从 NFS 转为云存储），应用层无需改动，管理员只需调整 PV 的配置，这样，应用只需要关注“我需要多少存储、什么权限”，而不用关心数据具体存储在哪里，从而实现了存储资源的统一抽象和灵活管理。
      2. PV（PersistentVolume）：
        1. 由管理员创建或通过动态供应自动生成，封装了底层存储（无论是 NFS、EBS、iSCSI 还是 HostPath）的具体细节。可以把它看作提前准备好的“仓库”，每个仓库都有明确的属性和存储位置
      3. PVC（PersistentVolumeClaim）：
        1. 用户提交的存储需求，就像填写一张申请单，说明需要多大容量和什么样的读写权限。Kubernetes 根据 PVC 的要求自动选择并绑定合适的 PV，实现存储层与业务层的解耦
      4. 注意事项：
        1. 使用 PV/PVC 的主要目的是将存储的细节与应用解耦，重点关注“能否使用持久化存储”，而不是“具体数据存放在哪个目录”。
  5. ==ConfigMap==：
    1. ConfigMap 用于存储非敏感的配置信息，如应用配置、环境变量和配置文件等。它可以包含配置文件、属性对，甚至 JSON 对象。设计初衷是方便多个 Pod 共享和管理配置。
    2. 如果应用的配置文件较小（例如小于 1MB），推荐使用 ConfigMap，而不是通过 NFS 挂载
  6. ==Secret==：
    1. 与 ConfigMap 不同，Secret 通过 RBAC 控制访问，用于存储敏感信息，如密码、Token 和私钥等，具有更高的安全性。

10.7. Secret 实战步骤

10.7.1. Secret 的组成

Secret 主要由以下三部分组成：

  1. ==Service Account Secret==：
    1. 当 Pod 中的应用需要动态获取 Kubernetes 资源或管理 Pod 时，使用该 Secret。
    2. Kubernetes 会自动创建一个默认的 Service Account Secret。也可以根据需求创建专门用于访问 API Server 的 Service Account Secret。该 Secret 挂载在 Pod 中每个容器 /run/secrets/kubernetes.io/serviceaccount，Pod 启动时即可使用。
    3. 需要注意的是，自动创建的 Secret 默认没有绑定任何角色，也就是说其几乎没有权限。
  2. ==Opaque Secret==：
    1. 适用于存储密码、密钥、API Key 等敏感信息。可由用户自定义创建和管理。
    2. Opaque Secret 中的数据必须以 base64 编码的形式存储，虽然不能完全防止恶意用户访问，但它能够确保数据在配置文件中是以编码的方式存储的，从而增加了一定的安全性。
    3. 需要注意的是，Base64 编码只是使其适合在文本中传输，不提供任何加密功能，任何人都可以轻松解码回原始数据。
  3. ==dockerconfigjson==：
    1. 当 Pod 需要从私有仓库（例如阿里云或 Docker Hub 私有仓库）拉取镜像时，用于存储 Docker 认证信息。

[!NOTE] 注意事项：关于访问 API Server

  1. 当使用 kubectl get pods 等命令时，kubectl 会通过本地的 Kubeconfig 文件自动附加认证信息，然后向 API Server 发起请求
  2. 但是Pod 内部的应用若要访问 API Server，必须手动提供 Token 进行身份验证。如果对应的 Service Account 具有权限，则能获取到 Pod 信息；反之，将返回 403 错误。

三、补充

1. 操作符 operator

operator 字段是用来判断的操作符，常见有以下几种：

  1. **Equal**：
    • 判断键值对是否相等。
    • 适用于您需要完全匹配键和值的情况。
    • 示例：key: "key1", operator: "Equal", value: "value1"
  2. **In**：
    • 判断值是否在指定的集合中。
    • 适用于多个可能值的容忍度判断。
    • 示例：key: "key1", operator: "In", values: ["value1", "value2"]
  3. **NotIn**：
    • 判断值是否不在指定的集合中。
    • 适用于您需要排除某些值的情况。
    • 示例：key: "key1", operator: "NotIn", values: ["value1", "value2"]
  4. **Exists**：
    • 只检查键是否存在，而不关心它的值。
    • 适用于您只关心键的存在性，而不关心具体的值。
    • 示例：key: "key1", operator: "Exists" （不需要指定 value）
