源码：LockSupport 源码解析

Last updated on 2025-09-02T09:37:40+08:00

1. 概述

LockSupport 是一个具体类，是整个 AQS 阻塞/唤醒机制的基石，几乎所有同步器的 “挂起线程/唤醒线程” 都是靠它完成的。

之所以不使用 Object 的 wait/notify/notifyAll，是因为在 Object 中，这些方法必须在 synchronized 块里调用，因为调用线程必须持有该对象的锁

而 LockSupport 没这个限制，任何线程在任何地方都能 park()/unpark()，这让 AQS（同步器）不需要依赖 synchronized，而是用 CAS + LockSupport 就能造出高性能的同步器。

4.1.1. private static void setBlocker(Thread t, Object arg)

private static void setBlocker(Thread t, Object arg) 用于 在一个线程即将因为 LockSupport.park 挂起前，把 “阻塞原因” 对象（arg）写到该 Thread 对象的内部字段 parkBlocker 上。

需要注意的是，由于该方法是 private static，所以作为应用层开发者，你是不能直接用它的，不管它是不是 static，它其实是 park 的小助手

4.1.2. public static void setCurrentBlocker(Object blocker)

public static void setCurrentBlocker(Object blocker) 与 private static void setBlocker(Thread t, Object arg) 类似，只不过是作用于当前线程

需要注意的是，该方法可以被应用层开发者使用，并且通常在线程阻塞前设置。因为如果你等到真正进入阻塞方法才设置 blocker，就会有一点小时间差，调试工具在那微小的窗口里看到线程挂起了，却没有任何解释，信息就不完整了。

4.1.3. public static Object getBlocker(Thread t)

public static Object getBlocker(Thread t) 用于 获取某线程的阻塞原因对象

4.1.4. public static void park()

public static void park() 用于 让当前线程进入阻塞状态（WAITING）直到：

  1. 被中断
    1. 不同于 Thread 的 sleep 和 Object 的 wait 不同，线程在阻塞期间若被中断，当进入可运行态并重新获得 CPU 时间片后，会主动响应中断，并抛出 InterruptedException。
    2. 而 park 的线程在阻塞期间若被中断，当进入可运行态并重新获得 CPU 时间片后，不会响应中断
  2. 被 unpark
    1. 不同于 Thread 的 sleep 和 Object 的 wait 不同，unpark 的线程可用被提前唤醒（即 unpark 可以在 park 之前就调用）
  3. 线程虚假唤醒

4.1.5. public static void parkNanos(long nanos)

public static void parkNanos(long nanos) 和 public static void park() 类似，只不过让当前线程进入的阻塞状态是 TIMED_WAITING，等待 nanos 纳秒，超时后线程自动进入可运行态

4.1.6. public static void parkUntil(long deadline)

public static void parkUntil(long deadline) 和 public static void parkNanos(long nanos) 类似，让当前线程进入的阻塞状态是 TIMED_WAITING，等待到指定时间点，超时后线程自动进入可运行态

需要注意的是，传入的参数是 Unix 毫秒级时间戳，例如 System.currentTimeMillis() + 3000 是暂停三秒

4.1.7. public static void park(Object blocker)

public static void park(Object blocker) 和 public static void park() 类似，只不过 设置了阻塞原因对象
