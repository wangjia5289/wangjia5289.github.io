1.2. 核心流程详解

1.2.1. DisableEncodeUrlFilter

在早期的 Servlet 规范中有这样一项功能：如果浏览器不支持 Cookie，那么服务器只能通过在 URL 中附加 Session ID 来维持会话。

因此 Servlet API 提供了 response.encodeURL("xxx") 和 response.encodeRedirectURL("xxx") 这类方法，用来在检测到客户端不支持 Cookie 时自动将 ;jsessionid=xxxx 拼接到 URL 后，以保证 HttpSession 跟踪能够正常进行

然而在今天，几乎所有浏览器都支持 Cookie，并且裸露的 jsessionid 不但影响美观，还可能泄露带来安全风险。

为了解决这一点，DisableEncodeUrlFilter 会将 HttpServletResponse 包装成一个代理对象，使得所有对 encodeURL(...) 和 encodeRedirectURL(...) 的调用都直接返回原始 URL，而不会再拼接 ;jsessionid=xxxx，从而彻底禁用 URL 重写功能

1.2.2. WebAsyncManagerIntegrationFilter

在 Spring Security 中，当前用户信息（Authentication）会被保存在本请求所在线程的 SecurityContextHolder 中（SecurityContextHolder/SecurityContext/Authentication），而 SecurityContextHolder 默认是基于 ThreadLocal 的实现。也就是说，它只在同一个线程内有效

当你使用异步请求的时候，请求的处理逻辑可能会切换到一个新的线程。但是，由于 ThreadLocal 无法跨线程传递，这就导致了一个常见现象：在 Controller 中还能正常获取 SecurityContext，但在异步线程里安全上下文却 “失联” 了。

为了解决这一点，WebAsyncManagerIntegrationFilter 进行了一系列的工作：

  1. 首先获取 WebAsyncManager（这是 Servlet 容器用于管理异步任务的机制）
  2. 然后将当前的 SecurityContext 作为属性绑定到异步任务的请求上下文中
  3. 当异步任务真正执行时，它会自动恢复原先的 SecurityContext，从而保证在异步调用链中依然可以拿到正确的用户信息
  4. 异步处理完成后，再清理上下文，避免遗留数据污染线程池中的其他线程

1.2.3. SecurityContextHolderFilter

该过滤器会自动为本请求所在的线程初始化 SecurityContextHolder，并根据 Cookie 中的 Session ID 向 HttpSession 查找 SecurityContext（其内保存最重要的 Authentication）

  1. 如果找到 SecurityContext
    1. 将其加载到本请求所在线程的 SecurityContextHolder 中（SecurityContextHolder 默认是由 ThreadLocal 保存的）
  2. 如果未找到 SecurityContext
    1. 则在本请求所在的线程中，自动初始化一个新的 SecurityContext

即使我们不打算通过 HttpSession 实现 “记住我” 功能（例如使用 JWT/Remember-Me Cookie），我们仍然建议保留这个过滤器，因为它自动为本线程初始化 SecurityContextHolder、并自动创建 SecurityContext，这个能力实在太香了。

在 API 执行后，它会自动将本请求所在线程的 SecurityContext 存入服务器的 HttpSession，以便在后续请求中维持用户身份（该功能需要手动开启）

随后，它会清空本请求所在线程的 SecurityContextHolder，防止 SecurityContext 在后续请求中被无意复用，从而确保每个请求都能独立执行认证和授权流程。

1.2.4. HeaderWriterFilter

该过滤器会在 HttpServletResponse 中自动写入一组预设的安全响应头。这些响应头旨在提升浏览器端的防护能力，用于抵御常见的前端攻击风险

1.2.5. CsrfFilter

  1. 当客户端发起请求时，CsrfFilter 过滤器会尝试从我们配置的 CSRF Token 存储位置中加载 Token，且所有请求都会经过这一尝试
    1. 如果成功加载 Token
      1. Token 会被放入 HttpServletRequest 中
    2. 如果没有成功加载 Token
      1. 创建一个新的 Token，保存在我们配置的 CSRF Token 存储位置，并同样放入 HttpServletRequest 中
    3. 如果用户正在进行登录操作，那我们可以从 HttpServletRequest 中拿到 Token，并返回给前端。前端需妥善保存该 Token，在执行修改服务器状态的敏感操作（如 POST、PUT、DELETE、PATCH 等）时，前端必须将该 Token 携带，通常以请求头（默认名称：X-CSRF-TOKEN）或请求参数（名称：_csrf）的方式携带
  2. 当用户发起修改服务器状态的敏感请求（如 POST、PUT、DELETE、PATCH 等）时，CsrfFilter 会检查是否携带了 Token
    1. 如果携带 Token
      1. 将前端传来的 Token 与 我们配置的 CSRF Token 存储位置中的 Token 进行匹配
        1. 如果匹配
          1. 过滤器放行请求
        2. 如果不匹配
          1. 抛出 `InvalidCsrfTokenException

[!NOTE] 注意事项

  1. CSRF 相关的异常都是 AccessDeniedException 的子类，所以我们在处理 AccessDeniedException 时，注意处理这两个异常

1.2.6. LogoutFilter

该过滤器主要用于前后端不分离的场景，用于处理默认 /logout 路径下的退出请求。在前后端分离的架构中无需深入关注其具体逻辑，只需了解其在过滤器链中的位置，以便在插入自定义过滤器时能准确定位。

该过滤器主要负责：

  1. 清空 SecurityContext
  2. 使当前 HttpSession 失效
  3. 清理相关 Remember-Me Cookie

1.2.7. UsernamePasswordAuthenticationFilter

该过滤器主要用于前后端不分离的场景，用于处理默认 /login 路径下的登录请求。在前后端分离的架构中无需深入关注其具体逻辑，只需了解其在过滤器链中的位置，以便在插入自定义过滤器时能准确定位。

该过滤器主要负责：

  1. 自动调用 AuthenticationManager 执行认证逻辑
  2. 将认证成功后返回的 Authentication 对象存入 SecurityContext

1.2.8. RequestCacheAwareFilter

当用户访问一个需要登录的页面（例如 /profile）时，如果此时还没有登录，Spring Security 会可能会将用户重定向到登录页（如 /login）。当用户完成认证后，理想的行为是自动返回原本想访问的页面（/profile）

为了解决这一点，RequestCacheAwareFilter 会检测当前请求是否属于一次 “被拦截的访问”。如果是，它会在重定向之前，将原始请求信息缓存下来，以便在用户完成登录后，能够准确恢复目标 URL，并将用户顺利地引导回最初想访问的页面

需要注意的是，该过滤器同样主要用于前后端未分离的场景

[!NOTE] 注意事项

  1. RequestCacheAwareFilter 的默认实现是 HttpSessionRequestCache，它会在用户被拦截时将原始请求保存到当前的 HttpSession 中
  2. 在分布式部署中，如果登录请求被路由到另一台机器，而该节点无法访问之前保存的 Session，那么登录成功后就无法恢复原始目标地址并跳转回去了
  3. 如果应用依赖 Session，而单机版 HttpSession 无法满足分布式部署的需求，通常会采用分布式 Session 来解决

1.2.10. RememberMeAuthenticationFilter

该过滤器会检查当前请求所在线程中，是否已经存在 Authentication 对象

  1. 如果存在 Authentication
    1. 过滤器放行请求
  2. 如果不存在 Authentication
    1. 调用 RememberMeServices 尝试自动登录，RememberMeServices 会检查请求中是否携带有效的 Remember-Me Cookie
    2. 如果 Remember-Me Cookie 有效，则基于令牌创建一个新的 Authentication 对象，并将该对象写入 SecurityContext

1.2.11. AnonymousAuthenticationFilter

该过滤器会检查当前请求所在线程中，是否已经存在 Authentication 对象。

  1. 如果存在 Authentication
    1. 过滤器放行请求
  2. 如果不存在 Authentication
    1. 自动创建一个匿名 Authentication，以避免后续流程中出现空指针异常。

1.2.13. ExceptionTranslationFilter

该过滤器会捕获整个过滤器链中抛出的 AuthenticationException 和 AccessDeniedException ，并执行相应的处理逻辑。

1.2.14. AuthorizationFilter

该过滤器首先检查当前请求所在线程中是否存在 Authentication（无论是否为匿名身份）对象

  1. 如果存在
    1. 接着检查是否为匿名用户访问受保护资源
      1. 如果是
        1. 抛出 AuthenticationException 异常
      2. 如果不是
        1. 最后检查当前用户是否具备访问目标资源权限（包括资源级别和方法级别的访问控制）
          1. 如果具备权限
            1. 过滤器请求放行
          2. 如果不具备权限
            1. 抛出 AccessDeniedException
  2. 如果不存在
    1. 抛出 AuthenticationException，表示用户尚未进行认证

[!NOTE] 注意事项

  1. 整个流程中的异常由 ExceptionTranslation 过滤器统一处理，负责捕获整个过滤器链中抛出的 AuthenticationException 和 AccessDeniedException ，并执行相应的处理逻辑。

2.6. CsrfFilter 配置

2.6.1. 配置概述

Spring Security 的跨站请求伪造（CSRF）防护机制，是基于 CSRF Token 实现的，其工作原理如下：

  1. 当客户端发起请求时，CsrfFilter 过滤器会尝试从我们配置的 CSRF Token 存储位置中加载 Token，且所有请求都会经过这一尝试
    1. 如果成功加载 Token
      1. Token 会被放入 HttpServletRequest 中
    2. 如果没有成功加载 Token
      1. 创建一个新的 Token，保存在我们配置的 CSRF Token 存储位置，并同样放入 HttpServletRequest 中
    3. 如果用户正在进行登录操作，那我们可以从 HttpServletRequest 中拿到 Token，并返回给前端。前端需妥善保存该 Token，在执行修改服务器状态的敏感操作（如 POST、PUT、DELETE、PATCH 等）时，前端必须将该 Token 携带，通常以请求头（默认名称：X-CSRF-TOKEN）或请求参数（名称：_csrf）的方式携带
  2. 当用户发起修改服务器状态的敏感请求（如 POST、PUT、DELETE、PATCH 等）时，CsrfFilter 会检查是否携带了 Token
    1. 如果携带 Token
      1. 将前端传来的 Token 与 我们配置的 CSRF Token 存储位置中的 Token 进行匹配
        1. 如果匹配
          1. 过滤器放行请求
        2. 如果不匹配
          1. 抛出 InvalidCsrfTokenException

这样，即使攻击者能利用浏览器自动携带 Cookie 的特性，对我们的网站发起请求，但是由于敏感操作必须携带合法的 Token，而攻击者无法获取或伪造该 Token，因而能有效防止 CSRF 攻击。

[!NOTE] 注意事项

  1. 开发过程中，可以先禁用 CSRF 防护，到最后再进行 CSRF 攻击防护

2.11.2.1. HttpSession 的创建策略
  1. SessionCreationPolicy.ALWAYS
    1. 始终创建 HttpSession
    2. 每个请求都会新建一个 HttpSession，覆盖之前的 HttpSession，并返回新的 Session ID
  2. SessionCreationPolicy.IF_REQUIRED（默认）
    1. 按需创建 HttpSession
    2. 当需要使用 HttpSession 时，Spring Security 会自动创建 HttpSession，并返回对应的 Session ID
  3. SessionCreationPolicy.NEVER
    1. 从不创建 HttpSession，
    2. 需要注意的是，如果已有 HttpSession，会使用该 HttpSession
  4. SessionCreationPolicy.STATELESS
    1. 完全不使用 HttpSession
2.11.2.2. 会话固定防护攻击策略
  1. none()
    1. 不做处理
  2. migrateSession()（默认）
    1. 重新生成 Session ID
  3. newSession()
    1. 创建全新 HttpSession

    1. 要求用户必须具备指定角色才能访问该资源路径
    2. 需要注意的是，该方法会自动在角色名前添加 "ROLE_" 前缀，即：hasRole("ADMIN") 代表 ROLE_ADMIN
  4. hasAuthority()
    1. 要求用户必须具备指定权限才能访问该资源路径
    2. 需要注意的是，不会自动添加前缀，提供完整权限名称即可
  5. authenticated()
    1. 要求用户已通过身份验证（即非匿名 Authenticated）后才能访问该资源路径
  6. access()
    1. 支持使用 SpEL 表达式，实现更复杂的访问控制逻辑。
    2. 例如 access("hasRole('ADMIN') and hasIpAddress('192.168.1.0/24')") 是指，该路径资源仅允许拥有 ADMIN 角色且 IP 地址位于 192.168.1.0/24 网段的用户访问

3. CORS 跨域资源共享 配置（配置类）

3.1. 配置类名称

配置类名称为：SecurityConfig

3.2. 配置概述

CORS 是指：出于安全考虑，浏览器默认会阻止网页从一个源（如 http://a.com）向另一个源（如 http://b.com/api/data）发起请求，这种 “同源策略” 能够有效防范诸如 CSRF、XSS 等跨站攻击。但在一些场景下我们确实需要跨源访问，例如当前端和后端分离部署在不同的源时，仍然需要让前端能够访问后端接口。

这时，如果后端的 http://b.com/api/data 启用了 CORS 跨域资源共享机制，那么浏览器就允许网页从 http://a.com 向这个接口发起请求并成功获取数据。

需要注意的是，CORS 跨域资源共享与域名解析得到的 IP 地址无关，而是基于 “源”（Origin）来判断。那么什么是 “源” 呢？“源” 由协议（scheme）+ 域名（host）+ 端口号（port）三部分组成，不包括 URL 后面的路径和参数。例如，https://example.com/page 与 https://example.com/another 属于同源，但 http://example.com/page 与 https://example.com/page 就不同源。即便它们甚至解析到相同的 IP，只要协议、域名或端口任一部分不完全相同，就被视为不同源。

除此之外，CORS 只能解决跨域访问的问题，用户权限不足时依然会被拒绝访问。如果用户没有登录，也就是证明没有开启 “记住我” 功能，那么每个请求都会以匿名身份（匿名 Authentication）访问，对于需要认证的资源，尤其是权限要求较高的资源，这些请求会被 Spring Security 拒绝甚至是重定向的登录页。换句话说，即使跨域访问成功，没有足够权限也无法访问受保护资源。

通常情况下，我们希望用户先登录，这样就证明启用了 “记住我” 功能，当访问后端的时候，就会携带凭证，自动完成认证逻辑，Spring Security 就能自动识别用户权限。但需要注意的是，如果使用基于 HttpSession 的认证，由于后端服务器可能是集群架构，用户需要在多台服务器上重复登录，带来管理上的复杂性。

5. configuration.setAllowCredentials
在讲解这个之前，我们先补充一个前置知识：当我们是同源访问时，前端访问后端，一般是使用 AJAX 进行请求的，然后浏览器会自动携带 Cookie 发送到后端，这点大家都很熟悉。

但是在跨域访问时，如果是通过 AJAX 发起的请求，出于安全考虑（担心你用 AJAX 伪造请求），浏览器默认不会携带 Cookie（直接防止了 CSRF 攻击）。如果想让浏览器携带 Cookie，前端必须在 AJAX 请求中添加 credentials: 'include'，这样 Cookie 才会被发送。

但如果是通过 HTML 元素导航发起的请求，比如 <img src="https://bank.com/transfer?to=attacker&amount=10000" />，无论是否跨域，浏览器都会自动携带对应 Cookie（这是我们进行 CSRF 攻击防护的核心原因）

但并不是说你前端愿意携带 Cookie，我后端就理所当然地愿意接收。因为后端并不知道你发送过来的 Cookie 是出于正常用途，还是存在安全隐患。简单来说，如果我们的 configuration.setAllowCredentials 默认值是 false，即使前端执意携带 Cookie 发起请求，接口本身执行并正常返回了响应，浏览器也会因为安全策略自动屏蔽返回结果，导致前端无法正常获取数据。

但如果将其设置为 true，意味着前后端达成共识，允许使用 Cookie，这样跨域请求才能正常携带 Cookie 并获得返回结果：

[!NOTE] 注意事项

6. 方法级别的访问控制 配置（配置类）

6.1. 配置类名称

配置类名称为：SecurityConfig

6.2. 配置概述

我们先来了解一下资源级别的访问控制：只有具备指定权限或身份的用户，才能访问特定路径下的资源，例如 /api/admin/**

而方法级别的访问控制则是指，用户必须具备指定的权限或身份，才能调用某方法

6.3. 具体配置

[!NOTE] 注意事项

  1. 方法级别的访问控制际上是通过 Spring AOP 实现的，它并不依赖于 Spring Security 的过滤器链，因此你在过滤器中看不到与方法权限控制相关的过滤器
  2. 当用户调用某个被限制访问的方法而权限不足时，Spring Security 会在方法调用阶段直接抛出 AccessDeniedException

6.4. 使用方法

方法级别的访问控制，一般进行在服务层（Service 层）、控制器层（Controller 层）

[!NOTE] 注意事项

  1. 如果你在 Service 层进行了方法级别的访问控制，那么就有可能出现，虽然能够执行前半部分逻辑，但调用 Service 方法时会被拒绝，并抛出 AccessDeniedException
