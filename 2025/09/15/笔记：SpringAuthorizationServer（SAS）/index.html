1.1. 核心流程概述

在 SAS 中，我们需要配置两条过滤器链：

  1. authorizationServerSecurityFilterChain 用于拦截访问 OAuth 相关 URL 的请求，这条链的优先级较高
  2. SecurityFilterChain 用于拦截其他 URL 的请求

在查看 authorizationServerSecurityFilterChain 默认注册的过滤器链时可以发现，它是在 Spring Resource Server 的基础上，额外添加了与 OAuth 相关的过滤器。

虽然 Spring Authorization Server 本身也能作为资源服务器使用，但专业的事情还是交给专业的人来处理比较稳妥。

[!NOTE] 注意事项

  1. 如果没有为 SAS 配置自定义密码加密器，那么当 SAS 从数据库读取到像 {noop}123456 这样的 client_secret 时，它会根据数据库中存储的加密前缀 {noop} 自动处理我们传入的 123456，然后将处理后的结果与 {noop}123456 去掉前缀的内容进行匹配
  2. 但是，如果配置了自定义密码加密器，它会全局使用该加密器，既不会识别 {noop} 等前缀，自动选择合适的加密器，也不会去掉前缀进行匹配
  3. 例如，当配置 NoOpPasswordEncoder 时，
    1. 如果我们传入123456
      1. SAS 会对传入的 123456 进行处理（仍为 123456）
      2. 然后直接与数据库中的 {noop}123456 对比，匹配失败。
    2. 如果我们传入 {noop}123456
      1. SAS 会对传入的 {noop}123456 进行处理（仍为 {noop}123456）
      2. 然后接与数据库中的 {noop}123456 对比，匹配成功
  4. 同样地，当配置 BCryptPasswordEncoder 时
    1. 如果我们传入 {noop}123456
      1. SAS 会对传入的 {noop}123456 进行处理（为 XXXX）
      2. 然后直接与数据库中的 {noop}123456 对比，匹配失败

9. 客户端 配置

9.1. 配置概述

在 Spring Authorization Server 中，RegisteredClient 定义了客户端需要注册哪些信息

1. id
id 是数据库中的主键，不需要客户端注册，Spring Authorization Server 会使用该 ID 作为数据结构中的键，例如 Map<String, RegisteredClient>

[!NOTE] 注意事项

  1. Spring Authorization Server 要求 id 字段为 String 类型
    1. 如果数据库中使用 int 类型作为主键，需要额外编写 TypeHandler 来做类型转换
    2. 在数据库中推荐使用 varchar 类型进行存储，并使用 UUID 作为主键
  2. 为什么不直接用 clientId 作为数据库主键？
    1. 因为 clientId 可能会被重置，而数据库主键必须 唯一且不可变

2. clientId
clientId 是 OAuth2 协议中用于标识客户端的重要凭证之一，同样不需要客户端注册，而是当客户端进行注册时，由我们生成一个 clientId 并返回给他们。

[!NOTE] 注意事项

  1. Spring Authorization Server 要求 clientId 字段为 String 类型

3. clientIdIssuedAt
clientIdIssuedAt 是 clientId 的发放时间，同样不需要客户端注册，而是在生成 clientId 时由我们创建，该时间用于后续跟踪和判断 clientId 的有效性。

[!NOTE] 注意事项

  1. Spring Authorization Server 要求 clientIdIssuedAt 字段为 Instant 类型
  2. Spring Authorization Server 并不会对该时间戳进行任何处理，这主要是为了便于我们开发人员对 clientId 的生命周期进行跟踪和管理。

4. clientSecret
clientSecret 同样是 OAuth2 协议中用于标识客户端的重要凭证之一，同样不需要客户端注册，而是当客户端进行注册时，由我们生成一个 clientSecret 并返回给他们，可以将 clientId 理解为账号，而 clientSecret 则相当于密码。

[!NOTE] 注意事项

  1. Spring Authorization Server 要求 clientSecret 字段为 String 类型，并且格式要求为：{bcrypt}$2a$10$...
  2. {bcrypt} 是加密前缀，用于指示 Spring Authorization Server 使用那种密码加密器（PasswordEncoder）验证客户端提交的 clientSecret 是否与数据库中存储的值一致
  3. 常用加密前缀包括：
    1. {noop}
      1. 使用 NoOpPasswordEncoder
    2. {bcrypt}
      1. 使用 BCryptPasswordEncoder
    3. {pbkdf2}
      1. 使用 Pbkdf2PasswordEncoder
    4. {scrypt}
      1. 使用 SCryptPasswordEncoder

5. clientSecretExpiresAt
clientSecretExpiresAt 是 clientSecret 的过期时间同样不需要客户端注册，而是在生成 clientId 时由我们创建。

Spring Authorization Server 在校验 client_secret 时，会检查 clientSecretExpiresAt，如果已过期，则拒绝认证。

[!NOTE] 注意事项

  1. Spring Authorization Server 要求 clientSecretExpiresAt 字段为 Instant 类型
  2. clientSecretExpiresAt 一般为永久有效（null）

6. clientName
clientName 是客户端的名称，需要客户端注册，用于展示给用户，例如在授权确认页面中，会显示为 “XXX 正在请求访问你的信息，是否同意”。

[!NOTE] 注意事项

  1. Spring Authorization Server 要求 clientName 字段为 String 类型

7. clientAuthenticationMethods
clientAuthenticationMethods 是客户端进行身份认证的方式（传递 client_id 和 client_secret 的方式），需要客户端注册，常用身份认证的方式包括：

  1. client_secret_basic（默认）：
    1. 客户端通过 HTTP Basic 的方式传递 client_id 和 client_secret
    2. 即将其放入请求头的 Authorization 中，格式为：Authorization: Basic <Base64 编码后的 clientId:clientSecret>
  2. client_secret_post：
    1. 客户端在表单参数中传递 client_id 和 client_secret
  3. none：
    1. 仅传 client_id，不传 client_secret，常用于 SPA 单页应用或移动端应用

[!NOTE] 注意事项

  1. Spring Authorization Server 要求 clientAuthenticationMethods 字段为 Set<ClientAuthenticationMethod> 类型
  2. 一个客户端可以注册多个身份认证方式，Spring Authorization Server 会按顺序依次尝试这些方式，直到认证成功为止（不需要在请求参数中携带你使用的是具体那种方式）
  3. 客户端在传递 client_secret 的时候，不要把加密前缀带上

8. authorizationGrantTypes
authorizationGrantTypes 是客户端支持的授权模式，需要客户端注册，常用授权模式包括：

  1. authorization_code：
    1. 授权码模式
  2. client_credentials：
    1. 客户端凭证模式
  3. refresh_token：
    1. 令牌刷新模式

[!NOTE] 注意事项

  1. Spring Authorization Server 要求 authorizationGrantTypes 字段为 Set<AuthorizationGrantTypes> 类型
  2. 一个客户端可以注册多个授权模式，但是 Spring Authorization Server 并不会按顺序依次尝试这些方式，而是在发起授权请求时，客户端必须明确指定所使用的授权模式。如果使用的授权模式未在客户端注册，授权服务器会拒绝处理该请求

9. redirectUris
redirectUris 是用户授权成功后的回调地址，需要客户端注册，而且客户端在发起授权请求时，必须携带完全一致的值，否则服务器将拒绝处理该请求

[!NOTE] 注意事项

  1. Spring Authorization Server 要求 redirectUris 字段为 String 类型
  2. Spring Authorization Server 虽然允许一个客户端配置多个 redirectUris，并会依次处理这些地址。但是出于安全考虑，通常我们只允许一个客户端应用注册一个重定向地址

10. postLogoutRedirectUri
postLogoutRedirectUri 是用户从授权服务器退出后的回调地址，一般留空即可

11. scopes
scopes 是客户端请求的权限范围，需要客户端注册，而且客户端在发起授权请求时，需要携带所需的 scope，服务器根据请求的 scope 颁发相应的 Access Token，保证客户端用该 Access Token 只能访问授权范围内的信息，如果携带的权限范围未在客户端注册，授权服务器会拒绝处理该请求。

[!NOTE] 注意事项

  1. Spring Authorization Server 要求 scopes 字段为 Set<String> 类型
  2. 一个客户端可以注册多个权限范围，但是不一定都能用得上
  3. 在颁发 Access Token 时，必须区分自有客户端与第三方客户端的权限
    1. 对于自有客户端，可以颁发具备完整访问权限的 Token
    2. 对于第三方客户端，则只能获取有限的数据访问权限
    3. 因此，可以定义一个仅供内部使用、特殊 scope，通过该 scope 即可赋予自有客户端完整权限

11. clientSettings
clientSettings 是客户端级别的配置，需要客户端注册，常用客户端级别的配置包括：

  1. settings.client.require-authorization-consent
    1. 是否在授权时，显示用户授权确认页面
    2. 常用选项包括：
      1. true
        1. 用户登录之后，会被引导到授权确认页面（consent page），要求勾选要授予客户端哪些权限（scopes）
        2. 如果用户之前已经授权过该客户端，SAS 会记录该用户的授权决定。下次访问时，只要请求的 scopes 没有变化，就不会再次显示授权确认页面。
      2. false
        1. 用户登录之后，会直接颁发 Authorization Code
  2. settings.client.require-proof-key
    1. 是否启用 PKCE
    2. 常用选项包括：
      1. true
        1. 启用 PKCE
      2. false
        1. 不启用 PKCE

[!NOTE] 注意事项

  1. Spring Authorization Server 要求 clientSettings 字段为 ClientSettings 类型

12. tokenSettings
tokenSettings 是 Authorization Code、Access Token、Refresh Token、ID Token 的配置，需要客户端注册，常用令牌配置包括：

  1. settings.token.authorization-code-time-to-live
    1. Authorization Code 的有效期，以秒为单位
  2. settings.token.access-token-time-to-live
    1. Access Token 的有效期，以秒为单位
  3. settings.token.access-token-format
    1. Access Token 的格式
    2. 常用形式包括：
      1. REFERENCE
        1. Access Token 是一个随机字符串，不包含任何信息
        2. 授权服务器需要维护一个数据结构，将 Access Token 对应当相应信息
      2. SELF_CONTAINED
        1. Access Token 本身就包含了用户的基本信息、权限、有效期等，SAS 用 JWT 实现
  4. settings.token.refresh-token-time-to-live
    1. Refresh Token 的有效期，以秒为单位
  5. settings.token.reuse-refresh-tokens
    1. Refresh Token 是否可以重复使用
    2. 常用选项包括：
      1. true
        1. Refresh Token 可以重复使用
      2. false
        1. Refresh Token 不可以重复使用，使用后 SAS 会返回新的 Refresh Token

11. Access Token 公私钥 配置

11.1. 配置概述

SAS 默认返回的 Access Token 是一种 JWE，使用 RS256 算法进行签名。这意味着系统需要一对 RSA 公私钥：私钥用于生成签名，公钥用于验证签名。

默认情况下，SAS 在启动时会随机生成一对 RSA 公私钥，并仅存储在内存中。在分布式部署场景下，不同的 SAS 实例会各自生成不同的密钥对，导致签发的 Token 无法在其他实例上被验证。为了解决这个问题，我们需要统一管理 RSA 密钥对，确保所有实例共享同一套公私钥。

这些数据存储在内存中，而不是保存在 HttpSession 中，因此不能通过分布式 Session 来解决。在遇到此类数据存储在内存的场景时，常见的处理方式通常有两种：

  1. 数据存入数据库
  2. 数据存入配置中心

在这里，我们选择采用数据存入数据库的方案。
