2. OAuth2 协议

2.1. OAuth2 概述

O 表示开放（Open），Auth 表示授权（Authorization），OAuth2 是一种开放式的授权协议，用于在不暴露用户凭据的前提下，让第三方应用获得访问权限

2.2. OAuth2 核心角色

  1. 资源所有者（Resource Owner）
  2. 客户端应用（Client）
  3. 资源服务器（Resource Server）
  4. 授权服务器（Authorization Server）

4. 用户登录和授权
当访问 /oauth2/authorize 时，请求会经过 authorizationServerSecurityFilterChain 过滤器链，其中 OAuth2AuthorizationEndpointFilter 专门负责处理这个端点。

它会先检查客户端是否合法：

  1. 如果客户端合法
    1. 检查当前请求所在线程的 Authentication 是否认证
      1. 如果已认证：
        1. 进行用户授权
          1. 如果客户端配置为无需用户授权，用户不会看到授权页，直接授权成功
          2. 但多数情况下，会要求用户手动勾选要授予哪些权限（scope），以便用户清楚地知道第三方应用可以访问什么
      2. 如果未认证：
        1. 抛出 AuthenticationException，然后由 ExceptionTranslationFilter 捕获并将请求重定向到 /login
        2. 需要注意的是，这个跳转到的 /login 请求本身不带额外参数，但在用户成功登录后，系统会自动重定向到原先的请求。其实，就是 表单登录的特性：访问受保护的资源 ➝ 被拦截后跳转到登录页 ➝ 登录成功再返回原请求页面。在 SAS 这种前后端不分离的传统场景里，这种流程非常常见。但在我们熟悉的前后端分离开发中，这种模式用得比较少，我们理解即可。
  2. 如果客户端不合法
    1. 直接报错

[!NOTE] 注意事项

  1. 在实际使用中，为了避免用户频繁登录，我们通常会实现授权服务器的 “记住我” 功能
  2. 为了实现这一功能，过去我们通常采用 JWT 的方式，由前端发送 JWT 给后端，然后后端再去验证。
  3. 但在这里，问题来了：前端究竟是谁呢？
  4. 如果前端是第三方客户端，它可不会乖乖配合把 JWT 按我们的预期带上。
  5. 于是我们换个思路：把浏览器当作前端，通过 Cookie 来实现。
  6. 例如，可以使用 Remember-Me Cookie，或者直接把 JWT 存放在 Cookie 中，这样验证环节就能顺利跑通了。

3. OAuth2.1 协议

3.1. OAuth2.1 概述

OAuth2.1 目前仍处于草案阶段，但正逐渐被广泛接受，有望成为主流。

OAuth2.1 较于 OAuth2 的主要变化包括：

  1. 对授权码模式（Authorization Code）进行了增强，扩展了 PKCE（授权码交换的密钥证明），并且配套了刷新令牌模式（Refresh Token）
    1. OAuth2.1 明确指出，只要使用授权码模式，必须同时使用 PKCE
    2. 授权码模式与刷新令牌模式强相关，因为客户端凭证式根本用不到刷新令牌
  2. 移除了不再安全的隐藏式（Implicit）和密码式（Password），新增了刷新令牌模式（Refresh Token）模式

[!NOTE] 注意事项

  1. Spring Authorization Server 从来没有支持过 Implicit 和 Password

3.2. OAuth2.1 授权模式

3.2.1. 授权码模式（Authorization Code） + PKCE + Refresh Token

该模式与普通的授权码模式相比，多了 PKCE 和 Refresh Token

先说 PKCE，它是为了解决授权码可能被劫持的问题：

  1. 在传统授权码模式中，用户完成授权后，回调地址会携带 Authorization Code
  2. 如果攻击者截获了这个 Authorization Code，就可以去交换 Access Token
  3. 为防止这一风险，传统做法要求令牌请求由后端发起，并携带 client_secret
  4. 但是对于前端应用，由于没有后端，所以 client_secret 必须由前端携带
  5. 可是如果前端携带 client_secret，就极易被截获，这就带来了安全隐患

为了解决这一问题，PKCE 要求在发起授权请求时携带 code_challenge 和 code_challenge_method，授权服务器会记住这两个参数

此外，PKCE 还要求在发起令牌请求时携带 code_verifier，授权服务器会根据 code_challenge_method 对 code_verifier 加密，然后和 code_challenge 进行对比

你可能会疑惑：这和 client_secret 有什么区别？前端不还是需要保存 code_challenge、code_challenge_method 和 code_verifier 吗？

其实，Code Verifier 和 Client Secret 最大的区别在于：Code Verifier 是临时的、一次性的，只在交换 Access Token 时使用一次，用完即废。即便泄露，也无法被重复利用，因此安全性远高于长期有效的 Client Secret

  2. PKCE 参数
    1. code_verifier

  1. grant_type
  2. refresh_token

[!NOTE] 注意事项

  1. 即使客户端能够安全地携带 Client Secret，同时也可以使用 Code Verifier，在这种情况下 PKCE 的安全作用可能不大，但仍然 推荐同时开启 PKCE，两者结合使用可以提供更高的安全保障。
