二、Zookeeper 基础

1. Zookeeper 应用场景

1.1. 分布式锁

在分布式锁的实现中，所有客户端都会在 Zookeeper 的同一路径下创建临时顺序节点。Zookeeper 会按照节点的序号排序，序号最小的节点获得锁，其余客户端则处于等待状态。

当持锁客户端释放锁（即对应临时节点被删除）时，Zookeeper 会通知下一个顺序节点的客户端，从而保证锁的有序传递，实现类似“排队取号”的效果。

1.2. 配置中心

把配置信息写入 Znode，并通过观察者模式让所有节点实时监听该 Znode。一旦 Zookeeper 中对应的 ZNode 数据被修改，Zookeeper 会主动通知各个客户端，从而保证配置在集群中的一致性与及时更新。

1.3. 元数据存储

1.4. 分布式队列

1.5. 软负载均衡

ZooKeeper 本身不负责请求的负载均衡，而是提供服务注册、发现和命名的能力。像 Dubbo、gRPC 等框架会从 ZooKeeper 或 Nacos 等注册中心获取可用服务节点列表，再由客户端根据预设策略在本地选择一个节点并发起请求。

1.6. Leader 选举

在 Leader 选举过程中，所有候选者会在 Zookeeper 的指定路径下创建临时顺序节点。Zookeeper 会按照节点的序号排序，序号最小的节点即被选为 Leader。

如果当前 Leader 节点因掉线或宕机导致临时节点被删除，其余候选节点会立即感知，从而触发一次新的选举，按同样规则产生新的 Leader。

1.7. 服务注册、发现、命名

对集群中各个节点的状态进行统一跟踪和管理，包括节点的上下线情况以及运行状态变化。

  1. 节点上下线状态监控
    1. 每个服务节点会在 Zookeeper 上注册一个临时节点，一旦服务节点宕机或发生网络中断，该临时节点会被自动删除
    2. 其他客户端能够立即感知这一变化，从而实时掌握集群中节点的存活情况，并及时做出调整
  2. 节点运行状态监控
    1. 除了存活信息外，节点的运行状态（如负载、健康度等）也可以写入 Zookeeper 的 ZNode
    2. 客户端通过监听该 ZNode，即可在节点运行状态发生变化时实时获取通知
    3. 这样能够帮助系统根据节点的实时状态动态调整资源分配与任务调度

2. Zookeeper 节点类型

  1. 持久节点（Persistent Znode）
    1. 持久节点
      1. 节点一旦创建，除非被显式删除，否则会一直存在
    2. 顺序持久节点
      1. 创建时，Zookeeper 会在节点名称后自动加上递增序号，例如 Znode_002
  2. 临时节点（Ephemeral Znode）
    1. 临时节点
      1. 客户端创建节点后，如果与服务器断开连接，所创建的节点会自动删除
    2. 顺序临时节点
      1. 创建时，Zookeeper 会在节点名称后自动加上递增序号，例如 Znode_002

天璇剑（数据的组织形式）

Zookeeper 在内存中的数据组织形式，是一个典型的树形结构，类似于文件系统的目录树，被称之为 Znode 树

每一个 Znode 都有一个唯一的以 / 分隔的路径，例如 /app/config/db-host，并且可以存储少量数据

[!NOTE] 注意事项

  1. 一个 Znode 仅可以存储 1MB 数据，还需要包含自身的元信息，例如版本号（version）、访问控制列表（ACL）、时间戳以及子节点数量等。因此，ZooKeeper 通常仅用于存储元数据和配置信息，不适合承载大规模数据。
  2. 节点的数据类型为字节数组（byte[]），理论上可以存储任意内容

元数据写入在北斗七剑的选择如下：

  1. 天枢剑：主从复制
  2. 天璇剑：树状结构
    1. Zookeeper 内部维护一个类似于文件系统的、层次化的命名空间，这个数据模型的核心是 znode（Zookeeper Data Node）
    2. 每个 znode 都有一个唯一的、以 / 分隔的路径，例如 /app/config/db-host
    3. 每个 znode 不仅可以存储少量数据（默认只能够存储 1MB 数据），还包含自身的元信息，如版本号（version）、访问控制列表（ACL）、时间戳以及子节点数量等
  3. 天权剑：整体存储
    1. Zookeeper 并不对数据进行拆分，整个 znode 树的完整副本被存储在集群中的每一个服务器上
    2. 无数据拆分这种设计，意味着 Zookeeper 不适合存储大量数据，其设计初衷就是用于存储和协调少量的、关键的元数据
  4. 天玑剑：同步复制（大多数）
  5. 玉衡剑：快照 + 增量（日志拉取）
  6. 开阳剑：共识协议（ZAB 协议）
  7. 瑶光剑：
    1. 持久化逻辑：混合持久化（定期删除旧日志）
    2. 存储引擎实现：写时复制模型（记录 WAL）
    3. 操作系统交互：强制刷盘（同步刷盘）
      1. Zookeeper 在写入事务日志后，会显式调用 fsync，强制操作系统将数据从页缓存（Page Cache）刷到物理磁盘

1.3.6. Swap 分区要求

swap 分区，简单来说，是操作系统在系统内存不足时，将某些程序不活跃或暂时不需要的数据移动到 swap 分区（位于硬盘），以释放内存资源供当前活跃程序使用。

由于 swap 分区依赖硬盘，其读写速度远低于内存。如果系统频繁使用 swap，会造成大量磁盘 I/O，严重影响系统性能，甚至导致系统响应迟缓或无响应。因此，在性能敏感的场景（如数据库、实时计算、大内存服务器）中，通常建议禁用 swap，确保数据始终在高速内存中运行。

五、Spring Cloud Zookeeper

1. Spring Cloud Zookeeper 概述

Spring Cloud Zookeeper 主要用于微服务层面的服务注册、发现、命名和作为配置中心，不适合用于 Znode 的直接增删改查，也不支持高级分布式协调能力（例如分布式锁、分布式队列、Leader 选举）

六、Apache Curator

1. Apache Curator 概述

Apache Curator 专注于 ZooKeeper 的 Znode 操作，并提供完整的分布式协调能力（例如分布式锁、分布式队列、Leader 选举）。

[!NOTE] 注意事项

  1. 虽然 Apache Curator 也能实现 Spring Cloud Zookeeper 的功能，但是不如直接使用 Spring Cloud Zookeeper 更合适。
