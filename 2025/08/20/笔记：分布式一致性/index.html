笔记：分布式一致性

Last updated on 2025-09-04T23:19:48+08:00

分布式一致性

分布式一致性是用来描述多个副本之间数据是否一致、何时一致，以及对用户来说是否可见一致，一致性的模型有很多：

  1. 强一致性
  2. 线性一致性
  3. 顺序一致性
  4. 因果一致性
  5. 会话一致性
  6. 最终一致性

但是我们仅需要将其简单地划分为两大阵营：

  1. 强一致性
    1. 任何时刻，所有客户端看到的都是相同且最新的数据
  2. 最终一致性
    1. 数据可能暂时不一致，但在没有新的写入操作后，所有副本最终会达到一个相同的值
    2. 最终一致性通常可以在我们人为干涉的情况下，达到一种看似强一致性的效果，这也被称为“伪强一致性”。

复制协议

我们可以把一致性看作系统的目标，而复制协议则是实现这一目标的手段，不同的复制协议会直接影响系统的一致性。

需要注意的是，复制协议不仅仅是用来解决 “复制” 问题的，它既用于 “复制”，也用于 “写入”。这听起来像个悖论，但实际上，在分布式系统中，写入和复制本质上是同一操作的两个方面。因为一次成功的写入操作，其定义就是：数据在所有（或多数）副本上成功完成复制。

下面内容，是复制协议的六脉神剑：

  1. 少商剑：节点的拓扑结构
    1. 这个维度决定了数据流动的方向，也就是谁负责写入，谁负责同步
    2. 主从复制（单主复制）
      1. 系统中只有一个主节点，具有多个从节点
      2. 客户端所有写请求都必须发到主节点，多个从节点负责同步数据
      3. 客户端所有读请求既可以发到主节点也可以发到从节点
      4. 在复杂系统中，可以对集群进行分区，使得不同分区拥有各自的主节点以及对应的从节点（例如 ES），这也被称之为分区主从
    3. 多主复制
      1. 在主从复制模式下，所有写入操作必须经由主库完成。一旦主库因故障或网络原因不可访问，写入就会受阻。为了解决这一瓶颈，人们提出了多主复制的方案，即允许多个主节点同时承担写入任务。
      2. 根据节点间数据 “复制模式”（维度一）的不同，多主复制又可分为异步多主和同步多主。
        1. 异步多主
          1. 异步多主通常应用于多个数据中心场景。每个数据中心内部依旧采用常规的主从复制，但在不同数据中心的主节点上，可以同时写入同一份数据。随后，各个数据中心之间再进行异步的数据同步
          2. 之所以称为 “异步多主”，是因为跨数据中心的数据传播大多依赖异步复制（例如 MySQL 环状复制）
        2. 同步多主
          1. 常用于单数据中心内的集群，每个节点彼此平等，都可以同时执行读写请求，不再需要永远固定的 “老大哥主节点”，这样能减少单点故障风险，实现较高的可用性或更严格的分布式事务
          2. 之所以称为 “同步多主”，是因为一般通过同步复制来保证节点间的数据一致性
          3. 例如 RabbitMQ 的所有节点在架构上是对等的，在元数据写入时，执行写入的那个节点会临时充当 “主节点”（事务协调者，进行事务协调），启动一个分布式事务来确保所有节点一致。因此，RabbitMQ 在元数据写入方面，实际上符合同步多主的模式
          4. 之所以不将这种方式归类为主从复制，是因为：
            1. 它不符合典型的 “主—从” 角色划分，不存在一个始终固定的主节点
            2. 它不依赖单一主节点作为统一访问入口，而是允许任何节点在需要时接收请求
            3. 每个节点都有可能在特定时刻临时充当主节点，本质上更接近“多个主节点”的模式
          5. 之所以不讲这种方式归类未无主复制，是因为：
            1. 无主复制的核心是 Quorum NWR 模型
      3. 多主复制面临的核心挑战在于写入冲突
        1. 由于系统中存在多个写入入口，就可能同时产生彼此矛盾的更新。
        2. 例如，你在在线文档中输入 “项目将在本周五发布”，而同事几乎同时写下 “项目将在下周一发布”。
        3. 这类冲突必须通过机制来检测、处理或避免，才能保证数据一致性。
    4. 无主复制
      1. 集群中不存在 “主从” 之分，所有节点地位对等、功能相同，每个节点都可直接执行写入操作，其他节点负责同步数据
      2. 无主复制本质上不强制依赖 Quorum NWR 模型，但在工程实践中，为了保证读写结果的正确性和可预期性，几乎都会配合 Quorum NWR 来控制一致性
        1. 若满足 R + W > N，则读操作必然会命中至少一个包含最新写入的副本
        2. N = 副本总数
        3. W = 写请求需要确认的副本数
        4. R = 读请求需要确认的副本数
      3. 无主复制面临的核心挑战同样在于写入冲突
  2. 商阳剑：数据的分布策略
    1. 这个维度决定了数据是作为整体存放在某一位置内，还是被拆分后分布在多个位置
    2. 整体存储
      1. 数据完整地存储在某一位置
    3. 数据拆分
      1. 分库
      2. 分表
      3. 分片
        1. 例如 Minio 在存储对象时，会将其切割成若干块，并在多个驱动上进行分布
  3. 中冲剑：数据的同步时机
    1. 这个维度直接决定了我们的分布式系统提供强一致性还是最终一致性
    2. 同步复制：提供强一致性
      1. 当主节点收到一个写入请求后，它必须等待所有/大多数从节点都成功复制了数据，才能向客户端返回成功响应
    3. 异步复制：提供最终一致性
      1. 当主节点收到一个写入请求后，它立即向客户端返回成功响应，然后在后台异步地将数据同步给从节点
    4. 半同步复制：折中方案
      1. 强一致性和最终一致性的折中方案，主节点在提交事务后，只等待至少一个从节点确认收到数据，就向客户端返回成功
  4. 关冲剑：同步的传输内容
    1. 日志拉取
      1. 不直接同步数据本身，而是同步引起数据变化的、确定性的操作指令 (Command/Operation)
      2. 所有副本都以完全相同的顺序执行这些指令，从而达到一致的状态。基于日志是状态机复制理论的直接体现
      3. 主节点在执行写入操作时，会将变更顺序地记录到日志中
      4. 从节点通过拉取这些日志条目并按顺序回放，来重建数据状态（从哪里开始缺失就从哪里开始拉取）
    2. 状态同步
      1. 状态转移
        1. 主节点直接把 “最终状态” 发送给从节点，用新的状态整体覆盖旧内容（可能是全部状态，也可能仅是需要被覆盖的部分状态）
        2. 例如字符串从 Hello ➔ Hello World，主节点不会发 “请追加 World”，而是直接把完整的 Hello World 状态传输过去。
        3. 再例如 MySQL 中，修改后的数据页会替换掉硬盘上的旧数据页，这就是最纯粹的状态转移。
        4. 与快照不同的是，状态转移是每一轮同步的基本方式，而快照更像是一种辅助优化（例如每隔 5 秒才做一次），并非每次都用。
      2. 状态合并
        1. 例如在 A 节点上点了 5 次赞（结果 105），在 B 节点上又点了 8 次赞（结果 108）。
        2. 如果用状态转移方式，A 把 105 发给 B，B 把 108 发给 A，会造成冲突，导致增量信息丢失
        3. 改为传递 “状态合并”，即 A 记录并发送“我增加了 5 个赞”，B 记录并发送“我增加了 8 个赞”。这样合并后，大家都得到正确的 113
        4. 状态合并的优势在于与同步顺序无关，只要变更被正确传播，最终状态就是一致的
    3. 请求转发
      1. 这是最简单也最危险的方式，直接将客户端发来的**原始请求 (Raw Request)**，原封不动地转发给所有副本，期望大家独立执行后能得到相同的结果
      2. 然而，如果请求中包含诸如 timestamp() 或 random() 这样的非确定性操作，各节点独立执行时就可能得到不同结果，从而产生状态分歧，直接破坏强一致性。
      3. 此外，如果请求的执行逻辑依赖于节点当前已有的状态，而这些状态在不同节点上并不完全相同，那么即使是同一个请求，也可能得出不同结果，依旧存在一致性风险
    4. 快照 + 增量
      1. 快照，简单来说，就是把某个时间点之前所有历史操作执行后的最终状态记录下来
      2. 增量同步则相对简单，可以基于前面提到的日志复制、状态同步、请求转发等方式来实现
      3. 因此，“快照 + 增量同步” 可以看作其他方式的增强机制：快照负责 “快速赶上旧账”，增量负责 “实时保持跟进”，两者结合既提高了效率，又兼顾了可靠性
      4. 使用快照的典型场景：
        1. 新节点首次加入时，直接加载快照，再拉取最新增量；
        2. 节点宕机过久，错过了太多日志，且这些日志已被清理，只能重新加载快照恢复
  5. 少泽剑：状态的协调机制
    1. CRDT
    2. 共识协议
    3. 分布式事务
    4. Quorum NWR 模型
  6. 少冲剑：数据的持久化保障
    1. 持久化逻辑
      1. 状态持久化（基于快照的持久化）
        1. 在某个特定时间点，将完整数据状态一次性地写入磁盘
      2. 操作持久化（基于日志的持久化）
        1. 不直接记录数据本身，而是将所有导致数据变化的写操作指令按照顺序追加到日志文件中
        2. 恢复时，可以通过重放所有操作来重建最终的数据状态
      3. 混合持久化（基于 快照 + 日志 的持久化）
        1. 结合快照和日志的优点，取长补短，系统定期生成快照，并在两次快照之间，以日志形式记录增量操作
    2. 存储引擎实现
      1. 原地更新模型
        1. 数据以高度结构化的形式（如 B+ 树）存储在磁盘页（Page）中
        2. 当数据更新时，系统直接定位到磁盘对应页，将其读入内存修改，再写回原来的位置
        3. 这种模型高度依赖 预写日志（WAL，Write-Ahead Log），任何磁盘页的修改必须先记录到 WAL 并持久化，以保证崩溃恢复
        4. 如果系统发生故障，可以通过重放 WAL 来恢复未完成的操作
      2. 写时复制模型
        1. 不直接修改已写入的数据，而是将修改写入 新的数据块或新文件
        2. 在写入新数据时，同时记录 WAL（操作日志），保证崩溃恢复的完整性
        3. 数据写入完成后，再通过 替换/合并 的方式更新原来的文件或数据结构
      3. 需要注意的是，并非所有系统都必须使用 WAL，例如 Redis，其 RDB 快照机制允许在快照间隔内丢失部分数据，因此在写入快照时 不依赖 WAL。
    3. 操作系统交互
      1. 无论采用哪种持久化逻辑思想和存储引擎层模型，最终都必须通过操作系统将数据写入物理磁盘。
      2. 然而为了提高性能，操作系统引入了页缓存（Page Cache），这意味着应用程序的 write 调用默认并不保证数据立即落盘。
      3. 因此，必须采用特定机制来确保数据真正写入磁盘，从而保证持久化的可靠性。
        1. 强制刷盘（fsync 系统调用）
          1. 当需要确保关键数据绝对可靠时（例如 WAL），就要在 write 之后调用 fsync，将内存中的数据强制写入磁盘。
          2. 根据调用 fsync 的时机，强制刷盘可分为三类：
            1. 同步刷盘
              1. 每次写入操作完成后立即调用 fsync，确保数据即时落盘
            2. 间隔刷盘
              1. 定期调用 fsync，在一定时间间隔内批量刷盘，以兼顾性能和可靠性
            3. 操作系统自由安排
              1. 完全依赖操作系统决定何时刷盘，应用程序不显式调用 fsync
        2. 绕过缓存（直接 IO）
          1. 这是一种更底层的优化手段，允许应用程序绕过操作系统的页缓存，直接在自身的内存缓冲区和磁盘之间传输数据

Raft 协议概述

Raft 协议是一个强一致性协议，在三个维度上的选择如下：

  1. 维度一：同步复制
  2. 维度二：主从复制
  3. 维度三：基于日志的复制
    1. 从 Raft 协议的设计来看，本质是基于日志的复制。只要日志可以无限增长，并且新加入的节点能够完整同步所有历史日志，Raft 就能正常工作。
    2. 但在真实实现中，几乎不会只依赖日志复制，而是采用 快照 + 增量同步 的方式。这么做是因为：
      1. 存储空间有限：
        1. 志会随着集群的持续运行不断追加，如果不做处理，最终会耗尽存储
      2. 同步耗时过长：
        1. 新加入的节点，如果从零开始，需要从第一个日志条目起依次同步。
        2. 长时间宕机的节点，也必须从其停机点之后的日志开始追赶。
        3. 你想一想，如果系统写入量很大，比如数据库每秒产生 1,000 条写入，每条日志 1KB，那么每秒就是 1MB，一天就是 86.4GB，仅仅传输日志就会成为严重瓶颈。

Raft 协议核心角色

1. Follower（跟随者，默认）
Follower 是集群中的默认角色，主要职责包括：

  1. 不直接处理客户端的写请求
  2. 等待并执行 Leader 的指令
  3. 接收并持久化 Leader 的日志复制
  4. 响应 Leader 的心跳，如在超时未收到心跳，则转换为 Candidate 发起选举

2. Candidate（候选者）
Candidate 是竞选 Leader 的临时角色，主要职责包括：

  1. 当 Follower 长时间未收到 Leader 心跳时，会转为 Candidate，并向其他节点发起 RequestVote 请求以争取选票
    1. 若赢得多数选票，则晋升为 Leader
    2. 若选举失败，则可能再次进入 Candidate 状态，或回退为 Follower

3. Leader（领导者）
Leader 是集群中唯一负责处理写请求并向 Follower 复制日志的角色，整个集群在任意时刻最多只有一个 Leader。主要职责包括：

  1. 接收并处理客户端的写入请求
  2. 定期向 Follower 发送心跳以维持领导地位
  3. 将日志复制到各 Follower 节点并确保数据一致性

Raft 协议核心流程

1. Raft 启动与第一次选举 Leader
启动时，所有节点默认处于 Follower 状态，term 初始为 0，并各自设置一个 150–300ms 之间的随机选举超时时间。如果在超时时间内未收到来自 Leader 的心跳信号，该节点将从 Follower 状态切换为 Candidate，将 term 加 1，发起第一轮选举

一旦某个 Candidate 获得的票数超过半数（即 > N/2，这里的 N 指的是集群中的所有节点数，而非存活节点数，因此无需担心宕机会影响选票），就会当选为 Leader。这里有两个关键点需要注意：

  1. 选举必须赢得多数票
    1. 这一机制有效防止了脑裂问题，因为在网络分区时，只有一方能获得多数，从而成为唯一的 Leader
  2. 建议集群节点数为奇数
    1. 这样可以避免选举时多个 Follower 平票的情况，提高选举成功率
    2. 若没有任何一个 Candidate 获得多数票，则本轮选举失败。各节点会等待各自的选举超时器再次超时，随后将 term 再增加 1，并发起新一轮选举

若某个 Candidate 当选为 Leader，会立即向其他节点发送心跳信号。其他节点一旦收到来自新 Leader 的心跳，便会自动 “认输”，切换回 Follower 状态

这个全局递增的 term，可以理解为 “当前是第几届村长执政”。每个节点都会保存一个本地的 term，谁的 term 高，谁就拥有更高的优先级，从而避免 “过时的村长” 重新掌权

2. 数据写入与复制
Leader 负责整个系统的读写操作（由 Leader 负责读取，是为了避免脏读），这种设计虽然保证了强一致性，但也将压力集中在了 Leader 节点。

当 Leader 接受到一个写入请求后，会创建一个新的日志条目，Raft 要求包含当前的日志条目的索引号、当前任期号、客户端具体操作命令，例如：

索引 (Index) 任期 (Term) 命令 (Command)
1 1 set("name", "Alice")
2 1 set("age", "30")
3 2 increment("counter")
4 2 set("city", "New York")

Leader 首先会将该日志条目追加到本地日志中，常见的实现方式包括：

  1. 文件追加模式
    1. 这是最常见、最直观的方式，每个节点维护一个专用日志文件，新生成的日志条目会简单地追加到文件末尾
    2. 通常采用分段存储，当日志文件过大时，会开启新的文件，有助于后续的日志压缩与清理
    3. 多数实现会使用 WAL（Write-Ahead Log）机制，先将日志写入磁盘，再应用到内存中的状态机，以确保系统崩溃后日志仍然可恢复
  2. 基于键值对存储
    1. 某些实现会选择使用键值存储保存日志条目，其中索引号作为键，日志内容作为值
    2. 这种方式支持快速随机访问，且管理相对灵活

然后，Leader 并行地向所有 Follower 发送 AppendEntries RPC，请求它们复制这个新的日志条目，然后我们就进入 Raft 设计最精妙的地方：一致性检查：

  1. 在 AppendEntries RPC 中，Leader 不仅会带上新的日志条目，还会附带新日志条目前一个日志的索引 (prevLogIndex) 和任期 (prevLogTerm)
  2. Follower 收到请求后，会检查自己本地日志在 prevLogIndex 位置的条目，看其任期是否与 prevLogTerm 匹配
    1. 如果匹配
      1. 说明日志到此为止都是一致的，Follower 会安全地将新的日志条目追加到自己的日志中，并向 Leader 回复成功
    2. 如果不匹配
      1. 说明 Follower 的日志在此处与 Leader 有分歧，Follower 会拒绝该请求
      2. 当 Leader 收到 Follower 拒绝的响应后，它会向前回溯，找到与该 Follower 的日志共同的、一致的那个点，然后从该点开始，将后续的所有日志条目重新发送给 Follower，强制覆盖掉不一致的部分。
      3. Raft 协议是让我们使用日志进行记录，即便是刚上任的 Leader，也能够向前回溯，因为在投票前，节点会先检查候选人的日志是否至少和自己一样新，才会投票
      4. 但是在大多数情况下，我们其实还是会使用快照，如果我们使用了快照，我们最多也只能回溯到快照处，如果发现需要回溯到比这个快照更造，那就会判断 Follower 的日志是严重落后（Follower 长期下线、新节点加入）
      5. 在这种情况下，Leader 不会再尝试逐条回溯并同步日志，而是直接向这个 Follower 发送一个快照（这比回溯和同步大量的旧日志要高效得多）
  3. 所以一个典型的 AppendEntries RPC 包含以下关键字段：
    1. leaderId
      1. Leader 的节点 ID
      2. 用于 Follower 响应客户端请求时的重定向，它可以告诉客户端 “去找这个 Leader”
    2. term
      1. Leader 的当前任期号
      2. Follower 收到 AppendEntries RPC 后，会首先检查这个 term
        1. 如果 term 小于自己的任期号，说明这个 Leader 是一个 “过时” 的 Leader，Follower 会拒绝这个 RPC，并告知它当前的正确任期，促使它下台
        2. 如果 term 大于或等于自己的任期号，Follower 会接受这个 RPC，并更新自己的任期号。这是 Raft 确保集群中所有节点都在同一个 “时代” 工作的基本机制
    3. prevLogIndex
      1. 新日志条目前一个条目的索引号
      2. Follower 收到后，会检查自己的日志中是否存在索引号为 prevLogIndex 的日志条目
    4. prevLogTerm
      1. 新日志条目前一个条目的任期号
      2. 与 prevLogIndex 配合使用，提供一个双重校验，Follower 不仅要检查自己的日志中是否存在 prevLogIndex 处的条目，还要确保这个条目的任期号与 prevLogTerm 完全一致
      3. 如果两者都匹配，那么 Leader 和 Follower 的日志在这一点上就是一致的，可以安全地进行追加。如果不匹配，则说明日志存在分歧，需要进行修复。
    5. entries[]
      1. 要复制的日志条目数组
      2. Leader 要同步给 Follower 的新日志条目，通常只包含一个新条目，但在日志修复或优化时，也可能包含多个
    6. leaderCommit
      1. Leader 节点已提交的最高日志条目的索引号
      2. 当 Leader 收到大多数 Follower 对某个日志条目（比如索引号为 10）的成功复制响应时，它会立即更新自己的 commitIndex 为 10，并将该条目应用到自己的状态机中。
      3. Leader 会在下一次发送的 AppendEntries RPC（无论是新的日志条目还是心跳）中，携带这个更新后的 leaderCommit = 10。
      4. Follower 收到这个 RPC 后，才会将自己的 commitIndex 更新为 10，并且如果发现自己的日志中，索引小于或等于 leaderCommit 的条目还没被提交，它就会将这些条目标记为已提交
      5. 一旦日志条目被标记为已提交，Follower 就可以将其安全地应用到自己的状态机（state machine）中，从而更新自己的状态
      6. 所以我们知道了，Follower 收到日志后，只是将其持久化到本地，但不会立即执行，而是等 Leader 的下一次指令，这是为什么呢？这是因为只有 Leader 才有权决定一个日志条目是否已提交，Follower 必须被动地等待 Leader 的提交指令。

当 Leader 发现超过半数的 Follower 都已成功复制了某条日志后，该日志条目就被认为是 “已提交” (Committed) 的，它会立即更新自己的 commitIndex ，然后将该日志中的命令应用到自己的状态机，并向客户端返回成功

Leader 在后续的心跳中会通知所有 Follower 哪些日志已经被提交（心跳会发送不包含任何日志条目的 AppendEntries RPC），Follower 收到通知后也将相应的命令应用到自己的状态机

[!NOTE] 注意事项

  1. 你可能会疑惑，当 Leader 发现 超过半数 的 Follower 已成功复制某条日志时，该日志就被视为 已提交（Committed）。此时，Leader 会立即更新自己的 commitIndex，并将该日志中的命令应用到状态机。但是在极端情况下，若 Leader 尚未来得及发送下一次心跳，Follower 还未更新自己的 commitIndex、也未执行该日志时，Leader 突然宕机。此时会触发新一轮选举，新的 Leader 当选后，会不会因为部分 Follower 尚未执行这条日志而产生影响呢？
  2. 新 Leader 上任后，其 commitIndex 可能暂时停留在日志条目 A 之前的某个索引，此时它不会立即将日志条目 A 标记为已提交。
  3. 因为根据 Raft 的提交规则：
    1. 本任期的日志
      1. Leader 必须将日志复制到多数节点后才能提交
    2. 前任期的日志
      1. Leader 需要先在本任期提交一个后续日志条目，才能一次性将之前的日志条目标记为已提交
  4. 如果系统在此时没有新的写操作，那么自然不会产生新的日志条目。为了解决这个问题，Raft 规定：新 Leader 选举成功后，会立即在自己的日志中追加一个 no-op 日志条目，这个条目是本任期的第一个操作。
  5. 在提交这个 no-op 条目之前，Leader 会先执行该条目之前未提交的所有日志，确保它们被正确应用；然后再提交并执行这个 no-op 条目，从而完成本任期的初始化。

3. Leader A 宕机
假设集群中已经经过几次选举了，此时 A 是当前的 Leader（term=10），定期向 B、C、D、E 发送心跳信号。而如果 A 宕机后，B、C、D、E 都不再收到心跳信号，触发各自的选举超时定时器（150–300 ms 随机）

  1. 假如 B 最先超时未收到心跳
    1. B 超时在先（如 180 ms），检测到未收到心跳，便将 term 从 10 增加到 11，变为 Candidate。B 立即给自己投票（票数=1），并向 C、D、E 以及已掉线的 A 广播 RequestVote(term=11, candidateId=B, lastLog…)。
    2. C、D、E 收到 B 的请求后，发现请求中的 term=11 比自己当前的 term=10 更新，便更新 term → 11，切回 Follower 身份，检查 B 的日志无落后后，投票给 B。
    3. B 收到来自 C 和 D 的投票（B 自己 + C + D = 3 票），满足多数要求，成功当选为新 Leader（term=11）。
    4. B 立即向其他活跃节点（C、D、E）以及离线的 A 发送心跳 AppendEntries(term=11)。C、D、E 收到后重置选举定时器，保持 Follower 状态；
    5. 如果 A 之后恢复上线，它将收到 B 的心跳 RPC，并发现 term=11 比自己的 term=10 更新，于是更新自己的 term → 11，退回 Follower 状态，放弃原先的 Leader 身份。这样保证：一个处于旧 term 的节点无法参与当前任期的投票或发起选举，只有“意识到要进入新一届”的节点才能重新参选。
  2. 假如 B、C 几乎同时超时未收到心跳
    1. 若 B 和 C 的选举超时非常接近（如均为 180 ms 左右），两者几乎同时将 term 从 10 升为 11，变为 Candidate，并向其余节点广播 RequestVote(term=11)（B 向 A、C、D、E 发，C 向 A、B、D、E 发）。
    2. 假设 D 先收到 B 的请求，检查通过后将票投给 B，并更新 term → 11。
    3. 同时，E 先收到 C 的请求，也检查通过并投票给 C，更新 term → 11。
    4. 当 B 收到来自 C 的 RequestVote(term=11) 请求时，B 发现自己在当前 term（11）中已将票投给自己，因此拒绝给 C 投票。C 也是同理（只能投一次，投给自己就没办法投给别人了）
    5. B 最终得票为自己 + D = 2，C 得票为自己 + E = 2，均未获得多数 → 选票分裂（split vote），本轮选举失败。
    6. B、C、D、E 的选举超时器重新启动，等待下一轮触发。再次进入超时的节点将把 term 增加到 12 并发起新一轮选举，如此循环，直到有人获得多数票成功当选。term 会随着失败轮次不断累加。

由于会检查日志是否落后之后再去投票，而且 Raft 协议还是多数成功，就直接杜绝了日志落后的节点“混进多数”并被选为 Leader 的可能性。

4. A 节点恢复
A 节点重启后，term 仍是 10。当它收到新 Leader（如 D）发来的心跳 AppendEntries RPC 中 term=11 后，立刻发现对方的 term 更新，于是更新自己的 term → 11，退回 Follower 身份，并且进行日志同步，重新与集群保持一致。

分布式事务

事务概述

在数据库里，事务具有四个关键特性（ACID）

  1. 原子性（A）
    1. 事务中的所有操作要么全部执行成功，要么全部执行失败回滚。
    2. 对于数据库来说，事务是一个不可分割的最小操作单元。
  2. 一致性（C）
    1. 事务执行前后，数据库都处于一致的状态。
    2. 也就是说，事务执行前后，数据库的状态应该满足所有的定义约束、触发器、级联操作等。
  3. 隔离性（I）
    1. 事务的执行不会受到其他事务的干扰。
    2. 多个事务并发执行时，一个事务的中间状态对其他事务是不可见的。
  4. 持久性（D）
    1. 事务一旦提交，其结果就永久保存到数据库中。
    2. 即使系统发生故障，事务的结果也不会丢失。

分布式事务概述

分布式事务是指：把事务的概念，从单机数据库扩展到了跨多个微服务、多个数据库的场景。

过去，在单体应用的电商系统里，订单、支付、库存都塞在一个大项目，数据也都放在同一个数据库里。这样一来，所有操作都在单机环境的同一个数据库里完成，事务自然很简单。例如在 MySQL 里只要 BEGIN … COMMIT，数据库就能帮你搞定一致性。

后来大家觉得“大一统”的架构太笨重，就拆成了微服务：有 订单服务、库存服务、支付服务。每个服务不仅代码独立，还各自拿着自己的数据库，技术选型五花八门（这边用 MySQL，那边用 MongoDB，甚至还有 Redis 出来凑热闹）

可是问题也随之出现，在电商下单场景里：

  1. 订单服务要插入一条订单记录（向数据库 A）
  2. 库存服务要扣减一条库存（向数据库 B）
  3. 付服务要完成扣款（向数据库 C）

这三件事必须保证 要么全部成功，要么全部失败，否则就会出现 “订单生成了但没扣钱”、“扣了钱却没库存” 的大灾难。

但是数据库 A 有自己的本地事务，数据库 B 有自己的本地事务，数据库 C 也有自己的本地事务，他们各自为政，互不干涉，却没有一个 “老大哥” 能帮你全局管控。

于是，这正是 分布式事务 要解决的问题：在分布式场景下，每个服务都能开启自己的 本地事务，只保证自己的一亩三分地。而分布式事务的任务，就是把这些零散的本地事务“绑在一起”，形成一个 逻辑上的整体事务，让它们的命运合二为一。

强一致性分布式事务

2PC 提交协议

2PC 核心角色

1. 协调者（Coordinator）
负责事务的发起、管理和最终决定（提交 / 回滚）。

2. 参与者（Participant）
一般是数据库、消息队列节点，实际执行事务操作。

2PC 核心流程
阶段一：准备阶段

在这个阶段，协调者需要确认所有参与者都 “有能力” 提交事务。于是，它会向所有参与者发送 Prepare 请求，就好比在问：“你们是否能提交事务？”

参与者收到 Prepare 请求后，开始执行事务操作并锁定相关资源，但不会提交事务：

  1. 执行事务操作，在这个过程要将相关资源加锁（行锁、表锁），确保这批数据在最终提交前不会被其他事务破坏
  2. 并且将事务的 Redo Log 和 Undo Log 写入本地的持久化日志文件

如果一切顺利，参与者就向协调者返回 YES，表示自己已经做好提交的准备。若中途遇到异常（如约束条件失败、磁盘空间不足等），则返回 NO。

[!NOTE] 注意事项

  1. 在这里，Prepare 阶段通常依赖我们 “四维度理论” 所强调的底层技术来实现
  2. 其他地方中的多次 “交互” 本质上只是一条网络信号的传递
阶段二：提交阶段

协调者在收到所有参与者的响应消息后，会根据反馈结果通知各参与者：执行 commit 提交 或 rollback 回滚 操作。

1. 事务提交
当所有参与者均返回 YES 时，协调者发起正式的提交流程：

  1. 协调者向所有参与者发送 commit 请求。
    1. 参与者接到请求后，正式提交事务，并释放事务执行期间锁定的资源（行锁、表锁）和占用的数据库连接
  2. 提交完成后，参与者向协调者返回 ACK
  3. 协调者收到所有参与者的 ACK 后，确认整个事务提交完成

2. 事务回滚
若任意一个参与者返回 NO，或在超时时间内未返回响应，则协调者触发回滚流程：

  1. 协调者向所有参与者发送 rollback 请求
  2. 参与者根据第一阶段保存的 undo 信息 执行回滚，并释放事务执行期间锁定的资源（行锁、表锁）和占用的数据库连接
  3. 回滚完成后，参与者向协调者发送 ACK 确认消息
  4. 协调者收到所有参与者的 ACK 后，确认事务已取消

[!NOTE] 注意事项

  1. 在 阶段二 中，如果协调者迟迟没有收到某个参与者返回的 ACK，根据严格的 2PC 协议规范，协调者会进入阻塞等待状态，等待该参与者恢复正常，或由管理员人工干预
  2. 在工程实践中，很多实现会加入重试机制，但依旧无法改变阻塞的事实
2PC 优缺点

协调着也有可能卡死，参与者也有可能卡死

  1. 缺点
    1. 性能问题
      1. 在分布式场景下，2PC 严重依赖多次网络通信
      2. 而且在第二阶段，资源会被锁定，其他节点想要访问这部分资源，只能阻塞等待
        1. 这也是 2PC 不适合长事务的原因：事务持续越久，锁定的资源越多，等待的请求越多，系统吞吐量就会迅速下降
      3. 此外还会长期占用数据库连接
      4. 在高并发场景（成千上万事务同时执行）下，使用 2PC 很容易导致系统 “卡爆”
      5. 因此，在对响应速度要求极高的场景（如电商秒杀、社交消息流），几乎不会采用 2PC
    2. 可靠性问题
      1. 协调者（TM）是整个流程的核心，存在单点故障问题
      2. 尤其是在提交阶段，一旦 TM 宕机，资源会被一直锁定，数据库连接也会被持续占用
      3. 尽管可以通过重新选举产生一个新的协调者，但这并不能解除因原协调者宕机而导致的参与者长期处于阻塞状态的问题
    3. 数据一致性问题
      1. 在提交阶段中，如果协调者已经向部分参与者下发了 commit，但由于网络中断，另外一些参与者没有收到指令，就会导致一部分节点提交成功、另一部分节点停留在未提交状态，最终造成数据不一致。
      2. 2PC和3PC都无法保证数据绝对的一致性，一般为了预防这种问题，可以添加一个报警，比如监控到事务异常的时候，通过脚本自动补偿差异的信息。

3PC 提交协议

[!NOTE] 注意事项

  1. YES/NO 与 ACK 的区别
    1. YES/NO
      1. 用于协调者的询问阶段，仅表示参与者的意向或能力，不会实际修改资源
    2. ACK
      1. 用于指令执行阶段，当协调者下达操作命令后，参与者完成执行并以 ACK 确认结果。
    3. nack
    4. error
3PC 概述

三阶段提交（3PC）被设计出来的唯一目的，就是为了解决两阶段提交（2PC）的一个致命缺陷：协调者单点故障导致的参与者永久阻塞问题。

3PC 的出现主要是为了解决以下问题：

  1. 2PC 阻塞问题
    1. 在 2PC 中，一旦参与者在准备阶段返回了 YES，它就不得不锁定公共资源并占用数据库连接，直到协调者发来最终指令。
    2. 如果此时协调者宕机，参与者将一直无法释放资源，陷入阻塞状态。
  2. 2PC 数据不一致问题
    1. 在提交阶段中，如果协调者已经向部分参与者下发了 commit，但由于网络中断，另外一些参与者没有收到指令，就会导致一部分节点提交成功、另一部分节点停留在未提交状态，最终造成数据不一致。

而 3PC 通过引入一个额外的 PreCommit（预提交） 阶段，并加入 超时机制，来缓解这些问题

[!NOTE] 注意事项

  1. 虽然 3PC 缓解了 2PC 的部分问题，但它也带来了更高的通信成本和实现复杂性，且依旧具有性能问题和数据一致性问题
  2. 因此，在现代分布式系统中，人们更倾向于使用 2PC 而非 3PC，甚至更倾向于采用共识协议来解决这部分问题
3PC 核心流程
阶段一：CanCommit（准备阶段）

协调者向所有参与者发送 CanCommit 请求，询问它们是否都 “有能力” 提交事务

参与者在收到 CanCommit 请求后，会先对自身环境进行检查（例如：本地数据是否满足约束、资源是否充足等），此时不会真正执行事务操作或锁定资源

检查无误后，它会向协调者返回 YSE。若检查失败，则返回 NO

阶段二：PreCommit (预提交阶段)

协调者根据参与者在阶段一的反应情况，来决定如何进行 PreCommit 操作，有以下两种可能：

1. 事务预提交
当所有参与者均返回 YES 时，协调者向所有参与者发送 PreCommit 请求。

参与者收到 PreCommit 请求后，开始执行事务操作并锁定相关资源，但不会提交事务（这步才真正等同于 2PC 的第一阶段）

执行成功后，参与者向协调者返回 ACK。若执行失败，则返回 NO

2. 事务中断
若任意一个参与者返回 NO，或在超时时间内未返回响应，协调者向所有参与者发送 Abort 请求

参与者收到 Abort 请求后，直接中断事务

[!NOTE] 注意事项

  1. 在 3PC 中引入 “超时机制”，主要体现在第二和第三阶段。如果参与者在规定时间内没有收到协调者的指令，就可以自主作出决策，默认将事务中止，从而避免因协调者故障而无限期阻塞。
阶段三：DoCommit（提交阶段）

协调者根据参与者在阶段二的反应情况，来决定如何进行 DoCommit 操作，有以下两种可能：

1. 事务提交
当所有参与者均返回 ACK 时，协调者向所有参与者发送 DoCommit 请求。

参与者收到 DoCommit 请求后，正式提交事务，并释放事务执行期间锁定的资源（行锁、表锁）和占用的数据库连接

提交完成后，参与者向协调者返回 ACK

协调者收到所有参与者的 ACK 后，确认整个事务提交完成

2. 事务中断
若任意一个参与者返回 NO，或在超时时间内未返回响应，协调者向所有参与者发送 Abort 请求

参与者收到 Abort 请求后，根据阶段二记录的 Undo 日志执行回滚，并释放事务执行期间锁定的资源（行锁、表锁）和占用的数据库连接

中断完成后，参与者向协调者返回 ACK

协调者收到所有参与者的 ACK 后，确认整个事务中断完成

[!NOTE] 注意事项

  1. 在 3PC 中引入 “超时机制”，主要体现在第二和第三阶段。如果参与者在规定时间内没有收到协调者的指令，就可以自主作出决策，默认将事务中止，从而避免因协调者故障而无限期阻塞。

最终一致性分布式事务

TCC 模式

TCC 概述

TCC 的出现主要是为了解决以下问题：传统强一致分布式事务协议（如 2PC、3PC）实现复杂、性能低下，而且常常导致资源长时间锁定

而 TCC 通过在业务层进行灵活的编排，将锁粒度和持锁时间最小化，从而以更高效的方式实现最终一致性

TCC 核心流程
阶段一：Try 阶段（资源预留）

假设你在 Mall 商城里买了一台手机，这个操作涉及到三个服务：

  1. 订单服务
    1. 创建订单
  2. 库存服务
    1. 扣减库存
  3. 支付服务
    1. 扣除余额

在 Try 阶段，事务协调者会调用各服务的 Try 接口，各服务先执行资源预留，但尚未进行最终确认：

  1. 订单服务
    1. 在数据库插入一条订单记录
    2. 状态设为 Pending（待确认）
  2. 库存服务
    1. 从 available_quantity（可用库存）中扣减对应数量
    2. 在 reserved_quantity（冻结库存）中增加相应数量
  3. 支付服务
    1. 从 available_balance（可用余额）中扣减相应金额
    2. 在 reserved_balance（冻结金额）中增加对应金额

[!NOTE] 注意事项

  1. 常见的资源预留实现方式可以分为两类：
    1. 库存预留
      1. 数据库设计：
        1.  建立两类字段或表，available_quantity（可用库存） 和 reserved_quantity（冻结库存）
      2. Try
        1. 扣减 available_quantity，并相应增加 reserved_quantity
      3. Confirm
        1. 直接从 reserved_quantity 扣除，完成真正消耗
      4. Cancel
        1. 回滚操作，即增加 available_quantity，并减少 reserved_quantity
    2. 优惠卷预留
      1. Try
        1. 给记录加上 “冻结” 标志（如 Locked=Y），并通常设置过期时间（例如 10 分钟）
        2. 在冻结期间，该优惠券对其他用户不可见
      2. Confirm
        1. 将标志更新为 “已领取”
      3. Cancel
        1. 将标志恢复成原始状态（解除冻结）
  2. 状态预留不会像 2PC、3PC 那样锁住数据库资源，而是通过业务字段来标记和控制资源的占用状态
阶段二：Confirm/Cancel 阶段

事务协调者根据各服务在阶段一的反应情况，来决定如何进行阶段二的操作，有以下两种可能：

1. Confirm 阶段
当所有服务都 Try 成功，事务协调者会调用各服务的 Confirm 接口：

  1. 订单服务
    1. 把 Pending 修改为 Confirmed
  2. 库存服务
    1. 直接从 reserved_quantity（冻结库存）扣除，完成真正消耗
  3. 支付服务
    1. 直接从 reserved_balance（冻结金额）扣除，完成真正支付

2. Cancel 阶段
当任一服务在 Try 阶段有任何失败，事务协调者会调用各服务的 Cancel 接口：

  1. 订单服务
    1. 把订单取消甚至直接删除
  2. 库存服务
    1. 解锁刚才冻结的库存
  3. 支付服务
    1. 解锁刚才冻结的金额

[!NOTE] 注意事项

  1. 幂等保护
    1. Confirm 和 Cancel 必须设计成安全的重复调用
    2. 我们可以通过事务 ID（分布式 ID）来记录，确保不会重复结算
  2. 允许空回滚
    1. 空回滚是指：一个服务根本没成功执行过 Try，就被要求执行 Cancel
    2. 造成空回滚的原因包括：
      1. 网络抖动
        1. 事务协调器的 Try 请求因为网络抖动超时未到达服务，但协调者认为失败了，就会触发 Cancel
      2. 服务宕机
        1. 事务协调器的 Try 请求因为服务宕机无法未到达服务，但协调者认为失败了，就会触发 Cancel
    3. 我们可以在各服务维护一个事务日志表，对每个事务 ID 记录 Try/Confirm/Cancel 状态
      1. 在 Cancel 的时候，如果查不到事务日志记录，说明 Try 根本没执行过，就直接返回（即空回滚），不能真的去执行操作
  3. 防悬挂控制
    1. 悬挂是指：Cancel 比 Try 先到达，然后 Try 又 “姗姗来迟”
    2. 我们可以在处理 Try 时，先检查事务日志表 ，查看这个事务 ID 是否已经是 Cancel 状态
    3. 如果已经是 Cancel 状态，说明 Cancel 已经到达过，就拒绝执行 Try

Saga 模式

Saga 概述

Saga 是指：将一个大事务拆分为一系列可以独立提交的局部事务，每个局部事务成功后立即提交，避免长时间持有锁与资源，如果中途出现失败，则按既定顺序依次执行对应的补偿操作，将已完成的动作回滚，从而保证整体达到最终一致性

只要每个小事务都是自洽的（要么提交，要么补偿），那么整个业务过程在逻辑上就是完整、可恢复的。

Saga 恢复策略
  1. 向前恢复（补偿事务）
    1. 一旦执行过程中出错，就回滚之前已经成功的局部事务
    2. 这个恢复策略需要每个局部事务必须有一个对应的补偿，来导致状态回退
    3. 但有些操作可能不好补偿，比如发短信都已经发出去了，很难 “收回”
  2. 向后恢复（端点续跑）
    1. 出错后不撤销之前成功的事务，而是不断重试失败的那一步，直到成功为止
    2. 这个恢复策略假设所有局部事务最终一定能成功，不需要补偿
Saga 核心流程
编排模式（Orchestration）

在 编排式 Saga（Orchestration Saga） 中，会有一个中心控制器 （Saga Orchestrator），它负责统一调度所有子事务及其补偿事务：

  1. 正常执行
    1. Orchestrator 按顺序依次调用各个局部事务：Txn1 → Txn2 → Txn3 → …
    2. 每个事务执行成功后立即提交，不阻塞后续流程
  2. 异常回滚
    1. 一旦有事务失败，Orchestrator 会主动触发补偿逻辑：Cn(n-1) → Cn(n-2) → …
舞会模式（Choreography）

在 协作式 Saga 中，不再有中心控制器，而是每个服务通过事件驱动自发参与事务流程：

  1. 正常执行
    1. 当某个服务完成自己的局部事务后，会发布一个事件（如 OrderCreate）
    2. 下一个相关服务订阅并监听该事件，执行自己的局部事务并继续发布事件（如 InventoryReserved）
    3. 整个 Saga 事务就这样靠事件链条一步步推进
  2. 异常回滚
    1. 如果某个服务在执行时发生错误，它会发布一个补偿事件（如 PaymentFaile）
    2. 订阅该补偿事件的服务按照约定，自行触发补偿逻辑
    3. 由此实现链式的按需回滚

本地消息表

本地消息表概述

本地消息表 的出现主要是为了解决以下问题：
在分布式系统中，我们常通过消息队列来传递事件（因为消息队列优势很多，详见笔记：MQ 概述）。

但如果采用 “先写数据库，再发消息” 这种常见流程，就会产生一致性问题：

  1. 如果 “写库成功，但发消息失败”，队列中就没有事件，消费者无法感知
  2. 如果 “发消息成功，但写库失败”，消费者就会处理到一条不存在的业务（脏数据）
  3. 这些问题都会导致数据不一致

而 本地消息表 通过先将消息写入本地消息表，随后由后台定时任务扫描该表，并将消息推送到消息队列。这样，即使因宕机等原因导致消息一时未能发送，消息记录仍然安全地保存在本地，后续任务会对其进行扫描和补发，从而保证最终一致性。

Seata 不只是单一的 2PC，而是封装了多种模式，适合不同业务需求：

  1. AT 模式（Automatic Transaction）

    • 最常用，适合关系型数据库。

    • 原理：Seata 拦截 SQL，自动生成 Undo Log，失败时自动回滚。

    • 对应用开发者几乎无感知。

  2. TCC 模式（Try-Confirm-Cancel）

    • 业务层自己定义三步操作：Try 预留资源、Confirm 确认提交、Cancel 回滚。

    • 灵活，但业务代码要额外编写。

  3. Saga 模式

    • 把长事务拆成一系列本地事务，如果中间失败，就用补偿操作撤销。

    • 适合跨服务、长流程场景。

  4. XA 模式（标准两阶段协议）
