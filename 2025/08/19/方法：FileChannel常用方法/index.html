方法：FileChannel 常用方法

Last updated on 2025-08-20T10:01:36+08:00

前言

虽然 FileChannel 可以通过 FileInputStream、FileOutputStream、RandomAccessFile 的 getChannel() 方法获取：

  1. 使用 FileInputStream 获取的是读通道
  2. 使用 FileOutputStream 获取的是写通道
  3. 使用 RandomAccessFile 获取的是读写通道

他们的 API 方法列表是完全一致的，区别在于他们只能调用符合权限的方法，否则会抛异常。

  1. 例如通过 FileInputStream 获取的读通道如果调用 write()，会抛出异常
  2. 例如通过 FileOutputStream 获取的写通道如果调用 read()，同样会抛异常
  3. 而使用 RandomAccessFile 获取的读写通道则没有这些限制，读写操作都支持

实例方法

int read(ByteBuffer)

  1. 一定要牢记这两个流程：
    1. 程序读数据：网络 / 文件 ➔ 数据 ➔ Channel ➔ Buffer ➔ 程序
    2. 程序写数据：程序 ➔ 数据 ➔ Buffer ➔ Channel ➔ 网络 / 文件
  2. 如果你发现文件内容莫名消失，原因可能是你创建的 FileOutputStream 没有开启追加模式，只需在创建时设置为追加模式即可：

跳过 JVM 堆内存，让你的 Java 程序直接在内核的 Page Cache 上读写文件

回想一下传统 read() 的流程：
硬盘 -> Page Cache -> [**CPU拷贝**] -> JVM 堆内存 (byte[])

再回想一下传统 write() 的流程：
JVM 堆内存 (byte[]) -> [**CPU拷贝**] -> Page Cache -> 硬盘

这个魔法做了什么？

  1. 不分配 JVM 堆内存: 它不会在你的 Java 堆里创建一块巨大的内存区域来装文件数据。
  2. 建立虚拟内存映射: 它会在你的 Java 进程的虚拟地址空间中，找到一块连续的、大小为 size 的区域。
  3. 直接指向 Page Cache: 然后，它将这块虚拟地址直接映射到内核空间的 Page Cache 中，而 Page Cache 本身又映射着磁盘上的物理文件。

使用 MappedByteBuffer 的读写流程（对比传统 I/O）

读取流程 (get() a MappedByteBuffer)

  1. 第 1 步: 调用 map()

    • 你调用 fileChannel.map(...)。
    • 内核在你的进程虚拟地址空间创建一块映射区，指向文件的 Page Cache。
    • 注意：此时数据并未从硬盘加载到内存！ map() 方法的返回非常快，它只是建立了一个“约定”。

  2. 第 2 步: 第一次访问内存 (mbb.get())

    • 你的 Java 代码第一次尝试从 MappedByteBuffer 中读取一个字节。
    • CPU 尝试访问这个虚拟内存地址，但发现这块内存还没有对应的物理内存页（数据还没加载）。
    • 这会触发一个**缺页中断 (Page Fault)**。

  3. 第 3 步: 内核处理缺页中断

    • CPU 控制权从用户态切换到内核态。
    • 内核捕获到这个中断，知道是时候去加载数据了。
    • 内核命令 DMA 将文件对应的数据块从硬盘拷贝到 Page Cache 中。
    • 【关键】：数据加载完成！

  4. 第 4 步: 返回用户态，读取成功

    • 内核将 Page Cache 的物理地址与之前那个虚拟地址关联起来。
    • CPU 控制权从内核态切换回用户态。
    • mbb.get() 操作现在可以成功地从内存中读取数据了，因为数据已经在它“应该在”的地方了。

与传统 I/O 对比：

  • 少了一次拷贝: 数据只经历了一次 硬盘 -> Page Cache 的拷贝。完全没有 Page Cache -> JVM 堆内存 这第二次拷贝。
  • 更少的上下文切换: 一旦数据被加载到 Page Cache，后续的 get() 操作都只是纯粹的内存访问，不会再触发系统调用和上下文切换，直到你访问到尚未加载的区域。

写入流程 (put() into a MappedByteBuffer)

写入流程更加简单直接。

  1. 第 1 步: 调用 map()

    • 同读取，建立内存映射。

  2. 第 2 步: 写入内存 (mbb.put(data))

    • 你的 Java 代码调用 mbb.put()。
    • 这个操作会直接将数据写入到 Page Cache 中（因为 MappedByteBuffer 就直接代表着那块 Page Cache）。
    • 被写入的 Page Cache 区域会被内核标记为“脏页”。

  3. 第 3 步: 异步刷盘

    • 和传统 write 一样，内核会在它认为合适的时机（或你主动调用 mbb.force()），将这些“脏页”异步地刷写回硬盘。

与传统 I/O 对比：

  • 少了一次拷贝: 数据只经历了一次 你的代码 -> Page Cache 的拷贝。完全没有 JVM 堆内存 -> Page Cache 这次拷贝。你的 put 操作就像是直接在内核的缓存区上操作。

哦～你问到 MappedByteBuffer map(…)，这可是 Java NIO 里非常有意思的功能：内存映射文件（Memory-Mapped File） ✨

它的威力在于：
👉 把文件（或文件的一部分）直接映射进内存，当你操作缓冲区时，就等价于在直接修改/读取文件。

2. 参数含义

  • mode：映射模式（FileChannel.MapMode 的三种）：

    • READ_ONLY：只读映射（尝试写会抛异常）
    • READ_WRITE：读写映射（改缓冲区 = 改文件）
    • PRIVATE：拷贝时写（Copy-On-Write）：缓冲区修改时不会写回文件，只在内存里有副本

  • position：要映射的文件起始位置（偏移量，字节为单位）

  • size：映射的区域大小（position + size <= file.size()，且不能超过 2GB）

返回值：
👉 一个 MappedByteBuffer，操作它就是操作文件。

4. 跟普通 IO 的区别

  • 普通的 read() / write() → 你 自己搬字节
  • map() → 操作系统直接把文件的一部分映射进进程地址空间，你访问内存就像访问数组一样，本质上 OS 在幕后负责 IO。

特点：

  • 优点：速度极快，尤其对大文件的随机读写。
  • 缺点：
    • 受限于 JVM 地址空间 → 单次映射最多 2GB（因为 ByteBuffer 下标是 int）。
    • 释放内存映射不一定及时（要等 GC 回收 DirectBuffer，或者用一点 trick 提前清理）。
    • 写的时候不会更新 FileChannel.position()，因为这是直接在内存里改数据。

5. 用途

  • 处理超大文件（比如日志、数据库数据文件、二进制索引等）。
  • 实现高性能文件读写，避免用户空间和内核空间的数据拷贝。
  • 任意位置随机访问（不需要自己调用 channel.position(x) + read/write）。

6. 小比喻 📖

  • 普通 read/write：你拿个小水桶，每次去仓库舀一点米回来。
  • map()：直接把仓库那一块地铺到你家厨房，伸手就能抓米，下锅就能炒。

✅ 总结一句
map(mode, position, size) 会返回一个 MappedByteBuffer，操作它就等于操作文件里 [position, position+size) 这一段数据。

  • mode 控制可读可写
  • position 指定映射起点
  • size 指定映射范围
    常用于 超大文件随机访问 & 高性能 IO。

🔹 5. 文件拷贝 / 传输

  • long transferTo(long position, long count, WritableByteChannel target)
    把文件的一段内容直接“零拷贝”到另一个通道（通常是网络 SocketChannel），高效地进行大文件传输。

  • long transferFrom(ReadableByteChannel src, long position, long count)
    从另一个通道拷贝数据写入到文件。

💡特点：比自己写循环 read/write 更快，因为使用了操作系统的底层优化（零拷贝）。

🔹 6. 锁定文件

  • FileLock lock()
    锁定整个文件，独占模式。

  • FileLock lock(long position, long size, boolean shared)
    锁定文件的部分区域，可以指定共享锁 (shared=true, 允许读) 或独占锁。

  • FileLock tryLock()
    非阻塞版本，抢不到锁直接返回 null。

💡特点：保证在多进程/多线程写文件时不会互相踩踏。

🔹 7. 关闭通道

  • close()
    释放资源、关闭文件。一定要记得调用，或者使用 try-with-resources 自动化管理。
