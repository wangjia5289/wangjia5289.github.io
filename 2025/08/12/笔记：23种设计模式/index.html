3.1. 单一职责原则（SRP）

单一职责原则是指：一个类应该有且只有一个引起它变化的原因。简单来说，如果多个功能因为同一个 “原因” 而发生变化，它们可以放在同一个类中。如果是不同的 “原因”，则应将它们拆分到不同的类中，这里的 ”原因“ 通常是指：

  1. 一个业务场景
  2. 一个具体角色

1. 你能用一句话清晰地、无歧义地描述这个类的职责吗？（业务场景）
我们的这个类负责管理用户的属性，并且还负责数据库的持久化，同时还负责数据展示。

当我们的描述中，出现了类似于同时、并且等字眼的时候，就要知道可能会违背单一职责原则了，一个职责清晰的类，描述通常应该非常简洁，例如：

  1. 这是一个负责管理用户属性的类
  2. 这是一个负责数据库持久化的类
  3. 这是一个负责数据展示的类

2. 什么角色会要求我修改这个类？（具体角色）
这是一个非常有效的思维，想象一下你的类：

  1. 如果管理用户属性发生变化：
    1. 比如，修改用户名长度的验证规则（changeUsername）
    2. 这通常是由产品经理或业务分析师驱动的
  2. 如果数据库持久化发生变化
    1. 比如，从 MySQL 数据库迁移到 MongoDB
    2. 这通常是由数据库管理员或架构师驱动的
  3. 如果数据展示发生变化
    1. 比如，从生成 HTML 报告变为生成 JSON 数据
    2. 这通常是由前端开发或 API 设计者驱动的

所以我们发现了，产品经理、业务分析师、数据库管理员、架构师、前端开发这几个完全不同的角色，都有可能让我们去修改同一个 Employee 类，这就是典型的违反了单一职责原则

3. 这个类的 import 列表是不是看起来很 ”分裂“？
假如这个类的顶部 import 语句，同时出现了：

  1. 处理数据库的包，如 java.sql.*
  2. 处理网络请求的包，如 org.apache.http.*
  3. 处理 UI 元素的包，如 javax.swing.*
  4. 处理文件操作的包，如 java.io.*

那么它几乎 100% 违反了单一职责原则。这说明这个类横跨了应用的多个层次（持久化层、业务逻辑层、表现层），是一个典型的“上帝类”（God Class）

当我们出现上述问题的时候，就需要考虑重构这个类了。在项目初期或简单应用中，将几个关联紧密的职责放在一起可能是可以接受的，随着业务复杂度的增加，当一个类开始因为不同原因而频繁被修改时，就是重构和拆分它的最佳时机

3.2. 开闭原则（OCP）

3.2.1. 开闭原则概述

开闭原则是指：软件实体（类、模块、函数等）应该对扩展开放，对修改关闭。这句话听起来很矛盾，我们把它拆开来理解：

  1. 对扩展开放
    1. 当系统需要增加新功能时，我们可以通过增加新代码（新的类、新的模块）来实现，而不是去动老代码
  2. 对修改关闭
    1. 一旦一个类或模块开发完成、测试通过，就应该尽可能地保持稳定，不应再去修改它的内部代码。
    2. 这是因为修改老代码很容易引入新的 Bug，影响已有的稳定功能。

  1. 这里有一个常见的误区，就是说类写好之后，再去动他就是违反了开闭原则，这是不正确的
  2. 如果你在类里新增了一些方法，而且这些方法不会影响原有方法的逻辑，那么这依然符合开闭原则，不过有两点需要注意：
    1. 即使没直接改原方法，如果新增方法需要访问或修改类的内部状态，还是有可能引入风险，导致原功能被间接影响
    2. 如果新增方法跟这个类的核心职责无关，那可能会违背单一职责原则，虽然没违反开闭原则
  3. 虽然不违反开闭原则，但我们更希望能够在外部扩展，而不是直接在类里加内容，这样能减少类膨胀

3.2.2. 开闭原则实现

3.2.2.1. 开闭原则实现核心

实现开闭原则的核心就是：抽象（接口、抽象类） + 多态（向上转型、向下转型），即面向抽象编程，而不是面向具体编程

  1. 接口
    1. 接口是用来定义行为规范的
    2. 当实现类之间没有太多共同代码，只是需要遵守同一个行为规范时，推荐使用接口
  2. 抽象类
    1. 当子类之间有大量通用代码，且你想定义一个固定的算法骨架（模板）时，推荐使用抽象类

[!NOTE] 注意事项

  1. 为什么在定义行为规范时，要使用接口而不是抽象类？
    1. 如果使用抽象类定义行为规范，子类必须继承它，而类的继承是单一的，即子类只能有一个父类，这就限制了扩展性；
    2. 相比之下，接口支持多实现，更加灵活。
  2. 为什么在实现通用代码时，要使用抽象类，而不是接口？
    1. 接口中虽然提供了 default 方法，但它的诞生是为了解决历史遗留的兼容性问题，而不是为了取代抽象类的作用：
      1. Java 8 之前
        1. 接口一旦发布，就不能随便加方法
        2. 因为如果加了新方法，所有已经实现该接口的类都要改代码，不然编译不过
        3. 对于已经在广泛使用的库和框架，如果接口贸然新增方法，后果会非常严重，几乎等同于 “天塌了”
      2. Java 8 之后
        1. 允许在接口中新增方法的同时，给一个默认实现，这样老的实现类不用动，也不会编译报错
        2. 简单来说，其实 default 方法本质上是为了接口的“向后兼容”，让接口也能适度演化，而不破坏已有实现类，而不是说去取代抽象类
    2. 抽象类比接口提供了更强大的基础支持结构
      1. 接口中不能持有实例变量、实例常量、静态变量，只能定义静态常量（static final）
      2. 接口中不能定义构造方法，因此无法控制子类的初始化过程

[!NOTE] 注意事项

  1. 虽然使用具体类加多态也能实现开闭原则，但不推荐。可以用普通类作为“策略基类”，让子类继承并重写方法，但这样抽象程度低，外部依赖的是具体类而非抽象类，容易违反依赖倒置原则。
  2. 接口与抽象类通常是配合使用的，而不是相互隔离。很多框架都采用这种组合：例如 List（接口）→ AbstractList（抽象类，实现大部分通用方法）→ ArrayList（具体类）。

3.5. 依赖倒置原则（DIP）

依赖倒置原则是指：高层模块不应该依赖低层模块，二者都应该依赖抽象；抽象不应该依赖细节，细节应该依赖抽象。

3.6. 迪米特原则（LKP）

迪米特原则的定义是：一个对象应当对其他对象有尽可能少的了解，只与直接的朋友通信，不与陌生的对象发生耦合。简单来说，中调用方法时，应优先调用：

  1. 本类自己的方法
  2. 本类中引用类型的常量、变量、方法参数或局部变量的方法

不应调用链过长的对象，例如 a.getB().getC().doSomething()。这里的 C 对 A 来说是 “陌生人”，违反了迪米特法则。

4. 创建型模式（5 种）

4.1. 单例模式

4.1.1. 单例模式概述

单例模式是指：保证一个类在整个系统中只有一个实例，并提供一个全局访问点来获取该实例

单例模式主要解决的问题是：在整个系统中，有些类只需要一个实例即可，例如系统配置类、日志管理器、线程池或数据库连接池等。如果不限制实例数量，可能会造成资源浪费或状态不一致。

4.1.2. 单例模式实现

4.1.2.1. 饿汉式

饿汉式是指：在类加载的时候就创建实例

其优点是：

  1. 线程安全

其缺点是：

  1. 不支持懒加载
  2. 类加载速度变慢
  3. 如果实例一直未被使用，会造成内存浪费（尤其是实例占用资源很大时）
4.1.2.2. 懒汉式

懒汉式是指：只有当用户第一次通过全局访问点获取实例时，才会创建实例

其优点是：

  1. 懒加载
  2. 线程安全

其缺点是：

  1. 第一次调用全局访问点时，有性能延迟
  2. 由于 synchronized 操作，全局访问点并发度很低，性能较差
4.1.2.3. 双重检验

双重校验是指：第一次先判断实例是否为空，为空时再加锁（保证性能较为高效）。加锁后再进行第二次判断，确认实例尚未被其他线程创建后才创建新实例，从而避免重复创建。它是一种广泛使用且高效的实现方式

其优点是：

  1. 懒加载
  2. 线程安全
  3. 性能较高

其缺点是：

  1. 写法复杂
4.1.2.4. 静态内部类

静态内部类是指：利用其不会随外部类加载而立即加载的特性，实例仅在首次使用时创建，同时天然保证线程安全，因此是一种强烈推荐的单例实现方式

其优点是：

  1. 懒加载
  2. 线程安全
  3. 写法优雅
  4. 性能最好

4.2. 简单工厂模式

4.2.1. 简单工厂模式概述

简单工厂模式不属于 23 种设计模式，其核心思想是：由工厂类负责决定创建哪种具体产品，而客户端无需直接使用 new 去实例化。

简单工厂模式通常包含三个角色：

  1. 抽象产品
    1. 通常是接口，也可以是抽象类
  2. 具体产品
    1. 实现或继承了抽象产品的具体类
    2. 客户端不直接创建具体产品实例，而是通过工厂来创建
  3. 工厂
    1. 具体类
    2. 将创建具体产品实例的逻辑集中在此，负责创建具体产品实例

4.3. 工厂方法模式

4.3.1. 工厂方法模式概述

工厂方法模式是指：通过定义一个用于创建对象的接口，让子类决定实例化哪一个具体类，从而将对象的创建与使用分离。简单来说就是，把对象的创建交给子类去实现，而不是在父类或调用者中直接 new

工厂方法模式通常包含四个角色：

  1. 抽象产品
    1. 通常是接口，也可以是抽象类
  2. 具体产品
    1. 实现或继承了抽象产品的具体类
    2. 客户端不直接创建具体产品实例，而是通过具体工厂来创建
  3. 抽象工厂
    1. 通常是接口，也可以是抽象类
  4. 具体工厂
    1. 具体类
    2. 一个具体产品对应一个具体工厂，该工厂专门用于生产这个产品

4.4. 抽象工厂模式

4.4.1. 抽象工厂模式概述

抽象工厂模式是指：提供一个接口，用于创建一系列相关或相互依赖的对象，而无需指定它们具体的类。

抽象工厂模式通常包含四个角色：

  1. 抽象产品（AbstractProduct）
    1. 通常是接口，也可以是抽象类
  2. 具体产品（ConcreteProduct）
    1. 实现或继承了抽象产品的具体类
    2. 客户端不直接创建具体产品实例，而是通过具体工厂来创建
  3. 抽象工厂（AbstractFactory）
    1. 通常是接口，也可以是抽象类
    2. 命名以 Factory 结尾
  4. 具体工厂（ConcreteFactory）
    1. 具体类
    2. 一系列的具体产品对应一个具体工厂，该工厂专门用于生产这个系列的产品
    3. 命名以 Factory 结尾

4.5. 建造者模式

4.5.1. 建造者模式概述

建造者模式是指：将一个复杂对象的构建过程与其表示分离，使同样的构建过程可以创建不同的表示。

建造者模式通常包含四个角色：

  1. 产品（Product）
    1. 具体类
    2. 创建的实例就是这个复杂产品类的实例
  2. 抽象建造者（Builder）
    1. 通常是接口，可以是抽象类
    2. 定义了用于构建产品各个部分（即属性）的方法
    3. 命名以 Builder 结尾
  3. 具体建造者（ConcreteBuilder）
    1. 实现或继承了 Builder 的具体类
    2. 持有对 Product 的引用
    3. 实现各个构建方法，负责构建产品的各个部分（属性）
    4. 命名以 Builder 结尾
  4. 指挥者（Director）
    1. 具体类
    2. 持有对 Builder 的引用
    3. 按一定顺序调用具体建造者中各构建方法的实现
    4. 命名以 Director 结尾

4.6. 原型模式

4.6.1. 原型模式概述

原型模式是指：使用原型实例指定创建对象的种类，并通过复制这些原型创建新的对象。

原型模式主要解决的问题是：如果创建对象的成本较高，例如对象中的数据需要经过复杂计算才能获得，或者依赖于从 RPC 接口、数据库等较慢的 I/O 操作中获取，那么此时可以使用原型模式，即通过从已有对象中拷贝来生成新对象，就无需每次都从零创建并执行耗时的初始化过程。

原型模式通常包含两个角色：

  1. 抽象原型（Prototype）
    1. 通常是接口，也可以是抽象类
    2. 定义了一个 clone() 方法，用于克隆
    3. 然而，在 Java 中通常不需要自己写抽象原型，因为在 Java 里 Object.clone() 已经天然提供了原型模式所需的克隆能力
  2. 具体原型（ConcretePrototype）
    1. 实现或继承 Prototype 的具体类
    2. 实现 clone() 方法，负责返回当前对象的副本

5.2.1. 桥接模式概述

桥接模式是指：将抽象部分与它的实现部分分离，使它们可以独立地变化。

桥接模式主要解决的问题是：
抽象部分和实现部分可以看作业务的不同维度（抽象部分代表主维度，实现部分代表其他维度）。例如，在支付场景中，支付渠道是主维度（如微信支付、支付宝支付），而支付方式则是另一维度（如密码、指纹、人脸）

在这种情况下，如果直接使用策略模式，可能需要为每种组合都写一个具体类：

类似地，还会有 微信 + 指纹、微信 + 人脸、支付宝 + 密码、支付宝 + 指纹、支付宝 + 人脸 等组合，这是典型的 “支付渠道 × 支付方式” 场景，会导致类数量迅速增加。

例如，如果再增加银行支付，就会出现 银行支付 + 指纹、银行支付 + 人脸、银行支付 + 密码 等组合，类会呈爆炸式增长。因此，桥接模式主要用于解决这种多维度组合导致的类爆炸问题。

桥接模式通常包含四个角色：

  1. 抽象化（Abstraction）
    1. 抽象类
    2. 持有对 Implementor 的引用
  2. 扩展抽象化（RefinedAbstraction）
    1. 继承 Abstraction 的具体类
  3. 实现化（Implementor）
    1. 通常是接口，也可以是抽象类
  4. 具体实现化（ConcreteImplementor）
    1. 具体类

[!NOTE] 注意事项

  1. 并不是说策略模式不好，它本身非常强大，只是策略模式更适合解决单维度的问题，对于多维度组合就显得力不从心
  2. 但是如果我们尝试为两个维度分别设计策略模式，你会发现最终落地的代码与使用桥接模式的代码几乎相同
  3. 桥接模式的核心在于先识别出抽象部分，也就是业务的主维度

6.1. 观察者模式

6.1.1. 观察者模式概述

观察者模式用于建立对象之间的一种一对多依赖关系：当一个对象的状态发生变化时，所有依赖它的对象会收到通知并自动更新。该模式通常包含四个角色：

  1. 抽象被观察者（抽象主题，Subject）
    1. 通常是接口，也可以是抽象类
    2. 定义了 “被观察者必须实现的方法” 的规范
    3. 主要方法包括：
      1. registerObserver(Observer o)
        1. 注册观察者
      2. removeObserver(Observer o)
        1. 移除观察者
      3. notifyObservers()
        1. 当状态改变时，通知所有观察者
  2. 具体被观察者（具体主题，ConcreteSubject）
    1. 实现或继承了 Subject
    2. 通常还维护一个数据结构，用于存储观察者的引用
    3. 当其内部状态改变时，会调用 notifyObservers() 方法通知所有观察者
  3. 抽象观察者（Observer）
    1. 通常是接口，也可以是抽象类
    2. 定义了 “观察者必须实现的方法” 的规范
    3. 主要方法包括：
      1. update(…)
        1. 当观察者在接收到被观察者的通知后，会执行各自的具体更新操作。
  4. 具体观察者（ConcreteObserver）
    1. 实现或继承了 Observer
    2. 持有对 Subject 的引用，负责向不同 ConcreteSubject 中维护的数据结构注册或移除自己

其缺点是：

  1. 不保证顺序
    1. 被观察者是向所有注册的观察者广播通知，因此本身不保证顺序
    2. 如果观察者之间存在依赖关系，那
  2. 性能问题
    1. 如果观察者数量非常多，或者观察者的 update 方法非常耗时，通知过程可能会导致性能瓶颈
  3. 内存泄露风险
    1. 如果一个观察者在创建后，被注册到被观察者维护的数据结构中
    2. 但是之后不再使用，而且未被移除，它将无法被垃圾回收，从而导致内存泄漏

6.2. 模板方法模式

6.2.1. 模板方法模式概述

模板方法模式是指：在一个方法中定义算法的骨架，把一些步骤延迟到子类去实现，这样就能在不改变算法整体结构的前提下，允许子类重新定义某些步骤的具体实现。

该模式通常包含两个角色：

  1. 抽象父类（模板类）
    1. 必须是抽象类，因为需要规定模板方法
    2. 命名以 Abstract 开头，以 Template 结尾
    3. 主要方法包括：
      1. 模板方法
        1. 一个具体方法，用来定义算法步骤的执行顺序
        2. 对修改关闭，不能被子类重写
      2. 具体方法
        1. 已实现的方法
      3. 抽象方法
        1. 子类必须重写的方法
      4. 钩子方法
        1. 一个具体方法，子类可以选择性地重写以修改行为
        2. 命名以 hook 开头
  2. 具体子类
    1. 继承了抽象父类

其缺点是：

  1. 扩展性差
    1. 流程的骨架若需大幅调整，必须修改父类的模板方法，这会影响到所有子类
    2. 所以它只适合骨架稳定的场景

6.3. 策略模式

6.3.1. 策略模式概述

策略模式是指：将一组算法封装成独立的策略类，使得算法可以在运行时自由切换，从而让算法的变化独立于使用算法的客户端。

    1. 持有对 Strategy 的引用，负责调用不同 ConcreteStrategy 的方法
    2. 如果有共享数据，需要在上下文中进行维护

6.4. 责任链模式

6.4.1. 责任链模式概述

责任链模式是指：将多个处理请求的对象用链式结构连接起来，请求沿着链传递，每个对象可以处理请求，也可以把请求交给链上的下一个对象。

该模式通常包含三个角色：

  1. 抽象处理者（Handler）
    1. 必须是抽象类，因为要提供设置下一个处理请求的对象的引用的方法
  2. 具体处理者（ConcreteHandler）
    1. 实现了 Handler
    2. 判断能否处理本次请求，可以处理就处理，否则将请求转发给他的下一个处理请求的对象
  3. 客户端（Client）
    1. 负责组装责任链，并将请求提供给责任链连头的对象

其缺点是：

  1. 性能开销大
    1. 请求可能会沿着整个链一直传递，直到找到处理者或者到达链尾
    2. 链条很长时，每个请求都要经过多个对象，可能会影响性能
  2. 链中断风险
    1. 如果链条设置不当（比如某个节点没有正确转发请求），请求可能被 “吞掉”，没有任何处理结果

6.5.1. 状态模式概述

状态模式是指：允许一个对象在其内部状态改变时改变它的行为，使对象看起来好像修改了它的类

对象在不同状态下表现出不同行为，并且能在运行时根据状态自动切换行为，无需通过大量 if-else 去判断状态。该模式通常包含三个角色：

  1. 抽象状态（State）
  2. 具体状态（ConcreteState）
  3. 上下文（Context）
    1. 持有对 State 的引用，负责状态转换和调用不同 ConcreteState 的方法

6.6. 备忘录模式

6.6.1. 备忘录模式概述

备忘录模式是指：在不破坏对象封装性的前提下，捕获并保存对象的内部状态，以便之后将对象恢复到先前的状态。

简单来说，它就像一种快照机制，让对象可以在任意时刻保存自己的状态，并在需要时回滚到之前的状态。

该模式通常有三个角色：

  1. 发起人（原发器，Originator）
    1. 维护一个或多个状态，这些状态需要被保存
    2. 负责创建自身某一时刻的状态的快照（即生成备忘录）
    3. 负责根据传入的备忘录，将自身的状态恢复到当时的状态
  2. 备忘录（Memento）
    1. 用于封装发起人（Originator）在某一时刻的内部状态
    2. 备忘录通常只提供给发起人访问，不允许其他对象修改
      1. 推荐将 Memento 设计为 Originator 的静态内部类，类的访问修饰符是 public，构造方法的访问修饰符是 private，状态访问方法的访问修饰符是 private
        1. 保证所有类能访问到 Memento 类（能访问到，但无法创建对象）
        2. 保证只能由 Originator 类创建 Memento 对象和访问其保存的状态
      2. 如果 Memento 类非常庞大或逻辑复杂，作为内部类可能让 Originator 文件显得臃肿，这是内部类的通病。但在大多数情况下，备忘录只是简单保存状态数据，不会太大，因此放在 Originator 内部更能实现强封装和逻辑内聚
  3. 负责人（管理者，Caretaker）
    1. 维护一个数据结构，用来存储备忘录对象的引用

其缺点是：

  1. 内存开销大
    1. 备忘录对象需要存储发起人（Originator）的状态快照，如果状态数据量很大（如大型游戏场景、复杂配置对象等），每创建一个备忘录都会占用大量内存
    2. 随着程序运行，Caretaker 中保存的历史记录会持续增长，若不限制记录数量，最终可能引发 OOM（内存溢出）

6.6.2. 备忘录模式实现

1. 创建发起人（Originator）、备忘录（Memento）

[!NOTE] 注意事项

  1. 关于内部类的创建，内部类本身和其构造方法都可以使用四种访问修饰符，这里仅讨论 public 和 private 的情况。并且是基于 static 内部类，因为非 static 内部类在其他类中创建对象的方式有所不同。

[!NOTE] 注意事项
2. 内部类被设计为 static 是经过深思熟虑的结果
1. 如果不是 static，普通内部类会默认持有外部类的引用，也就是每个 Memento 对象会隐式包含一个 Originator 地址
1. 而我们的 Memento 设计目标仅仅是保存状态，这个引用完全没有必要
2. 此外，还会带来内存泄漏风险，当 Originator 实例本身已经不再需要时，如果它创建的 Memento 依然存在，并且 Memento 持有对该 Originator 的引用，就会导致 GC 无法回收 Originator
2. 此外，如果不是 static，外部类在创建该对象时必须依赖一个 Originator 实例，而 static 内部类则无需依赖外部实例

[!NOTE] 注意事项
3. 你可能会想，如果把 Memento 单独拉出来，类设为 public，属性设为 private，且不提供 Setter 方法，不也能防止其他对象修改吗？
1. 确实可以，但使用内部类在封装性上更优，更适合大型系统的设计

6.7. 命令模式

6.7.1. 命令模式概述

命令模式是指：将请求封装成一个对象，从而可以用不同的请求参数化其他对象，并支持请求的排队、记录、撤销与重做等操作。该模式通常有五个角色：

  1. 抽象命令（Command）
    1. 通常是接口，也可以是抽象类
    2. 定义了 “具体命令必须实现的方法” 的规范
    3. 主要方法包括：
      1. execute()
        1. 执行操作
      2. undo()
        1. 撤销操作
  2. 具体命令（ConcreteCommand）
    1. 实现或继承了 Command
    2. 命名以 Command 结尾
  3. 接收者（实现者，Receiver）
  4. 调用者（请求者，Invoker）
  5. 客户端（Client）

6.8. 中介者模式

6.8.1. 中介者模式概述

中介者模式是指：用一个中介对象来封装一系列对象之间的交互，从而使对象不需要显式地相互引用，降低了对象之间的耦合性

中介者模式通常包含四个角色：

  1. 抽象中介者（Mediator）
  2. 具体中介者（ConcreteMediator）
  3. 抽象同事（Colleague）
  4. 具体同事（ConcreteColleague）
