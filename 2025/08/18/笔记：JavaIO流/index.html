2. 用户态与内核态

计算机 IO（输入、输出）本质上都是：用户空间 ↔ 内核空间 ↔ 硬件（磁盘/网卡）

在理解 IO 之前，一个关键的基础概念就是 用户态（User Mode）和内核态（Kernel Mode），这决定了你的程序到底能不能直接操作硬件和系统资源。

  1. 用户态（User Mode）
    1. 这是应用程序（比如你的 Java 代码）运行的地方。
    2. 在这个级别下，程序既不能直接访问硬件设备（如硬盘、网卡），也无法随意操作操作系统的关键内存区域，以及干预进程的调度机制。
    3. 有这种限制的目的，就是保护系统稳定性。想象如果一个小程序能随便写进硬盘文件系统的元数据，整个系统就可能立刻崩溃
  2. 内核态（Kernel Mode）
    1. 这是操作系统内核运行的地方
    2. 在这个特权级别下，代码可以执行任何敏感指令：访问硬件、操作内存、调度进程等

当程序在用户态里想做 “越权操作” 时，必须通过 系统调用（System Call） 请求操作系统内核帮忙

3. IO 流基础体系

4. IO 流分类

  1. 以数据流向分类
    1. 输入流
      1. 从数据源读取数据到程序中
    2. 输出流
      1. 将数据从程序写出到目标位置
  2. 以数据单位分类
    1. 字节流
      1. 以字节（8 bit）为基本单位进行读写
      2. 适合处理所有类型的文件，尤其是二进制文件（如图片、音频、视频等）
      3. 以 InputStream、OutputStream 为抽象基类
    2. 字符流
      1. 以字符为基本单位进行读写
      2. 专门为文本数据设计，仅适用于纯文本文件
      3. 以 Reader、Writer 为抽象基类
  3. 以来源/目的地分类（节点流）
    1. 文件流
      1. 文件字节流
        1. FileInputStream
        2. FileOutputStream
      2. 文件字符流
        1. FileReader
        2. FileWriter
    2. 内存流
    3. 管道流
    4. 网络流
      1. 网络字节流（TCP 通信）
        1. Socket.getInputStream()
          1. 用于服务端（Server）
        2. Socket.getOutputStream()
          1. 用于客户端（Client）
  4. 在基础的节点流之上，Java 提供了一类处理流。
    1. 缓冲流
      1. 缓冲字节流
        1. BufferedInputStream
        2. BufferedOutputStream
      2. 缓冲字符流
        1. BufferedReader
        2. BufferedWriter
    2. 数据流
      1. 数据字节流
        1. DataInputStream
        2. DataOutputStream
    3. 对象流
      1. 对象字节流
        1. ObjectInputStream
        2. ObjectOutputStream
    4. 转换流
    5. 打印流
    6. 压缩流
    7. 序列化流
    8. 需要注意的是，处理流不能单独存在，而是通过 “包裹节点流” 来对已有流进行功能增强。

1. 获取文件描述符
当我们执行 new FileInputStream("a.txt") 的时候，在 Java 层我们创建了一个 FileInputStream 对象，在系统层，这个构造函数内部会触发一个 open 系统调用，请求操作系统内核打开名为 “a.txt” 的文件

操作系统内核接收到请求后，会在文件系统中查找 “a.txt”，并会在其内部创建一个代表这个打开文件的结构体（包含了文件信息、当前读取位置等），然后返回一个唯一的数字ID给你的程序，这个ID就是**文件描述符 (File Descriptor)**。

FileInputStream 对象会保存这个文件描述符，现在，这个 Java 对象就和底层的内核文件结构体关联起来了。

[!NOTE] 注意事项

  1. 操作系统内部管理文件，大概有这样一层层的表结构：
    1. 进程级文件描述符表（per‑process fd table）
      1. 每个进程在内核态都有一张 “文件描述符表”，你调用 open() 得到的 整数 fd，其实就是这张表的 “下标”
        1. 该表项本身并不存放文件数据，而是一个 引用，指向 系统级打开文件表 的某个条目。
    2. 系统级打开文件表（system‑wide open file table）
      1. 内核维护这张全局表，用于记录所有被打开的文件实例
      2. 每个条目保存了与这次“打开行为”相关的状态，包括：
        1. 当前读写偏移量
        2. 打开模式（只读/可写/追加）
      3. 此外，每个条目还含有一个 引用，指向对应的 inode，这样可以找到文件的元信息和数据位置
    3. 文件系统 inode 表（inode table）
      1. inode 是文件系统维护的文件元信息结构，与某个用户进程无关。
      2. 一个 inode 描述了：文件大小、所有者、权限、时间戳，以及磁盘数据块的物理位置等
      3. 所有指向该文件的 “打开实例条目”，最终都会通过 inode 找到该文件的真实内容
  2. 每个新进程创建时，内核会默认为它打开三个文件描述符：
    1. 0
      1. 标准输入（stdin），默认连接到键盘输入
    2. 1
      1. 标准输出流（stdout），默认连接到终端屏幕
    3. 2
      1. 标准错误流（stderr），默认连接到终端屏幕，独立于标准输出
    4. 这就是为什么在 C 程序里常见的 printf 或者在 Java 里 System.out.println，最终都能直接在屏幕上看到结果。
  3. 我们可以通过 FileInputStream.getFD() 等方式拿到一个 FileDescriptor 对象。在这个对象内部，确实有一个记录底层文件描述符整数的字段（fd）。
    1. 不过，这个字段在 JDK 中被标记为 包级可见（package‑private），也就是说外部代码无法直接访问。Java 之所以这样设计，是有意避免开发者绕过抽象层，直接依赖平台相关的文件描述符数字。
    2. 如果真的需要查看这个数字，通常只能借助 反射 这个黑科技来访问该字段。

2. Java 层发起读请求
fis.read() 在 JVM 里其实会调用一个 native 方法（JNI），本质就是触发 read() 的系统调用。

它向内核传递了三个关键信息：

  1. 文件描述符
    1. 指明要从哪个文件读取数据
  2. 目标缓冲区
    1. 告诉内核数据应写入到 JVM 的哪块内存区域
  3. 读取长度

[!NOTE] 注意事项

  1. 当调用 int len = fis.read(byte[] buffer) 时，传递给内核的其实是 这个字节数组在 JVM 堆中的起始地址，内核据此将数据直接写入相应的内存区域。
  2. 如果调用的是 int b = fis.read()，情况就不同了。因为 b 只是一个基本类型变量，存放在局部变量槽中，并没有内存地址可以直接交给内核
  3. 为解决这个问题，Java I/O 库通常在 JNI 层准备一个 内部缓冲区（通常只有 1 个字节大小）。内核会先把数据写入这个缓冲区，再由底层代码取出并赋值给 b

3. 内核接管 CPU，检查 “高速缓存”
从这一刻开始，CPU 的执行状态会从 用户态 切换为 内核态。控制权交由操作系统内核，Java 线程在逻辑上则进入“挂起等待” 状态。

内核接手后，会先通过传入的文件描述符定位到对应的文件结构体，接着读取其中的当前读写偏移量，并据此计算出本次需要访问的具体数据范围。

然后，操作系统内核会检查内存中的 文件缓存区（page cache）。这是内核预留的一块专门缓存，以「页（page）」为单位管理（通常 4KB）。内核会先计算出目标数据所在的页面，再判断缓存情况：

  1. 缓存命中
    1. 这是最高效的路径，可能因为你或别的程序刚刚读过这个文件，相关内容已经被预加载到内存里。
    2. 内核在 Page Cache 中找到了所需的页面，从中精确拷贝出请求的那几个字节，并拷贝到你的用户空间
  2. 缓存未命中
    1. 这是第一次读文件，或者数据太久没被用过，缓存里没有现成的页。
    2. 内核会向硬盘驱动发出请求，要求读取包含目标字节的一个完整数据块（往往是 4KB 或更大，这是典型的预读优化策略）
    3. 因为磁盘 I/O 较慢，内核会把发起系统调用的 Java 线程标记为「阻塞」，先从 CPU 上撤下来，让处理器去忙别的任务
    4. 这时，硬盘控制器开始用 DMA（直接内存访问） 把数据搬到内存。完成后，它会给 CPU 发一个“中断”信号，报告：“货已送达！”
    5. 内核收到中断后启动对应的处理程序，把刚才的 Java 线程唤醒，将状态改为「就绪」，排队等待调度
    6. 当调度器再次分配 CPU 时间时，这个线程会 “从刚才阻塞的地方继续执行”，完成最后的 copy_to_user(user_buf, page_cache_data, len)。

[!NOTE] 注意事项

  1. 所谓 “逻辑暂停”，只是从 Java 程序员的视角看，线程像被卡住了一样。实际上 CPU 并没闲着，而是在执行内核态的代码
  2. 即便你只用 read() 读取 1 个字节，内核访问磁盘时仍然是按 页（page） 为单位的（通常 4KB）
  3. Page Cache 没有固定大小，而是动态伸缩的。
    1. 如果机器有 32GB 内存，Page Cache 占个二十几 GB 都很正常，这是内核的 “贪吃” 本性
    2. 当应用程序突然需要大量内存（比如一口气要 28GB），内核会逐步回收一部分 Page Cache，把空间还给应用程序。
    3. 操作系统提供了一系列可调参数，用来影响 Page Cache 的回收与写回策略

4. 唤醒线程
CPU 从内核态切换回用户态，把控制权交还给 Java 线程

2. Java 层发起写请求
fos.write("Hello".getBytes()) 在 JVM 里其实会调用一个 native 方法（JNI），本质就是触发 write(data) 的系统调用。

它向内核传递了三个关键信息：

  1. 文件描述符
    1. 指明要往哪个文件写入数据
  2. 数据源
  3. 数据长度

3. 内核接管 CPU，写入到 ”高速缓存“
从这一刻开始，CPU 的执行状态会从 用户态 切换为 内核态。控制权交由操作系统内核，Java 线程在逻辑上则进入“挂起等待” 状态。

内核接手后，会先通过传入的文件描述符定位到对应的文件结构体，接着读取其中的当前读写偏移量，确定要写入的位置

然后，操作系统内核会定位或分配一块属于该文件的 Page Cache 内存页，并将你的 data 从 用户空间 拷贝到 内核空间的 Page Cache 中。

此时数据已经落到了内存缓存里（Page Cache），还没有真正写入硬盘！

当数据被写入 Page Cache 后，内核会在对应的内存页上打一个 “脏页” 标记，意思是，该页内存中的内容比硬盘上的副本更新，未来需要在合适的时机回写到磁盘。

4. 返回成功状态
write(data) 系统调用在数据被写入 Page Cache 并打上“脏页”标记后，就算大功告成了。此时它会立即返回一个 “写入成功” 的状态给你的程序，CPU 也随之从 内核态 切回 用户态。

[!NOTE] 注意事项

  1. 这里的 “写入成功”，仅仅意味着数据已经可靠地进入了 内存缓存（Page Cache），但这并不等同于硬盘上的内容已经更新。如果在此之后系统突然崩溃，那么这些尚未来得及写入磁盘的数据依然可能丢失。
  2. 数据真正落盘的时机有：
    1. 内核定时任务
      1. 操作系统有一个后台进程（如 Linux 的 pdflush 或 kworker），会周期性地检查 “脏” 页。
      2. 如果一个页面“脏”了太久，就会被安排写入磁盘。
    2. 内存压力
      1. 当系统内存不足，需要回收一些 Page Cache 页面给其他程序使用时，如果这些页面是 “脏” 的，内核会先将它们写入磁盘，然后再回收。
    3. 用户主动要求
      1. fos.flush()
        1. 这个方法会触发一个 fsync(fd) 的系统调用，会将这个文件的所有 Page Cache 落盘，并等到底层存储设备报告 “写成功” 才返回
        2. 这个过程是阻塞的，确保数据持久化
      2. fos.close()
        1. 关闭文件流时，它会在内部自动调用一次 fos.flush()，然后再释放文件描述符等资源。
    4. sync 命令
      1. 在 Linux 系统中，执行 sync 命令会要求内核将所有内存中的 “脏” 数据（不仅仅是你的文件）全部写入磁盘。

5.1.3.1. FileReader
5.1.3.1.1. FileReader 概述

FileReader 是字符输入流，它用于从文件中读取文件内容，并将底层的字节自动解码为字符（根据解码方式，自动取出合适数量的字节来解码成一个完整字符）

虽然理论上字节流可以用于处理所有文件类型，但它是以 “一个字节一个字节” 的方式读数据。如果只是复制文件，这完全没问题。但如果直接拿来显示文本，就容易出现乱码，因为常见的编码方式（如 UTF-8）并不是 “一个字符 = 一个字节”，而是可变长度编码（1～4 个字节一个字符）。这就导致逐字节解析时可能把字符拆断，显示结果自然就乱了。

6.1.1. BufferedInputStream

6.1.1.1. BufferedInputStream 概述

BufferedInputStream 在字节节点流（任意字节节点流）之上，增加了一个默认大小为 8192 字节的内部缓冲区，从而减少底层 I/O 调用，提高性能，同时免去了我们手动维护字节数组的麻烦。

[!NOTE] 注意事项

  1. 对于普通的字节输入流（如 FileInputStream），如果逐字节读取，会导致底层频繁触发 I/O 调用，效率很低。虽然它也支持 read(byte[]) 批量读取，但这种方式需要我们自己准备和管理字节数组。而 BufferedInputStream 内部自带一个缓冲区（默认 8192 字节）。当缓存耗尽时，它会一次性尽量把缓冲区填满，这样接下来无论你是逐字节读取，还是用自己的外部数组接收数据，都优先从内存缓冲区取，避免频繁访问底层流
  2. 实际代码中，我们仍常常会声明一个外部的 byte[] b 来装数据。这看似是 “缓存套缓存”，但别担心，内部缓冲负责减少系统 I/O 调用，外部数组只是从内部缓存搬数据到用户逻辑做处理。内存中的拷贝开销极小，几乎可以忽略，而且速度极快，两者结合大大提高了整体性能

8. 对象流

8.1. 对象流概述

对象流也常被称为 序列化流 / 反序列化流，能够把 Java 内存中的对象序列化为 字节流，并且可以再从字节流中反序列化恢复成原始对象，这使得对象可以方便地 持久化到文件 或 通过网络传输。

[!NOTE] 注意事项

  1. 序列化的结果是 字节流，而不是字符流。字符流只适合处理文本数据；对象序列化必须通过字节流来表达。
  2. 被序列化的类，必须实现 Serializable 接口。这个接口是一个标记接口，是通过 “接口” 的形式，给 JVM 或相关的类库打个标记，说：“这个类支持某种特性”，而只要我们 Implements Serializable，他就告诉 JVM 这个类的对象可以被序列化和反序列化

[!NOTE] 注意事项
3. 序列化/反序列化，一定要注意固定 serialVersionUID
1. 我们之前定义了一个可序列化的类，如果对象已经序列化写入文件，当我们修改了类的结构（比如新增字段、修改字段类型等），在反序列化时就会报错
2. 这是因为 Java 在序列化时会自动为类生成一个 serialVersionUID（版本号），并将其写入序列化文件。
3. 反序列化时，JVM 会检查本地类的 serialVersionUID 是否与文件中存储的一致，如果不一致，就会抛出 InvalidClassException。
4. 因此，我们通常会手动指定一个固定的 serialVersionUID，以避免因类结构的轻微调整导致反序列化失败：

[!NOTE] 注意事项
4. 在 IDEA 中，可以开启这两个选项，这样当我们实现类 Serializable 接口但未定义 serialVersionUID 时会提醒你，并可以自动计算出一个 UID

8.2. ObjectInputStream

8.2.1. ObjectInputStream 概述

ObjectInputStream 用于从底层的字节输入流中反序列化（deserialize）对象，也就是说，把字节流还原成 Java 对象

[!NOTE] 注意事项

  1. JDK 的反序列化是基于 ObjectInputStream.readObject()。在默认情况下，它可以反序列化任意实现了 Serializable 的类。但从 Java 11 开始，引入了 ObjectInputFilter，可以通过黑白名单或自定义规则来控制哪些类允许反序列化，哪些类禁止反序列化。这样做的目的主要是防止外部不可信数据在进入 JVM 时被随意反序列化，降低安全风险。
  2. 在 Spring Security 6 之后，为了进一步防止反序列化漏洞，框架内部使用了 ObjectInputFilter 并内置了黑白名单策略，默认禁止任意类反序列化，只允许部分安全类。常见的解决方案包括：
    1. 直接禁用 Jackson 白名单限制
    2. 将某类添加到 Jackson 反序列化白名单中
