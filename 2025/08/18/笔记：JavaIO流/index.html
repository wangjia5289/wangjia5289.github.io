IO 流基础

用户态与内核态

在理解 Java 的 I/O 之前，一个关键的基础概念就是 用户态（User Mode）和内核态（Kernel Mode），这决定了你的程序到底能不能直接操作硬件和系统资源。

  1. 用户态（User Mode）
    1. 这是应用程序（比如你的 Java 代码）运行的地方。
    2. 在这里，程序不能直接访问硬件（比如硬盘、网卡），也不能随便操作操作系统关键内存区域
    3. 限制的目的就是保护系统稳定性。想象如果一个小程序能随便写进硬盘文件系统的元数据，整个系统就可能立刻崩溃
  2. 内核态（Kernel Mode）
    1. 这是操作系统内核运行的地方
    2. 在这个特权级别下，代码可以执行任何敏感指令：访问硬件、操作内存、调度进程 等

当程序在用户态里想做“越权操作”时（比如读写文件、收发网络包），它无法直接自己动手，必须通过 系统调用（System Call） 请求内核帮忙。触发系统调用后，CPU 会进入一个 “陷入（Trap）” 过程：

  1. 当前运行线程仍旧是同一个，但 CPU 从用户态切换到内核态
  2. 内核接管 CPU，执行相应的内核函数，完成硬件操作
  3. 内核把结果准备好，再把 CPU 特权级切回到用户态，继续让你的程序执行

读 IO

假设我们有这样一段代码：

1. Java 层发起调用
你写的 fis.read() 在 JVM 里其实会调用一个 native 方法（JNI），本质就是触发 Linux 的 read() 系统调用（Windows 下同理）。

在这一刻起，CPU 的执行状态从 用户态 切换到 内核态，操作系统内核会接管 CPU，并且操作系统知道这是一个可能耗时的 I/O 操作，于是立即将你的 Java 线程状态变成阻塞状态，什么也干不了，只能傻傻地等待。

2. 操作内核找到文件
在我们 FileInputStream fis = new FileInputStream("a.txt"); 的时候，会拿到一个文件描述符（fd），操作系统通过文件描述符到文件在内核里的 inode（文件元数据），并且定位到你要读取的文件偏移位置

3. 检查 page cache
内核查看 page cache（内核里专门的一块缓存区，单位是 page，一般 4KB）

  1. 命中
    1. 如果这一页已经在内存里，直接用
  2. 未命中
    1. 内核向磁盘控制器发出指令，要求从硬盘读取文件数据把目标页从磁盘读入 page cache
    2. 现在多用 DMA 技术，内核对 DMA 发号命令后，在 DMA 搬运数据期间，CPU 可以切换成用户态被调度去执行其他完全不相干的、处于 RUNNABLE 状态的线程或进程。这就是多任务操作系统的魔力
    3. 触发一次磁盘 I/O，把目标页从磁盘读入 page cache
    4. DMA 控制器完成了它的任务，通过一个“中断”信号通知 CPU

需要注意的是，即便你只调用 read() 读 1 个字节，但内核读磁盘是以 页 为单位的（4KB），所以它可能一次读入一个页，可能比你需要的更多数据

需要注意的是，每个**打开的文件描述符 (fd)**，除了关联 inode（文件元信息）外，还会维护一个 文件位置指针，指示“读/写到文件的哪个位置了”，这个指针就是 文件偏移量（以字节为单位）

所以这里就进行了第一次的数据拷贝，即硬件 -> 内核内存，此时，数据已经位于内核内存中。但你的 Java 线程还在用户空间，它看不见也摸不着这些数据。

4. 第 2 次数据拷贝 (内核内存 -> 用户内存)
内核现在需要把数据从自己的地盘（Page Cache）送到你的地盘，也就是 CPU 执行内存拷贝指令，把数据从 Page Cache 复制到你的 JVM 对应的位置

5. 唤醒线程
内核发现所有工作都完成了（数据已经成功拷贝到用户空间）内核会找到之前被挂起的那个 Java 线程，把它从“等待队列”里拿出来，将其状态从 BLOCKED 改回 RUNNABLE，CPU 恢复之前保存的该线程的用户态上下文，放回到 CPU 的“工作队列”里此时，线程被唤醒了，但可能还没轮到它立即执行。

统调用执行完毕，CPU 的状态从 内核态 切换回 用户态。

写流程

Page Cache 没有固定大小，而是动态伸缩的。如果你的机器有 32GB 内存，只有 2GB 被应用程序用掉，那么剩下的二十几 GB 里，Page Cache 可能会疯狂增长，占用十几 GB，一点都不稀奇。但如果此时应用程序突然需要 28GB 那么内核会逐步释放 Page Cache，把空间还给应用

在 Linux 上可以直接看内存状态：

Linux 内核对“该不该回收 Page Cache / 该留多少”有策略参数：

  • **vm.vfs_cache_pressure**：控制内核回收 inode/dentry 缓存的积极度。
  • vm.dirty_ratio 与 **vm.dirty_background_ratio**：控制脏页占用内存的比例，避免缓存太多脏数据。
  • **drop_caches**：可手动命令清理 Page Cache（不常用，但调优/测试时方便）。

1. Java 层发起调用

  • 初始状态: 文件 output.txt 可能不存在（会被创建）或为空。文件描述符 fd 对应的文件位置指针 = 0。

第 1 步 & 第 2 步：发起系统调用与模式切换

  • 动作: 你的 Java 线程执行 fos.write(dataToWrite)，这会触发一个 write 系统调用。
  • CPU 模式: 立即从 用户态 切换到 内核态。
  • 线程状态: 线程通常不会立即阻塞。因为写入内存（Page Cache）的速度非常快，这个操作大概率能瞬间完成，所以线程可能一直保持 RUNNABLE 状态。只有在 Page Cache 已满，需要等待数据刷盘才能腾出空间等极端情况下，write 才可能阻塞。

第 3 步：内核接管 - 定位与准备

  • 动作: 内核收到 write 请求，包含三要素：文件描述符 (fd)、要写入的数据 (dataToWrite)、数据长度 (5)。
  • 指针使用: 内核查看 fd 关联的文件位置指针，发现是 0。内核明白了：“好的，我要把这 5 个字节的数据，写入到文件的第 0 个字节开始的位置。”

第 4 步：数据的第一次旅行 (用户内存 -> 内核内存)

  • 动作: 这是 write 操作的核心。内核执行一次内存拷贝。
  • 路径: 用户空间的 dataToWrite 字节数组 -> 内核空间的 Page Cache
  • 结果:
    1. “Hello” 这 5 个字节的数据被完整地复制到了 Page Cache 中。
    2. 内核将 Page Cache 中对应的这块内存区域标记为 **“脏” (Dirty)**。

第 5 步：指针推进与快速返回

  • 指针推进: 在数据成功拷贝到 Page Cache 之后，内核立即更新文件位置指针。

    • 计算: 旧指针(0) + 实际写入的字节数(5) = 新指针(5)
    • 内核将 fd 对应的文件位置指针更新为 5。

  • 关键点 - 快速返回:

    • 一旦数据被拷贝到 Page Cache，内核就认为这次 write 操作已经“完成”了！
    • 它不会等待数据被真正写入硬盘。
    • 系统调用立即返回。CPU 从内核态切换回用户态。
    • 你的 Java 代码 fos.write(dataToWrite) 执行完毕，线程继续执行下一行代码。

对你的程序而言，到此为止，写入似乎已经大功告成。但真相是，数据仅仅是到达了内核的内存缓冲区。

第 6 步：数据的第二次旅行 (内核内存 -> 硬盘) - 异步刷盘

这一步对你的 write() 调用是异步 (Asynchronous) 的，你的线程不会等待它。

  • 触发时机: 操作系统内核会在以下几种情况，决定将“脏页”刷写（Flush）到硬盘：

    1. 定时任务: 内核中有后台线程（如 pdflush 或 flusher 线程）会周期性地检查脏页，并将其写回硬盘。
    2. 脏页比例: 当脏页占用的内存达到某个阈值时，系统会触发回写以释放内存。
    3. 用户主动调用 sync() 或 fsync(): 比如你在 Java 中调用 fos.flush() 或 fileChannel.force()，这会明确告诉内核：“请立即将这个文件的数据写入硬盘”。
    4. 关闭文件 (close()): 当你调用 fos.close() 时，它会隐式地触发一次 flush 操作，确保所有缓存数据在文件关闭前都写入硬盘。这是为什么在文件操作后总是强调要 close() 的重要原因之一！
    5. 系统内存压力大: 当系统需要更多可用内存时，会把脏页写回硬盘，然后回收这部分内存。

  • 路径: 内核空间的 Page Cache -> DMA 控制器 -> 硬盘

  • 过程: 内核命令 DMA 控制器，将脏页中的数据传输到硬盘的对应位置。

IO 流分类

Java IO 流分类的核心是：

  1. 以数据流向分类
    1. 输入流
      1. 从数据源读取数据到程序中
    2. 输出流
      1. 将数据从程序写出到目标位置
  2. 以数据单位分类
    1. 字节流
      1. 以字节（8 bit）为基本单位进行读写
      2. 适合处理所有类型的文件，尤其是二进制文件（如图片、音频、视频等）
    2. 字符流
      1. 以字符为基本单位进行读写
      2. 专门为文本数据设计，仅适用于纯文本文件
  3. 以来源/目的地分类（节点流）
    1. 文件流
    2. 内存流
    3. 管道流
    4. 网络流
  4. 在基础的节点流之上，Java 提供了一类处理流。
    1. 处理流不能单独存在，而是通过 “包裹其他流” 来对已有流进行功能增强。
    2. 缓冲流
    3. 数据流
    4. 对象流
    5. 转换流
    6. 打印流
    7. 压缩流

IO 流侧重于前两种分类方式，又形成 IO 世界的 “四大天王”：

  1. InputStream
    1. 字节输入流，抽象基类
  2. OutputStream
    1. 字节输出流，抽象基类
  3. Reader
    1. 字符输入流，抽象基类
  4. Writer
    1. 字符输出流，抽象基类

在这四个抽象类的基础上，根据功能又可分为：

  1. 文件流
    1. 文件字节流
      1. FileInputStream
      2. FileOutputStream
    2. 文件字符流
      1. FileReader
      2. FileWriter
  2. 缓冲流
    1. 缓冲字节流
      1. BufferedInputStream
      2. BufferedOutputStream
    2. 缓冲字符流
      1. BufferedReader
      2. BufferedWriter
  3. 数据流
    1. 数据字节流
      1. DataInputStream
      2. DataOutputStream
  4. 对象流
    1. 对象字节流
      1. ObjectInputStream
      2. ObjectOutputStream
  5. 转换流
  6. 序列化流
  7. 打印流
  8. 压缩流
  9. Commons-io

文件字符流

FileReader
FileReader 概述

FileReader 是字符输入流，它用于从文件中读取文件内容，并将底层的字节自动解码为字符（根据解码方式，自动取出合适数量的字节来解码成一个完整字符）

虽然理论上字节流可以用于处理所有文件类型，但它是以 “一个字节一个字节” 的方式读数据。如果只是复制文件，这完全没问题。但如果直接拿来显示文本，就容易出现乱码，因为常见的编码方式（如 UTF-8）并不是 “一个字符 = 一个字节”，而是可变长度编码（1～4 个字节一个字符）。这就导致逐字节解析时可能把字符拆断，显示结果自然就乱了。

[!NOTE] 注意事项

  1. JDK 11 之前，FileReader 只能使用系统默认解码规则
    1. Mac
      1. UTF-8
    2. Linux
      1. UTF-8
    3. Windows
      1. GBK

缓冲字节流

BufferedInputStream

BufferedInputStream 概述

BufferedInputStream 在字节节点流（任意字节节点流）之上，增加了一个默认大小为 8192 字节的内部缓冲区，从而减少底层 I/O 调用，提高性能，同时免去了我们手动维护字节数组的麻烦。

[!NOTE] 注意事项

  1. 对于普通的字节输入流（如 FileInputStream），如果逐字节读取，会导致底层频繁触发 I/O 调用，效率很低。虽然它也支持 read(byte[]) 批量读取，但这种方式需要我们自己准备和管理字节数组。而 BufferedInputStream 内部自带一个缓冲区（默认 8192 字节）。当缓存耗尽时，它会一次性尽量把缓冲区填满，这样接下来无论你是逐字节读取，还是用自己的外部数组接收数据，都优先从内存缓冲区取，避免频繁访问底层流
  2. 实际代码中，我们仍常常会声明一个外部的 byte[] b 来装数据。这看似是 “缓存套缓存”，但别担心，内部缓冲负责减少系统 I/O 调用，外部数组只是从内部缓存搬数据到用户逻辑做处理。内存中的拷贝开销极小，几乎可以忽略，而且速度极快，两者结合大大提高了整体性能

转换流

对象流

对象流概述

对象流也常被称为 序列化流 / 反序列化流，能够把 Java 内存中的对象序列化为 字节流，并且可以再从字节流中反序列化恢复成原始对象，这使得对象可以方便地 持久化到文件 或 通过网络传输。

[!NOTE] 注意事项

  1. 序列化的结果是 字节流，而不是字符流。字符流只适合处理文本数据；对象序列化必须通过字节流来表达。
  2. 被序列化的类，必须实现 Serializable 接口。这个接口是一个标记接口，是通过 “接口” 的形式，给 JVM 或相关的类库打个标记，说：“这个类支持某种特性”，而只要我们 Implements Serializable，他就告诉 JVM 这个类的对象可以被序列化和反序列化

[!NOTE] 注意事项
3. 序列化/反序列化，一定要注意固定 serialVersionUID
1. 我们之前定义了一个可序列化的类，如果对象已经序列化写入文件，当我们修改了类的结构（比如新增字段、修改字段类型等），在反序列化时就会报错
2. 这是因为 Java 在序列化时会自动为类生成一个 serialVersionUID（版本号），并将其写入序列化文件。
3. 反序列化时，JVM 会检查本地类的 serialVersionUID 是否与文件中存储的一致，如果不一致，就会抛出 InvalidClassException。
4. 因此，我们通常会手动指定一个固定的 serialVersionUID，以避免因类结构的轻微调整导致反序列化失败：

[!NOTE] 注意事项
4. 在 IDEA 中，可以开启这两个选项，这样当我们实现类 Serializable 接口但未定义 serialVersionUID 时会提醒你，并可以自动计算出一个 UID
