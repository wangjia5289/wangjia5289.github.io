笔记：Java IO 模式

IO 模式概述

同步阻塞操作（BIO）

Java BIO（同步阻塞式）采用传统的 “一连接一线程” 模型：每当有客户端发起连接请求，服务器端就会为其分配一个独立线程进行处理，如果某个连接长时间处于空闲状态，对应的线程也会一直阻塞等待，导致系统资源被浪费，增加了不必要的线程开销。

因此，Java BIO 适用于连接数较少且相对固定的应用场景，因为这种模型对服务器资源消耗较大，并发能力有限，难以应对大量并发连接。

在 JDK1.4 之前，BIO 是 Java 网络编程的唯一选择，但其实现方式简单，代码易于理解和维护

异步阻塞操作

异步非阻塞操作（AIO）

Java AIO（异步非阻塞式）采用 “一个有效请求对应一个线程” 的模式，客户端的 I/O 请求由操作系统先完成，然后再通知服务器应用启动线程进行处理。通常适用于连接数较多、且连接时间较长的场景。

因此 Java AIO 适用于连接数多且连接时间较长（重操作）的架构，例如相册服务器。它能够充分利用操作系统参与并发处理。从 JDK 7 开始提供支持，但编程相对复杂。

1.2. 同步、异步、阻塞、非阻塞

我们通常将操作分为下面的操作：

  1. 同步操作

    1. 同步简单操作
    2. 同步阻塞操作

  2. ==同步操作==：

    1. 同步简单操作：
      1. 快速执行、无明显耗时、不涉及复杂计算或 I/O
    2. 同步阻塞操作：
      1. 设计耗时计算或潜在阻塞操作（如 I/O），但是在当前线程同步执行
    3. 同步非阻塞操作

  3. ==异步操作==：

    1. 异步阻塞操作：
      1. 将同步阻塞操作通过异步机制（如线程池）执行，“伪非阻塞” 操作
    2. 异步非阻塞操作：
      1. 完全非阻塞，通常是基于时间驱动或回调，依赖外部异步机制（如 WebClient、 R2DBC、CompletableFuture）

  4. ==补充：I/O==：

    1. I/O，即输入/输出（Input/Output），是指计算机与外部世界的信息交换过程，包括但不限于：
      1. 网络请求：
        1. 如调用其他 HTTP API 接口（RestTemplate、WebClient）、JDBC、R2DBC 等等
      2. 文件操作：
        1. 读取或写入硬盘上的文件
      3. 用户交互：
        1. 键盘、鼠标等输入设备的信号获取，以及向显示器等输出设备发送信息。
      4. 硬件通信：
        1. 与打印机、扫描仪等外部设备的数据交换

这是一个很好的问题，因为这正是 Java 线程状态最容易让人感到困惑的地方。我们来总结一下，哪些常见的操作会导致线程在操作系统（OS）层面处于阻塞状态，但在 **Java 层面却依然是 RUNNABLE**。

这些操作通常都与 I/O (输入/输出) 有关，因为 I/O 操作需要等待外部设备（如硬盘、网卡）的响应，而这个等待过程是由操作系统管理的。

1.3. 常见的 I/O 阻塞操作总结

1.3.1. 文件 I/O

当线程执行文件读写操作时，如果数据不能立即就绪，就会发生阻塞。

  • 操作：

    • FileInputStream.read()：从文件中读取数据。

    • FileOutputStream.write()：向文件中写入数据。

  • 底层行为：Java 的 native 方法会向操作系统发出系统调用，请求读写磁盘。如果硬盘正在忙碌或数据量大，OS 会将线程挂起，直到 I/O 完成。

  • Java 线程状态：**RUNNABLE**。

1.3.2. 网络 I/O

网络通信是 I/O 阻塞最常见的场景之一。

  • 操作：

    • Socket.accept()：等待客户端连接。

    • Socket.connect()：连接远程服务器。

    • Socket.read() / Socket.write()：从网络中读取或写入数据。

    • DatagramSocket.receive()：接收 UDP 数据包。

  • 底层行为：native 方法会向操作系统请求网络资源。当线程在等待网络连接建立、或者等待数据从网络到达时，OS 会将线程挂起。

  • Java 线程状态：**RUNNABLE**。

1.3.3. 数据库 I/O

与数据库的交互本质上也是网络或文件 I/O。

  • 操作：

    • Connection.createStatement()

    • Statement.executeQuery()

    • ResultSet.next()

  • 底层行为：JDBC 驱动通过网络向数据库服务器发送请求。线程需要等待服务器处理请求并返回结果。

  • Java 线程状态：**RUNNABLE**。

1.4. 为什么这些操作会这样？

核心原因在于 Java Thread.State 的设计哲学。

  • BLOCKED, WAITING, TIMED_WAITING 这些状态，是专门为了反映 Java 语言层面的同步和等待机制。它们是 JVM 精心管理和控制的状态，比如等待 synchronized 锁或调用 wait() 方法。

  • I/O 阻塞是一个操作系统层面的行为。当 Java 调用 native 方法进行 I/O 时，它把控制权交给了 OS。从 JVM 的角度看，线程只是在执行一个 native 方法，它并没有通过 Java 自己的机制进入等待。一旦 OS 的 I/O 操作完成，这个 native 方法就会返回，线程就能继续执行 Java 代码。因此，JVM 认为这个线程一直都是可运行的。

通过理解这个区别，你就能更好地诊断和分析 Java 多线程程序中的性能瓶颈，区分是由于锁竞争（Java BLOCKED）还是由于 I/O 等待（Java RUNNABLE，OS 阻塞）造成的了。

BIO 工作机制

同步非阻塞式（NIO）

NIO 概述

NIO（同步非阻塞式）采用同步非阻塞模型，服务器通常通过 “一个线程处理多个连接” 的方式工作，所有客户端的连接请求都会注册到多路复用器（Selector）上，服务器线程通过轮询 Selector，发现有 I/O 事件时才进行处理，从而大大提升了资源利用率和并发能力。

因此，NIO 适合处理大量、短时连接的场景（如聊天服务器、弹幕系统、服务器间通信等）。

[!NOTE] 注意事项

  1. BIO 以流的方式处理数据，而 NIO 以块的方式处理数据，块 IO 的效率比流 IO 高很多
  2. BIO 是阻塞的，即便xxxx，线程也是阻塞的，而 NIO 则是非阻塞的，线程是xxxxx
  3. BIO 是基于字节流和字符流进行操作的，而 NIO 是基于 Channel 和 Buffer 进行操作的
    1. 数据总是从通道读取到缓冲区，或者从缓冲区写道通道
    2. 程序读数据：网络 / 文件 ➔ 数据 ➔ Channel ➔ Buffer ➔ 程序
    3. 程序写数据：程序 ➔ Buffer ➔ Channel ➔ 数据 ➔ 网络 / 文件
    4. BIO 中的要么是输入流，要么是输出流，是单向的，只能读或者只能写，但是 NIO 中的 Buffer 是既可以读也可以写，Channel 也是既可以读也可以写（依靠与 Buffer）

NIO 原理

NIO 原理图


通道（Channel）表示与 IO 设备（例如文件、套接字）的连接。

缓冲区（Buffer）是一个内存块，底层由数组实现，数据的读写都是通过 Buffer 完成的。

一个线程对应一个 Selector，一个 Selector 可以管理多个 Channel，而一个 Channel 通常对应一个 Buffer。程序在各个 Channel 之间的切换由时间和事件决定，Selector 会根据不同事件在各个通道间切换。

缓冲区（Buffer）

Buffer 概述

Buffer 就是一个数组，可以存放多个相同类型的数据。

想要掌握 Buffer，必须了解其几个重要概念：

  1. 容量 (capacity)
    1. 作为内存块，Buffer 具有固定大小，也称为 “容量”
    2. 缓冲区容量不能为负，并且创建后不可更改
  2. 限制 (limit)：
    1. 表示缓冲区中可操作的数据范围（超过 limit 的数据无法读写）
    2. 限制不能为负，也不能超过容量
    3. 写入模式下，limit 等于 Buffer 的容量
    4. 读取模式下，limit 等于已写入的数据量
  3. 位置 (position)：
    1. 指向下一个要读取或写入的数据索引
    2. 位置不能为负，也不能超过 limit
  4. 标记 (mark) 与重置 (reset)：
    1. 标记是通过 mark() 方法在 Buffer 中设置的一个特定 position
    2. 之后可以通过 reset() 方法将 position 恢复到标记的位置

Buffer 分类

Buffer 是一个抽象类，根据数据类型的不同，常用子类有：

  1. ByteBuffer
  2. ShortBuffer
  3. CharBuffer
  4. IntBuffer
  5. LongBuffer
  6. FloatBuffer
  7. DoubleBuffer

这些 Buffer 类在管理数据的方法上都非常相似，区别仅在于所管理的数据类型不同。

Buffer 创建/获取方式
直接缓冲区

直接缓冲区是指：Buffer 分配在操作系统的内存中，而不是 JVM 堆内存上。在 IO 操作中，直接缓冲区性能更高，因为它可以直接作用于本地系统的 IO。

从数据流的角度来看：

  1. 直接缓冲区的数据流
    1. 本地 IO ➔ 直接缓存 ➔ 本地 IO
  2. 非直接缓冲区的数据流
    1. 本地 IO ➔ 直接内存 ➔ 非直接内存 ➔ 直接内存 ➔ 本地 IO

需要注意的是，虽然 IO 性能更高，虽然直接缓冲区 IO 性能更高，但在分配和回收时开销较大。

[!NOTE] 注意事项

  1. 判断 Buffer 是否为直接缓冲区，可调用 xxBuffer.isDirect()，返回 true 就表示是直接缓冲区。
非直接缓冲区

非直接缓冲区是指：Buffer 分配在JVM 堆内存上

Buffer 常用方法

Buffer 的几个模式切换口诀

  • 写模式（写入数据 → read(channel, buffer)、put()）：

    • position 走到最后写入的位置
    • limit = capacity

  • flip() → 切换到读模式

    • limit = position（就是写入的长度）
    • position = 0，从头读

  • 读模式（读出数据 → get()）：

    • position 表示读到哪了
    • limit 是有效数据的末尾

  • **clear()**（准备再次写）：

    • position = 0
    • limit = capacity

通道（Channel）

Channel 概述

Channel 是 IO 源与目标之间打开的连接。Channel 类似于传统的 “流”，但有一点不同：Channel 本身不能直接访问数据，它只能与 Buffer 进行交互

Channel 分类

Channel 是一个接口，常用实现类有：

  1. 文件通道
    1. FileChannel
  2. 网络通道
    1. TCP 通道
      1. SocketChannel
        1. 与远程主机进行 TCP 通信的通道
      2. ServerSocketChannel
        1. 监听客户端 TCP 连接请求的通道
    2. UDP 通道
      1. DatagramChannel
        1. UDP 读写网络中的通道

[!NOTE] 注意事项

  1. FileChannel 实际上是阻塞的，而且不像网络通道那样可以通过 configureBlocking(false) 设置为非阻塞，因此不属于多路复用的一类，不能注册到 Selector
  2. 上述 Channel 分类只是 NIO 模式下 可用的通道类型。还有很多其他类型的 Channel，例如 AIO 使用的通道，并未在这里描述
FileChannel
FileChannel 概述

FileChannel 来自 Java NIO 包，表面上看似属于“新 I/O”，因此很多人会以为它支持非阻塞操作。但实际上，当你调用 fileChannel.read(ByteBuffer) 或 fileChannel.write(ByteBuffer) 时，当前线程会一直等待，直到数据传输完成才返回。

你可能会问：既然 FileChannel 也是阻塞的，为什么不直接使用 BIO 的输入流和输出流，而要用它并配合 Buffer？这是不是显得有些麻烦？

其实，FileChannel 的优势并不在于非阻塞，而体现在以下几个方面：

  1. 块读写
    1. BIO 的字节流和字符流在读写时是一个字节或一个字符一个字符地操作
    2. 而 FileChannel 可以直接将数据读写到 ByteBuffer 中，实现块读写，效率更高
  2. 随机访问
    1. FileChannel 可以直接跳转到文件的任意位置去读写（通过 position(long) 方法）
    2. 虽然 RandomAccessFile 也支持随机访问，但 FileChannel 的效率更高
  3. 内存映射
    1. map() 方法可以把文件直接映射到内存中。这样你像操作内存数组一样去读写文件（它底层利用操作系统的虚拟内存机制）
    2. 在在处理超大文件时，这种方式比传统 BIO 效率高得多
  4. 零拷贝文件传输
    1. transferTo() / transferFrom() 可以把数据直接在内核层搬运，避免用户态/内核态来回倒腾
    2. 在复制大文件（或者文件通过 socket 发送出去）时，性能优势非常明显
  5. 适合与 NIO 生态统一
    1. 如果你项目里既需要网络非阻塞 NIO，也需要文件操作，FileChannel 让接口和类型更统一
FileChannel 创建/获取方式

FileChannel 可以通过 FileInputStream、FileOutputStream、RandomAccessFile 的 getChannel() 方法获取：

  1. 使用 FileInputStream 获取的是读通道
  2. 使用 FileOutputStream 获取的是写通道
  3. 使用 RandomAccessFile 获取的是读写通道

Selector 是 SelectableChannel 对象的多路复用器，能够检测多个已注册通道上是否有事件发生，如果有事件发生，就会获取相应的事件，并对每个事件进行处理，从而实现一个线程同时监控多个 SelectableChannel 的 IO 状态
