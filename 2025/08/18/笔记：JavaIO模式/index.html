同步阻塞（BIO）

BIO 概述

BIO（Blocking IO）是最传统的 I/O 模型，采用传统的 “一连接一线程” 的方式，并且在进行 IO 操作时，如果数据还没准备好，那么发起 IO 调用的线程就会一直阻塞等待，直到数据可用。

伪异步 IO

在最传统的 BIO 模式 下，服务端的处理方式是 一连接一线程，每当有一个客户端接入，就创建一个线程专门处理这个连接，如果同时有 1000 个客户端，那么服务端就要开 1000 个线程去等待

然而，线程资源并不是无限的，Java 中每个线程都会占用一定的内存（线程栈空间），而且操作系统能同时调度的线程数也有限。当线程数急剧膨胀，就会导致：内存消耗过大 → 堆栈溢出、线程创建失败，最终可能导致进程宕机或僵死

我们可以采用 线程池 + 任务队列 的方式进行优化：

  1. 服务端接收到新的客户端连接后，不再立即创建独立线程，而是将客户端的 Socket 封装成一个 任务（Runnable/Callable）
  2. 这个任务会被放入 线程池的任务队列，后台仅维护固定数量的工作线程（例如 10 或 20 个）
  3. 工作线程会不断从队列中取出任务并执行，从而保证线程数量可控

这样一来，即使有大量客户端并发接入，也不会出现无节制的新建线程，系统能保持稳定运行。

需要注意的是，这种方式只是解决了 “线程爆炸” 的资源问题，本质上仍然是 阻塞 I/O，因为每个工作线程在执行具体的 IO 操作时依然会阻塞等待数据，所以这被称为 伪异步 IO

同步非阻塞（NIO）

NIO 概述

NIO（New IO）通过 “一个线程处理多个连接” 的方式，所有客户端的连接都会注册到多路复用器（Selector）上，服务器线程通过轮询 Selector，发现有 IO 事件时才进行处理，从而大大提升了资源利用率和并发能力。

[!NOTE] 注意事项

  1. NIO 的读写流程
    1. 程序读数据：硬件（磁盘/网卡） ➔ 内核空间 ➔ Channel ➔ Buffer
    2. 程序写数据：Buffer ➔ Channel ➔ 内核空间 ➔ 硬件（磁盘/网卡）

NIO 组件

Buffer

Buffer 概述

Buffer 就是一个数组，可以存放多个相同类型的数据。想要掌握 Buffer，必须了解其几个重要概念：

  1. 容量 (capacity)
    1. 作为内存块，Buffer 具有固定大小，也称为 “容量”
    2. 缓冲区容量不能为负，并且创建后不可更改
  2. 限制 (limit)：
    1. 表示缓冲区中可操作的数据范围（超过 limit 的数据无法读写）
    2. 限制不能为负，也不能超过容量
    3. 写入模式下，limit 等于 Buffer 的容量
    4. 读取模式下，limit 等于已写入的数据量
  3. 位置 (position)：
    1. 指向下一个要读取或写入的数据索引
    2. 位置不能为负，也不能超过 limit
  4. 标记 (mark) 与重置 (reset)：
    1. 标记是通过 mark() 方法在 Buffer 中设置的一个特定 position
    2. 之后可以通过 reset() 方法将 position 恢复到标记的位置

Buffer 分类

Buffer 根据存储类型，可分为：

  1. ByteBuffer
  2. ShortBuffer
  3. CharBuffer
  4. IntBuffer
  5. LongBuffer
  6. FloatBuffer
  7. DoubleBuffer

Channel

Channel 概述
Channel 分类

在 Java IO 流中，如果以来源/目的地进行分类，有：

  1. 文件流
  2. 内存流
  3. 管道流
  4. 网络流

这里除了内存流没有对应的 Channel，其他三种流都有对应的 Channel

  1. 文件流对应 Channel
    1. FileChannel
  2. 管道流对应 Channel
    1. 写入端 Channel
      1. Pipe.SinkChannel
    2. 读取端 Channel
      1. Pipe.SourceChannel
  3. 网络流对应 Channel
    1. TCP Channel
      1. SocketChannel
        1. 与远程主机进行 TCP 通信的通道
      2. ServerSocketChannel
        1. 监听客户端 TCP 连接请求的通道
    2. UDP Channel
      1. DatagramChannel
        1. UDP 读写网络中的通道

[!NOTE] 注意事项

  1. 虽然一般管道都是可读可写的双向管道但是 NIO Pipe 的设计模型就是一个负责都一个负责写的单向管道
  2. FileChannel 实际上是阻塞的，而且不像网络通道那样可以通过 configureBlocking(false) 设置为非阻塞，因此不属于多路复用的一类，不能注册到 Selector
  3. 虽然 Pipe 也支持非阻塞，也能注册到 Selector，但主要是在单机程序内部做信号桥，用于线程间通信，不是网络通信。
  4. 上述 Channel 分类只是 NIO 模式下 可用的通道类型。还有很多其他类型的 Channel，例如 AIO 使用的通道，并未在这里描述
FileChannel
FileChannel 概述

FileChannel 来自 Java NIO 包，表面上看似属于“新 I/O”，因此很多人会以为它支持非阻塞操作。但实际上，当你调用 fileChannel.read(ByteBuffer) 或 fileChannel.write(ByteBuffer) 时，当前线程还是会一直等待，直到数据传输完成才返回。

你可能会问：既然 FileChannel 也是阻塞的，为什么不直接使用 BIO 的输入流和输出流，而要用它并配合 Buffer？这是不是显得有些麻烦？

其实，FileChannel 的优势并不在于非阻塞，而体现在以下几个方面：

  1. 随机访问
    1. FileChannel 可以直接跳转到文件的任意位置去读写（通过 position(long) 方法）
  2. 内存映射
    1. map() 方法可以把文件直接映射到内存中。这样你像操作内存数组一样去读写文件（它底层利用操作系统的虚拟内存机制）
    2. 在处理超大文件时，这种方式比传统 BIO 效率高得多
  3. 零拷贝文件传输
    1. transferTo() / transferFrom() 可以把数据直接在内核层搬运，避免用户态/内核态来回倒腾
    2. 在复制大文件（或者文件通过 socket 发送出去）时，性能优势非常明显
  4. 适合与 NIO 生态统一
    1. 如果你项目里既需要网络非阻塞 NIO，也需要文件操作，FileChannel 让接口和类型更统一

SocketChannel
SocketChannel 概述

SocketChannel 是客户端连接远程服务器时使用的通道，既支持阻塞模式，也支持非阻塞模式。在非阻塞模式下，可以将多个通道注册到同一个 Selector，再通过轮询机制高效地管理和处理大量连接。

Selector

Selector 概述

Selector 是 SelectableChannel 对象的多路复用器，能够检测多个已注册通道上是否有事件发生，如果有事件发生，就会获取相应的事件，并对每个事件进行处理，从而实现一个线程同时监控多个 SelectableChannel 的 IO 状态

异步非阻塞操作（AIO）

Java AIO（异步非阻塞式）采用 “一个有效请求对应一个线程” 的模式，客户端的 I/O 请求由操作系统先完成，然后再通知服务器应用启动线程进行处理。通常适用于连接数较多、且连接时间较长的场景。

因此 Java AIO 适用于连接数多且连接时间较长（重操作）的架构，例如相册服务器。它能够充分利用操作系统参与并发处理。从 JDK 7 开始提供支持，但编程相对复杂。
