Redis 数据类型

数据类型一览图

[!NOTE] 注意事项

  1. 这里提到的数据类型，指的都是 值（Value）的数据类型。至于键（Key），在 Redis 中始终都是字符串类型

字符串（String）

String 是二进制安全的字符串，简单来说，它能存放各种数据，例如普通文本、数字、图片、序列化对象等等。表面上叫 “字符串”，实际上它能承载的内容远超字符串本身。

[!NOTE] 注意事项

  1. 单个 String 的最大大小是 512 MB。

位图（Bitmaps）

Bitmaps 就是把一个二进制字符串拿出来，按 bit 位进行操作

位域（Bitfield）

Bitfield 就是把一个二进制字符串拿出来，把一段连续的 bit 位拼成一个小整数来操作

哈希（Hash）

Hash 就是一个键值对集合，它的键叫做 field，它的值叫做 value

[!NOTE] 注意事项

  1. 单个 Hash 理论上可以存储 232-1 个键值对，即 42_9496_7295 个 field

列表（List）

List 是一个有序的字符串集合，底层实现是双端链表，既支持从左端插入/取出元素，也支持从右端插入/取出元素。

[!NOTE] 注意事项

  1. 单个 List 最多可以存储 232-1 个元素

集合（Set）

Set 是一个无序、去重的字符串集合，底层实现是哈希表

[!NOTE] 注意事项

  1. 由于 Set 底层实现是哈希表，所以增删改查的复杂读都是 O(1)
  2. 单个 Set 最多可以存储 232-1 个元素

1.2.5. 数据写入机制

1.2.5.1. 元数据写入
1.2.5.2. 数据写入

数据写入在北斗七剑的选择如下：

  1. 天枢剑（节点的拓扑结构）：主从复制（分区主从）
    1. Redis Cluster 采用虚拟槽分区，共计 16384 个槽（编号从 0 开始）。而一个主节点通常对应一个分区，本质上是由多个哈希槽组成的槽集合。
    2. 每个 Key 通过哈希函数（CRC16[key] & 16383）计算后，映射到编号为 0 到 16383 的槽内，槽隶属于哪个主节点，数据就存储在哪个主节点，而该主节点的从节点则负责对这些数据进行同步。
  2. 天璇剑（数据的组织形式）：Key-Value 结构（扁平化 Key-Value 结构）
    1. Redis 的核心数据模型就是一个巨大的字典（Dictionary），所有数据都以键值对的形式存储
  3. 天权剑（数据的分布策略）：整体存储（分区）
    1. 每个 Key 通过哈希函数（CRC16[key] & 16383）计算后，映射到编号为 0 到 16383 的槽内，槽隶属于哪个主节点，数据就存储在哪个主节点
  4. 天玑剑（数据的同步时机）：
    1. 同步复制
      1. Redis 提供了 WAIT 命令，可以临时将异步复制变为同步复制
    2. 半同步复制
      1. Redis 提供了 WAIT 命令，可以临时将异步复制变为半同步复制
    3. 异步复制（默认）
      1. 当主节点收到一个写请求后，它会先在自己的内存中执行该命令，然后立即向客户端返回成功响应
      2. 之后，它才将这个写命令放入一个复制缓冲区（Replication Buffer），在后台异步地被从节点拉取
  5. 玉衡剑（数据的同步传输）：快照 + 增量（日志拉取）
    1. 需要注意的是，这里的日志并不像 MySQL 那样的落盘二进制日志（Binary Log），而是基于命令的实时日志流，保存在内存的复制缓冲区中
  6. 开阳剑（数据的协调机制）：单点写入
  7. 瑶光剑（数据的持久化保障）：
    1. 持久化逻辑：
      1. 状态持久化（默认）
        1. 在 Redis 中，状态持久化叫做 RDB，是 Redis 默认使用的数据持久化方式，通过将某个时刻的内存数据快照保存为一个二进制文件（默认名为 dump.rdb），每次保存的是整个数据库的完整状态。
        2. Redis 会通过 fork 创建子进程，由子进程负责将数据写入快照文件。由于 fork 需要将主进程的页表复制一份，当数据量较大时，这一过程会带来明显的开销，从而对性能造成一定影响。
      2. 操作持久化（重写）
        1. 在 Redis 中，操作持久化叫做 AOF，会将 Redis 所执行的每一条写命令按顺序追加记录到日志文件中（默认文件名为 appendonly.aof）
      3. 混合持久化（推荐）
        1. 在 Redis 中，允许 RDB 和 AOF 一起开启。
    2. 存储引擎实现：
      1. RDB 的存储引擎实现：
        1. 写时复制（不依赖 WAL）
      2. AOF 的存储引擎实现：
        1. 写时复制（依赖 WAL，直接写入 WAL）
        2. 需要注意的是，这与传统数据库直接写入 WAL 略有不同，我们只是将 WAL 日志刷入磁盘（未依赖其他 WAL），而 Redis 的数据本体仍停留在内存中，并未被刷入磁盘文件。
    3. 操作系统交互：
      1. RDB 的操作系统交互：强制刷盘（定期分块刷盘、操作系统自由安排）
        1. Redis 将 RDB 的 快照 的刷盘时机，交由我们自己选择，我们可以定期分块刷盘、操作系统自由安排
        2. Redis 为我们提供了定期分块刷盘，避免一次性在最后刷盘造成长时间卡顿。
      2. AOF 的 WAL：强制刷盘（同步刷盘、间隔刷盘、操作系统自由安排）
        1. Redis 将 AOF 的 WAL 的刷盘时机，交由我们自己选择，我们可以配置同步刷盘、间隔刷盘、操作系统自由安排

1.4.11. Swap 分区要求

swap 分区，简单来说，是操作系统在系统内存不足时，将某些程序不活跃或暂时不需要的数据移动到 swap 分区（位于硬盘），以释放内存资源供当前活跃程序使用。

由于 swap 分区依赖硬盘，其读写速度远低于内存。如果系统频繁使用 swap，会造成大量磁盘 I/O，严重影响系统性能，甚至导致系统响应迟缓或无响应。因此，在性能敏感的场景（如数据库、实时计算、大内存服务器）中，通常建议禁用 swap，确保数据始终在高速内存中运行。

我们推荐 RabbitMQ 服务器关闭 Swap 分区

相关注解

@EnableCaching

在 启动类或配置类 上添加 @EnableCaching 注解，用于 启动注解式缓存

@CacheConfig

在 类 上添加 @CacheConfig 注解，用于 在类级别配置统一的缓存属性，让这个类的方法在使用缓存注解时，自动继承里面设置的默认值，避免重复填写

@Cacheable

在 方法 上 添加 @Cacheable 注解，用于 在方法执行前，先根据 Key 查询缓存，通常用于 查询 方法

  1. 如果缓存中有数据，则直接取缓存，不会执行方法
  2. 如果缓存中没有数据，就正常执行方法，并把方法的返回值放进缓存中

@CachePut

在 方法 上添加 @CachePut 注解，用于 在方法执行后，根据 Key 将方法返回值强制写入缓存，并覆盖已有缓存内容，通常用于 新增或更新 方法

需要注意的是，它与 @Cacheable 不同，即使缓存中已存在对应数据，标注了 @CachePut 的方法也一定会被执行，不会因缓存命中而跳过

@CacheEvict

在 方法 上添加 @CacheEvict 注解，用于 在方法执行后，根据 Key 将缓存删除，通常用于 删除 方法

@Caching

由于在同一个方法上，Spring 不支持直接标注多个相同的注解，例如标注多个 @Cacheable，后面的会覆盖前面的

在 方法 上添加 @Caching 注解，用于 将多个注解打包在一起，在一个方法上同时标注多个相同的注解

读操作基本流程

读操作基本思路

读操作通常先查 Redis，若未命中则回源 MySQL，将结果返回给客户端并写回 Redis

为避免 Redis 缓存未命中时多个线程同时访问数据库，可以让第一个请求在查询数据库前加一把互斥锁（分布式锁）。后续请求检测到有锁则等待，等第一个请求完成并回填缓存后，其他请求便直接从 Redis 读取数据，无需再访问数据库。如果每次读都一锁到底，即便 Redis 里早就有数据，也会让请求干等着，效率很低，通常会配合 “双重校验” 策略来提高并发下的效率。

写操作基本流程

写操作基本思路

我们需要确保 Redis 中的数据与数据库中的值保持一致，常见的思路包括：

  1. 先更新数据库，再更新缓存
    1. 问题一
      1. 如果更新 MySQL 成功，但更新 Redis 失败，导致 MySQL 与 Redis 数据不一致
    2. 问题二
      1. 多线程情况下，线程的执行速度有快有慢，如果 A 线程先更新数据库为 99，然后 B 线程再更新数据库为 90，此时如果 B 比 A 先执行，把 Redis 更新成 90 ，然后 A 再把 Redis 更新成 99，就出现了问题
      2. 想要解决这个问题，可以使用强一致性写，也就是使用分布式事务，但是一般很少有人这样做
  2. 先更新缓存，再更新数据库
    1. 不太推荐，因为业务上一般把 MySQL 作为底单数据库
    2. 此外仍然有上述两个问题
  3. 先删除缓存，再更新数据库
    1. 这是让客户端自动去 Redis 中拿数据过来，但会有一个问题
      1. 问题一：
        1. 就是如果 A 线程把缓存删了，然后更新数据库，B 这时候发现缓存没有了，去找数据库，如果 A 的操作还没有执行完，然后 B 把 这个数据回写到 Redis 中，那 A 提交后 MySQL 和 Redis 的数据不一致，但是由于 Redis 已经有数据了，不会再去数据库中拿了，就出现了问题
        2. 想要解决这个问题，只能在更新数据时，暂停读请求，等更新完在读数据，理论上能达到强一致，但生产环境下很少这样做，分布式环境下很难做到双系统实时一致性，一般都是考虑最终一致性
        3. 通常使用延时双删，也就是线程 A 先把 Redis 缓存删了，然后再更新数据库，暂停一段时间，然后再删除一次 Redis
          1. 暂停的这一段时间就是让线程 B 先读到 MySQL 中的数据，回写进缓存之后，你其他的线程看到 Redis 有数据了，就不会来到我 MySQL 中了对吧，然后就不会出现我把 Redis 缓存刚删除，你拿着我更新数据库之前的脏数据又写到 Redis 了我再把你的缓存删了，这样下一次进程来了就会访问数据库拿到新的数据并写到 Redis 中
          2. 所以暂停的时间要确保能让 B 先读再写到 Redis，这个一般有两种方法确定
            1. 在业务程序运行的时候，统计下线程读数据和写缓存的操作时间，自行评估自己的项目的读数据业务逻辑的耗时，然后在这个基础上加上个百毫秒就行
            2. 看门狗机制（WatchDog），能够自动加时
          3. 为了优化并发下的性能情况，通常将第二次删除作为异步操作
      2. 问题二：
        1. 如果先删除了缓存，更新数据库的时候出现了问题，那其他线程访问数据库，读到的还是脏数据，或者你把 Redis 都删除了，请求打到了 MySQL，你更新成功了还好说，但是你更新失败了也打到 Redis，那我还要你 Redis 干嘛呢
        2. 解决方案：
          1. 补偿机制，否则抗风险能力不足
  4. 先更新数据库，再删除缓存（推荐）
    1. 先更新数据库，再删除缓存，看似 Redis 中有缓存，没人影响线程 A 更新和删除，但是考虑到极限情况下，如果恰好 Key 过期，然后又出现了把 Redis 缓存刚删除，你拿着我更新数据库之前的脏数据又写到 Redis的情况下
    2. 延时双删
      1. 数据库更新后，立马删除缓存，然后延长一小段时间（比如 100 ms）再删除一次，这样即使 B 在第一次删除之后回写了脏数据，延时的第二次删除也能把它清掉。
      2. 第一次删除是为了 A 更新数据库后，B 过来查数据然后查到最新的数据写回缓存，不考虑这个的情况下，我们可以只延时删除一次，直接使用异步删存
      3. 第二次删除是为了 A 更新数据库后，删除缓存，B 把 A 更新之前的数据写回缓存，再删除一次，可以作为异步操作（伪异步 IO，性能终究还是不行地，也可以尝试其他的异步操作，例如 Canal + 消息队列）
      4. 删一次删两次看心意，反正都反正了
    3. 更新数据库完了，但是删除缓存的时候 Redis 宕机了，那我们不能不删除阿
    4. 异步删存
      1. 把删除发送到消息队列，然后依赖 MQ 的重试机制删除缓存，一般只需要删除一次，毕竟你也不差第一次删除那点失误了，如果长时间没有删除，那就需要汇报给运维了

缓存预热概述

在系统刚启动或者 Redis 刚部署/重启后，就提前把热点数据加载到 Redis 缓存里，而不是等用户第一次访问时才去查询数据库再写入缓存，可能造成大量请求都去数据库，数据库压力激增，可能直接崩溃

缓存雪崩

缓存雪崩概述

在 Redis 中，缓存的数据通常会设置过期时间（TTL），如果某一类数据（比如热点商品、热门文章）同一时间大批量过期，或者 Redis 宕机，短时间内所有请求都查不到缓存，于是大量请求都去数据库，数据库压力激增，可能直接崩溃

缓存过期时间随机化

给缓存设置过期时间时，加上一个随机值（比如 random(0, 300)），避免大量 key 在同一时间点集中失效

缓存永不过期 + 异步更新

缓存数据本身设置为物理永不过期，保证客户端访问时始终有缓存可用。同时通过异步任务在后台定期或按需刷新缓存内容，从而平衡高可用与数据新鲜度。
