Redis 数据类型

数据类型一览图

[!NOTE] 注意事项

  1. 这里提到的数据类型，指的都是 值（Value）的数据类型。至于键（Key），在 Redis 中始终都是字符串类型

字符串（String）

String 是二进制安全的字符串，简单来说，它能存放各种数据，例如普通文本、数字、图片、序列化对象等等。表面上叫 “字符串”，实际上它能承载的内容远超字符串本身。

[!NOTE] 注意事项

  1. 单个 String 的最大大小是 512 MB。

位图（Bitmaps）

Bitmaps 就是把一个二进制字符串拿出来，按 bit 位进行操作

位域（Bitfield）

Bitfield 就是把一个二进制字符串拿出来，把一段连续的 bit 位拼成一个小整数来操作

哈希（Hash）

Hash 就是一个键值对集合，它的键叫做 field，它的值叫做 value

[!NOTE] 注意事项

  1. 单个 Hash 理论上可以存储 232-1 个键值对，即 42_9496_7295 个 field

列表（List）

List 是一个有序的字符串集合，底层实现是双端链表，既支持从左端插入/取出元素，也支持从右端插入/取出元素。

[!NOTE] 注意事项

  1. 单个 List 最多可以存储 232-1 个元素

集合（Set）

Set 是一个无序、去重的字符串集合，底层实现是哈希表

[!NOTE] 注意事项

  1. 由于 Set 底层实现是哈希表，所以增删改查的复杂读都是 O(1)
  2. 单个 Set 最多可以存储 232-1 个元素

0.2.3. 节点角色与关系

集群各节点之间是对等关系：

  1. 无中心节点
    1. 集群中没有所谓的 “主” 节点来统一调度，所有节点彼此平等，地位对等、功能对等。
  2. 元数据全量同步
    1. 任何一个节点上的元数据操作，都会被实时同步到整个集群，确保所有节点信息一致
  3. 平等的客户端接入点
    1. 客户端可以直接连接集群中的任意节点来执行操作，每个节点都是一个功能完备的入口。

0.2.5. 数据写入机制

0.2.5.1. 元数据写入

RabbitMQ 的元数据存储在 Mnesia 数据库中。Mnesia 是 Erlang 内置的分布式数据库，专为 Erlang 应用设计。每个 RabbitMQ 节点都会运行一个独立的 Mnesia 实例，这些实例共同组成一个集群。因此，这里描述的主要是 Mnesia 的写入流程。

元数据写入在四个维度上的选择如下：

  1. 维度一：同步复制（全同）
  2. 维度二：多主复制（同步多主）
  3. 维度三
    1. 底层技术：快照 + 增量同步（基于日志）
    2. 实现方式：类 2PC 的事务机制
    3. 详细流程：
      1. 客户端会将元数据变更请求发送到集群中的任一节点，我们把它称为 协调节点。
      2. 协调节点在收到请求后，会首先启动一个 Mnesia 事务，为了保证隔离性和一致性，协调节点会主动向所有持有该表副本的节点发送锁请求。
        1. 并不是像标准 2PC 那样先进入 “准备阶段”，而是先尝试把相关资源的锁都拿到手
        2. 一般数据库都是边执行边逐步申请锁，而 Mnesia 是以访问模式尝试获取相应的锁
      3. 等所有相关节点都成功加锁后，协议才会继续进入接近传统 2PC 流程的后续阶段
  4. 维度四：整体保存
    1. 元数据会被完整的复制到每一个节点上

0.4.10. Swap 分区要求

swap 分区，简单来说，是操作系统在系统内存不足时，将某些程序不活跃或暂时不需要的数据移动到 swap 分区（位于硬盘），以释放内存资源供当前活跃程序使用。

由于 swap 分区依赖硬盘，其读写速度远低于内存。如果系统频繁使用 swap，会造成大量磁盘 I/O，严重影响系统性能，甚至导致系统响应迟缓或无响应。因此，在性能敏感的场景（如数据库、实时计算、大内存服务器）中，通常建议禁用 swap，确保数据始终在高速内存中运行。

我们推荐 Redis 服务器关闭 Swap 分区

1. database
database 是指 Redis 的数据库编号，其默认有 16 个逻辑库，设计初衷是用于区分不同业务，比如库 1 存 Session，库 2 存验证码，避免数据混淆。

但在实际生产中，大多数项目只用 DB 0，然后通过 Key 前缀来区分业务，例如：

  1. user:session:123
  2. order:lock:456
  3. sms:code:789

主要原因包括：

  1. 多个库并不是真正的环境隔离，仅是命令作用的划分
  2. Redis Cluster 模式只支持 DB 0
  3. 很多客户端或中间件默认只能使用 DB 0

相关注解

@EnableCaching

在 启动类或配置类 上添加 @EnableCaching 注解，用于 启动注解式缓存

@CacheConfig

在 类 上添加 @CacheConfig 注解，用于 在类级别配置统一的缓存属性，让这个类的方法在使用缓存注解时，自动继承里面设置的默认值，避免重复填写

@Cacheable

在 方法 上 添加 @Cacheable 注解，用于 在方法执行前，先根据 Key 查询缓存，通常用于 查询 方法

  1. 如果缓存中有数据，则直接取缓存，不会执行方法
  2. 如果缓存中没有数据，就正常执行方法，并把方法的返回值放进缓存中

@CachePut

在 方法 上添加 @CachePut 注解，用于 在方法执行后，根据 Key 将方法返回值强制写入缓存，并覆盖已有缓存内容，通常用于 新增或更新 方法

需要注意的是，它与 @Cacheable 不同，即使缓存中已存在对应数据，标注了 @CachePut 的方法也一定会被执行，不会因缓存命中而跳过

@CacheEvict

在 方法 上添加 @CacheEvict 注解，用于 在方法执行后，根据 Key 将缓存删除，通常用于 删除 方法

@Caching

由于在同一个方法上，Spring 不支持直接标注多个相同的注解，例如标注多个 @Cacheable，后面的会覆盖前面的

在 方法 上添加 @Caching 注解，用于 将多个注解打包在一起，在一个方法上同时标注多个相同的注解

缓存雪崩

缓存雪崩概述

在 Redis 中，缓存的数据通常会设置过期时间（TTL），如果某一类数据（比如热点商品、热门文章）同一时间大批量过期，短时间内所有请求都查不到缓存，于是全部请求都去数据库，数据库压力激增，可能直接崩溃

缓存雪崩解决方案

缓存过期时间随机化

给缓存设置过期时间时，加上一个随机值（比如 random(0, 300)），避免大量 key 在同一时间点集中失效

  • 过期时间随机化（错峰过期）

    • 给缓存设置过期时间时，加上一个随机值（比如 + random(0, 300) 秒）

    • 避免大量 key 在同一时间点集中失效

  • 缓存永不过期 + 后台异步更新

    • Redis 层面不设置过期时间

    • 用定时任务/消息通知去更新缓存

    • 前端查询时永远有数据，不会出现大面积空缓存

  • 热点数据加锁重建（互斥锁）

    • 当缓存失效时，只有第一个线程去数据库加载数据

    • 其他线程等待，避免同时打 DB

    • 可以用 Redis 的分布式锁实现

  • 多级缓存（本地缓存 + 分布式缓存）

    • 比如 Guava Caffeine（JVM 内存） + Redis

    • 当 Redis 数据过期，本地缓存还能顶一顶，缓冲数据库压力

  • 缓存预热（结合前面说的）

    • 在系统上线或缓存重启后，提前加载热点数据到缓存

    • 避免冷启动时，全部请求穿透到数据库

  • 服务限流 & 降级

    • 在极端情况下，限制流量进入数据库

    • 或者直接返回兜底数据（比如提示“系统繁忙”）
