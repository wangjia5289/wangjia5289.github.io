笔记：Spring WebFlux

Last updated on 2025-05-06T16:46:56+08:00

一、理论

1. Spring WebFlux 概述

==1.现有问题==
随着互联网应用的日益复杂和用户对实时性要求的提高，传统的基于阻塞 I/O 的Web框架在处理高并发和大数据流时逐渐显现出瓶颈。

==2.响应式编程解决方案==
基于以上问题，响应式编程模型应运而生，响应式编程是一种处理异步数据流、变化传播、非阻塞的编程范式，在响应式编程中，数据被建模为流，操作以非阻塞方式执行，从而保持应用程序在高负载下的响应性。

==3.Spring WebFlux 是什么==
Spring WebFlux，作为 Spring Framework 的响应式Web框架，基于 Reactive Streams 规范构建，采用事件驱动 + 非阻塞 I/O 的架构设计，特别适合处理高并发、低延迟和 I/O 密集型任务。

2. 对 Spring WebFlux 深入理解

==1.Spring Web（Servlet 栈）==
Spring Web 采用传统的阻塞式编程模型，基于 Servlet API，默认使用 Servlet 容器（如 Tomcat）。其线程模型为每个请求分配一个线程，并在请求处理期间阻塞该线程，直到响应生成完毕（线程阻塞）。其缺乏内建的流式数据处理支持，且依赖多线程调度，其内存开销较大。在高并发场景下，受限于线程池大小和阻塞模型，吞吐量较低。然而，其延迟较低，适合传统的请求/响应模型，尤其是 CPU 计算密集型 或 请求阻塞不明显 的传统 Web 应用开发。

对线程阻塞的深入理解：

  1. 一个请求通常由一个线程全程处理。当程序执行到某个耗时操作（如阻塞的网络请求、阻塞的数据库查询、文件读写、或 CPU 密集计算）时，这个线程会停下来等待，直到结果返回后才能继续往下执行。
  2. 在这段等待时间里，线程既不做其他事，又无法释放，造成严重的资源浪费，系统吞吐量受限。
  3. 虽然我们可以通过异步线程池来“解耦”这些耗时操作，把任务转交给其他线程执行，释放主线程，看起来像是实现“非阻塞”，但这其实只是线程的转移，本质上仍是阻塞，只不过阻塞发生在别的线程上。更糟的是：如果主线程还需要等待异步结果返回，等于主线程和工作线程两个线程都被卡住了，反而加重了系统开销。这种方式可以称为“伪非阻塞”。

==2.Spring WebFlux（Reactive 栈）==
Spring WebFlux 基于 响应式编程模型，默认使用 Reactive 容器（如 Netty），通过 事件驱动 和 非阻塞 I/O 实现高效处理。它借助 Reactor 库（Mono 和 Flux）实现异步操作，能够以较少的线程处理更多请求，资源消耗显著降低。在 I/O 密集型 场景下，其吞吐量和响应速度表现优异，尽管响应链较长可能导致延迟略高。Spring WebFlux 特别适合 高并发 和 大量 I/O 请求 的应用场景，例如聊天系统、实时通知和数据流处理等。

对非阻塞 I/O 的深入理解：

  1. Spring WebFlux 的非阻塞 I/O 模型就不同了。它不是靠线程切换来“逃避阻塞”，而是通过一种事件驱动 + 回调机制来彻底规避阻塞。我们必须注意的是：“Spring WebFlux 只是提供了非阻塞的条件，但到底能不能做到非阻塞，还要看你具体的技术栈”
  2. 例如，当主线程遇到一个耗时操作时，如果这个操作是异步非阻塞的（比如非阻塞网络请求、非阻塞数据库查询），主线程执行之后不会傻乎乎地干等，而是注册一个事件和对应的回调函数，然后立刻去处理其他任务。如果有数据返回，比如 R2DBC 查询返回了一条数据，就会触发事件，执行回调函数。假如操作还没彻底完成，只是返回了部分数据，主线程会继续忙其他事，等下次再返回数据时再回调、再继续后续业务流程。
  3. 但是，如果这个操作是同步简单操作、同步阻塞操作，那主线程还是要老老实实执行完这个步骤才能往下走，可是别忘了，我们的主 Reactive 线程是为成千上万个任务服务的，你让它在一个请求上耗太久，这显然是不行的，所以，我们约定：在主 Reactive 流（如 flatMap、doOnNext 等操作符内部）中，不得防止任何阻塞操作，只允许包含两类操作：同步简单操作、异步非阻塞
  4. 那你可能询问，同步阻塞操作我们该怎么操作，对于这些操作，WebFlux 允许你把它们丢到一个专门的 “阻塞友好线程池” 里去执行。执行完后，结果会返回给主线程。但是这跟传统阻塞模型不一样，虽然异步线程仍然阻塞，但是主线程在这个过程中不会一直傻等，而是继续处理其他任务，等结果回来后再回来处理这个请求。这样可以避免主 Reactive 线程被长时间占用，保持它轻量、高效地应付大量请求。
  5. 这样我们就可以实现，除了主 Reactive 线程执行同步简单操作时消耗点时间，执行异步阻塞、异步非阻塞操作时，都不会被阻塞，可以去执行其他任务，这样就能实现少量线程执行大量请求

==3.两者对比表==

项目 Spring Web（Tomcat） Spring WebFlux（Netty）

资源占用 多线程调度，线程上下文开销大 少量线程处理大量请求，资源占用低

  1. “元素”的概念是基于 Mono<T> 中的泛型类型来理解的。比如 Mono<String> 中一个 String 类型就是一个元素；如果是 Mono<Object>，那一个 Object 类型就是一个元素。

1.2. Mono 相关方法

1.2.1. 前言

我们约定：在主 Reactive 流（如 flatMap、doOnNext 等操作符内部）中，不得放置任何阻塞操作，只允许包含两类操作：

  1. 同步简单操作
  2. 异步非阻塞操作

对于 同步阻塞操作，必须通过 mono.subscribeOn 将其调度到 “阻塞友好线程” 中，因此，以后我们提到 “异步阻塞操作” ，实际上是指：将原本是同步阻塞的操作，调度到阻塞友好线程中执行，从而避免阻塞 Reactor 的工作线程，保持响应式流的非阻塞特性。

1.2.2. 创建 Mono 实例
  1. Mono.just(T data)：
    1. 从一个确定且非 null 的值创建 Mono，直接发出该值
  2. Mono.justOrEmpty(T data)：
    1. 从一个可能为 null 的值创建 Mono，若为 null 则返回空 Mono
  3. Mono.empty()：
    1. 创建不发出任何元素且立即完成的空 Mono
  4. Mono.fromCallable(Callable<R>)：
    1. 从 Callable 创建 Mono，无论 Callable 返回值是什么，最终都将被包装为一个 Mono，适用于封装同步简单操作、异步阻塞操作、异步非阻塞操作
    2. 这里异步阻塞操作指的是结合 subscribeOn 异步化阻塞操作（使成为异步阻塞操作），避免占用主 Reactive 线程过多时间
1.2.3. 转换操作
  1. mono.map(Function<T, R> mapper)：
    1. 将 Mono 中的元素映射为新值（同步转换），返回新 Mono
  2. mono.flatMap(Function<T, Mono<R>> mapper)：
    1. 将元素执行操作，映射为新的 Mono，适用于执行同步简单操作、异步阻塞操作、异步非阻塞操作
  3. mono.cast(Class<R> type)：
    1. 对元素进行强制类型转换
  4. mono.ofType(Class<R> type)：
    1. 过滤元素，仅当类型匹配时发出，否则返回空 Mono
  5. mono.switchIfEmpty(Mono<? extends T> fallback)：
    1. 若 Mono 为空，切换到指定的备用 Mono。
  6. mono.defaultIfEmpty(T defaultValue)：
    1. 如果当前 Mono 为空，提供一个默认值作为替代
1.2.4. 错误处理
  1. mono.onErrorResume(Function<Throwable, Mono<? extends T>> fallback)：
    1. 用于处理 Mono 或 Flux 的错误信号，当上游发生异常时，返回一个替代的 Mono 以继续流程
  2. mono.doOnError(Consumer<Throwable> onError)：
    1. 当错误发生时执行副作用操作（如日志记录），与 onErrorResume 不同的是，只执行副作用，不影响数据流
  3. mono.onErrorReturn(T fallbackValue)：
    1. 出错时返回一个固定的默认值
  4. mono.retry(long times)：
    1. 出错时重试整个 Mono 流程 times 次（总共尝试 1 + times 次）
1.2.5. 多流组合
  1. mono.zipWith(Mono<? extends U>other，BiFunction<T，U，R>combiner)：
    1. 将当前 Mono<T> 与另一个 Mono<U> 组合，通过 BiFunction 合并为结果 R，返回 Mono<R>
    2. 任一 Mono 为空则返回空 Mono
  2. Mono.zip(Mono<? extends T1> m1, Mono<? extends T2> m2, ...)：
    1. 同上，打包多个 Mono 的结果为 Tuple，返回 Mono<TupleN>
    2. 任一 Mono 为空则返回空 Mono
1.2.6. 生命周期钩子
  1. mono.doOnNext(Consumer<? super T> onNext)：
    1. 在每次发出元素（onNext）时执行副作用操作
  2. mono.doOnSubscribe(Consumer<? super Subscription> onSubscribe)
    1. 订阅发生时执行副作用
  3. mono.doonSuccess(Consumer<? super T> onSuccess)
    1. 成功完成时执行副作用操作
  4. mono.doFinally(Consumer<SignalType> onFinally)
    1. 在终止（完成、错误或取消）时执行副作用操作
1.2.7. 异步与调度
  1. ==常见调度器==：
    1. Schedulers.parallel()：
      1. 适用于 CPU 密集型任务，使用固定大小线程池
    2. Schedulers.boundedElastic()：
      1. 适用于 I/O 密集型任务（如数据库、网络、文件操作）
    3. Schedulers.immediate()：
      1. 在当前线程中执行
    4. Schedulers.fromExecutor(Executor)：
      1. 使用自定义线程池调度任务
  2. mono.subscribeOn(Scheduler scheduler)：
    1. 指定整个订阅过程在哪个调度器线程上运行（影响源的执行线程）
    2. 多个 subscribeOn 只会生效第一个
  3. mono.publishOn(Scheduler scheduler)：
    1. 指定 publishOn 之后的操作在那个线程池执行
  4. mono.delayElement(Duration delay)：
    1. 延迟指定时间后再发出 Mono 元素，常用于模拟延迟
  5. mono.timeout(Duration timeout)：
    1. 在指定时间内未完成则抛出 TimeoutException，用于超时控制
1.2.8. 订阅
  1. mono.sunscribe()：
    1. 仅触发流的执行，不处理返回值
  2. mono.subscribe(Consumer, Consumer, Consumer)：
    1. 既触发流的执行，也处理返回值
