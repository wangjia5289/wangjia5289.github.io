2.2. Lambda 表达式

2.2.1. Lambda 表达式概述

想象这样一个问题：我们的目的仅仅是传递一个回调方法，却被迫要创建一个的匿名内部类，然后再创建其实例，再通过其实例调用方法，显得非常笨重和繁琐

为了解决这一问题，Java8 引入了 Lambda 表达式。Lambda 表达式是一种语法糖，在编译阶段会被编译器自动转换为对应的函数式接口的具体实现类，最终返回该函数式接口的具体实现类实例，帮我们简化了一些步骤

[!NOTE] 注意事项

  1. 尽管 Lambda 表达式功能强大，但它只能用于函数式接口，返回一个函数式接口的具体实现类实例
  2. Lambda 表达式使用的是 invokedynamic 字节码指令来创建匿名内部类和匿名内部类实例的。详见笔记：JVM（obsidian 内部链接：，Hexo 链接：）

2.2.2. Lambda 表达式编写方式

Lambda 表达式由三部分组成：

  1. 参数列表
    1. 明确指出参数类型
      1. (int a, int b)
    2. 根据上下文自动推断参数类型
      1. (a, b)
    3. 需要注意的是：
      1. 不允许出现既明确指出参数类型，又根据上下文自动推断参数类型的情况（例如 (int a, b)）
      2. 只有在恰好只有一个参数，并且是根据上下文自动推断参数类型的情况下，Lambda 表达式的参数列表才可以省略 ()。在其他所有场景（包括没有参数），() 都必须显式写出（例如 a 是正确的，但是 Object a 是不正确的）
  2. ->
  3. 方法体
    1. 如果方法体只有一行
      1. 如果有 return
        1. 可以省略 return（例如 return a + b ➔ a + b）
      2. 如果没有 return
        1. 直接写即可（例如 System.out.println("Hello World")）
    2. 如果方法体有多行
      1. 必须有 {}，并且不能省略 return（例如 { int c = a + b; return c; }）

五、Java Stream

1. Java Stream 概述

我们知道 “在 Java 应用中，遍历某个数据结构有两种方式：基于索引遍历、基于引用遍历”。为了避免每一次想要遍历某个数据结构时，都要手动编写数据结构的遍历逻辑。所以我们通常使用迭代器封装数据结构的遍历逻辑，遍历时直接使用迭代器进行数据结构的遍历（需要注意的是：使用迭代器进行数据结构的遍历，同样需要借助循环）。可即便如此，我们还是发现：使用迭代器进行数据结构的遍历时，操作仍然显得非常繁琐

于是 Java 语言的设计团队提供了增强 for 循环语法糖来简化操作。可尽管如此，如果涉及到多个操作，操作仍然会显得太繁琐：

[!NOTE] 注意事项

  1. 增强 for 循环（for-each 循环）是 Java 提供的一种语法糖，在编译阶段会被编译器自动转换为使用迭代器进行数据结构的遍历（需要注意的是：是使用迭代器实现数据结构的遍历，而不是使用可拆分迭代器实现数据结构的遍历）

[!NOTE] 注意事项
2. 如果仅仅是为了代码看起来优雅，Java 语言的设计团队是不可能大动干戈引入 Java Stream 这一庞大的体系的。其实 Java Stream 被引入的真正原因是：惰性求值、并行处理
1. 惰性求值
1. 当执行 Java Stream 的中间方法时，并不会像传统的处理方法一样：执行每一步时都会创建一个新的数据结构，然后遍历某个数据结构，让数据结构中的每个元素都执行这一步，将元素被这一步处理后的结果 “增” 进这个新的数据结构中。执行完这一步，再执行下一步（即一步一步来）
2. 相反，在执行 Java Stream 的中间方法时，只是会记录下这一步的链式逻辑，并不会立即执行。直到执行到 Java Stream 的结束方法时，Java Stream 才会按照链式逻辑遍历某个数据结构，让数据结构中的每个元素都执行完整个链式逻辑。处理完这一个元素，再处理下一个元素，从而避免了大量的中间的数据结构的创建（即一个一个来）
3. 需要注意的是：90% 都是基于上述的流程，但是也有 10% 并不是这样。这是因为涉及到了排序方法，这样的话需要让 Java Stream 中的每个元素都执行完这一步，然后再按上述的流程继续执行
2. 并行处理
1. 我们知道 “在 Java 应用中，遍历某个数据结构有两种方式：基于索引遍历、基于引用遍历”。如果让我们自己编写数据结构的遍历逻辑，我们很可能是使用单线程进行数据结构的遍历
2. 因为如果我们希望使用多线程进行数据结构的遍历，就必须提前将数据结构中的元素进行拆分，然后手动创建新的数据结构来保存每一段被拆分后的元素。当使用多线程进行数据结构的遍历后，我们还需要手动合并结果。除此之外，我们还需要处理多线程并发情况下的安全问题。这样一套整下来，操作就显得非常繁琐，不如直接使用单线程进行数据结构的遍历来的方便
3. 但是 Java Stream 已经帮我们封装了这些复杂逻辑，使我们能更轻松地使用多线程进行数据结构的遍历（并行流）
3. Java Stream 的并行流是使用 ForkJoinPool 实现分而治之、迭代拆分、工作窃取的。然而，我们日常在使用 ForkJoinPool 时，是需要手动编写元素或数据的迭代拆分逻辑的，你可能疑惑：Java Stream 的并行流的迭代拆分逻辑被写到了哪里？
1. 我们知道 “不管是 Java Stream 的串行流还是 Java Stream 的并行流，都要以可拆分迭代器为起点”。所以能创建 Java Stream 的并行流或 Java Stream 的串行流的数据结构，通常会包含一个实现 Spliterator＜T＞ 接口的内部类，而并行流的迭代拆分逻辑就被写在这个内部类中

[!NOTE] 注意事项

  1. 猴哥的烦恼箱 (｡•́︿•̀｡)：
    1. 为什么提供了特化流？使用普通流难道解决不了我们的问题吗？
      1. 因为普通流涉及到了 Java 泛型。我们知道 “一旦某个东西涉及到 Java 泛型，那么其在处理基本数据类型的元素时，就会出现很多问题”
      2. 我们先来回顾一下集合在涉及到 Java 泛型后，在处理基本数据类型的元素时出现的问题：
        1. 内存方面的压力
          1. 如果我们只是想存储一亿个基本数据类型的元素，但是由于集合涉及到 Java 泛型，所以集合中只能存储引用数据类型的元素（需要注意的是：实际存储的是引用数据类型的元素的引用）。所以基本数据类型的元素只能在自动装箱后再 “增” 进集合中
          2. 例如一个 int 类型的数据是 4 byte，但是一个 Integer 类型的数据在 32 bit 虚拟机下可能需要 16 byte，那么对于一亿个数据，就多消耗了 1525 MB 的内存（需要注意的是：由于集合中实际存储的是引用数据类型的元素的引用，这一亿个引用也占用了不小的内存）
        2. CPU 方面的压力
          1. 如果我们对集合中的元素做一次简单的计算操作（例如对每个元素执行 +1）时，就会触发这一亿个元素的自动拆箱 ➔ 计算 ➔ 自动装箱
          2. 除此之外，由于集合中实际存储的是引用数据类型的元素的引用，所以要访问某元素时，就要先通过其引用指向的内存地址找到该元素。这个寻址的过程本身就存在较大的开销，而且它并不像数组那样，是分配连续内存的数据结构，一次性就能加载多个元素，而是 “东一块西一块” 的，从而大大增加了寻址开销（术语是：CPU 缓存命中率低）
      3. 而普通流在涉及到 Java 泛型后，在处理基本数据类型的元素时会出现更多的问题：
        1. 如果我们使用普通流遍历的的是一个数组（需要注意的是：数组是 Java 中唯一既能存储基本数据类型的元素，又能存储引用数据类型的元素（需要注意的是：实际存储的是引用数据类型的元素的引用）的数据结构）
          1. 这相当于人为地降低处理效率。因为普通流涉及到 Java 泛型，所以只能处理引用数据类型的元素。所以即便数组中存储的是基本数据类型的元素，为了被普通流处理，也需要先对其进行自动装箱（即创建一亿个包装类实例，仅为了开始）
          2. 除此之外，还具有集合在涉及到 Java 泛型后，在处理基本数据类型的元素时出现的问题
        2. 如果我们使用普通流遍历的是一个集合（需要注意的是：由于集合涉及到 Java 泛型，所以集合中只能存储引用数据类型的元素（实际存储的是引用数据类型的元素的引用））
          1. 由于集合中存储的元素，已经是自动装箱后的元素了，所以不需要再对其进行自动装箱了。但是仍然具有集合在涉及到 Java 泛型后，在处理基本数据类型的元素时出现的问题
        3. 除此之外，还会存在 GC 方面的压力：
          1. Java Stream 中最初的元素，都是被所遍历的数据结构所引用的，所以不用担心它们被 GC
          2. 但是在 Java Stream 的执行的过程中，如果创建了新的实例（例如我们在某一个中间方法中，要对 Java Stream 中的每个元素做一次简单的计算操作（例如对每个元素执行 +1），那么就会产生一亿个新的包装类实例）。由于这些新的实例并不是数据结构中的元素，所以如果它们在下一步被拦截了（例如拦截了 100 万个实例），那么这些实例就会失去引用，需要被 GC。我的意思是：Java Stream 中涉及到的元素，除了数据结构中的元素和在结束方法中被收集到某处的元素，所有的中间元素都是 “垃圾”，需要被 GC（需要注意的是：数据结构中的元素和在结束方法中被收集到某处的元素会不会被 GC，还要看强软弱虚四引用）
          3. 需要注意的是：这是整个 Java Stream 体系的 “通病”，而不是因为普通流在涉及到 Java 泛型后，在处理基本数据类型时才出现这个问题。只不过由于普通流在涉及到 Java 泛型后，在处理基本数据类型的元素时出现了大量的自动装箱、自动拆箱，大大加重了这个问题
    2. 为什么特化流只提供了处理 int、long、double 类型的元素的流？
      1. 因为处理 byte、short、char 类型的元素，可以直接使用处理 int 类型的元素的流（因为 int 类型的范围涵盖了它们）
      2. 而处理 float 类型的元素，可以直接使用处理 double 类型的元素的流（因为 double 类型的精度涵盖了它）
      3. 并且 boolean 类型的元素通常不需要参与计算

Java Stream 问题爆破

Java Stream 的并行流问题爆破

虽然并行处理是 Java Stream 被引入的真正原因之一，但遗憾的是，Java Stream 的并行流在日常开发中并不常见，因为它有这样几个 “坑”：

  1. 我们在使用 Java Stream 的并行流时，很容易因为它的编写风格而产生 “我在使用单线程进行数据结构的遍历” 的感觉，从而忽视了多线程并发情况下的安全问题。虽然 Java Stream 已经帮我们处理了很多的多线程并发情况下的安全问题，但是你像在结束方法中，将 Java Stream 中的元素收集到非多线程并发安全的数据结构（例如 ArrayList＜E＞）中的情况，需要我们自己特别注意
  2. 我们知道 “Java Stream 的并行流是使用 ForkJoinPool 实现分而治之、迭代拆分、工作窃取的”。但是 Java Stream 的并行流默认使用的是单个 JVM 中全局唯一的 ForkJoinPool.commonPool()。所以一个 Java 服务中，所有 Java Stream 的并行流，甚至是不相关但涉及到 ForkJoinPool.commonPool() 的业务（例如 CompletableFuture＜T＞），都会去竞争 ForkJoinPool.commonPool() 中的线程
  3. 由于 Java Stream 的并行流使用了 ForkJoinPool，而一旦我们的线程涉及到让其他线程去执行一些任务时，就势必会出现一个问题：ThreadLocal 因为无法跨线程而 “丢失”
  4. 由于 Java Stream 的并行流使用了 ForkJoinPool，而使用 ForkJoinPool 本身就算有成本的（例如创建任务、线程调度、上下文切换、数据的迭代拆分、工作窃取、结果合并等），如果处理的数据并不多，那么使用 Java Stream 的并行流反而得不偿失
  5. 我们知道 “对于底层是基于非连续内存的数据结构，不太建议使用 Java Stream 的并行流”。所谓的 “不太建议” 是指：需要我们自行权衡，如果对数据结构中的元素进行迭代拆分的时间 ＞ 使用 Java Stream 的并行流节省的时间，那么我们是肯定不能使用 Java Stream 的并行流
