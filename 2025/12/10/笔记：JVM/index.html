2.2. 程序计数器

2.2.1. 程序计数器组成部分

2.2.2. 程序计数器概述

程序计数器（PC 寄存器）是用于记录当前线程要去执行的下一条 JVM 字节码指令的内存地址（需要注意的是：实际记录的是这条 JVM 字节码指令的偏移量）

[!NOTE] 注意事项

  1. 程序寄存器既不会发生 GC，也不会抛出 OutOfMemoryError 错误
  2. 程序寄存器是线程私有的区域，每个线程都拥有独立的程序寄存器
  3. 执行引擎负责读取并执行该 JVM 字节码指令。执行引擎在进行读取时，会将 JVM 字节码指令的偏移量转换为 JVM 字节码指令的内存地址。当执行引擎读取到 JVM 字节码指令后，程序计数器就记录当前线程要去执行的下一条 JVM 字节码指令的内存地址，然后执行引擎才开始执行该 JVM 字节码指令
  4. 如果线程执行的是 native 方法，那么程序计数器会记录 undefined（因为 native 方法本质上是离开了 JVM 的控制）
  5. 猴哥的烦恼箱 (｡•́︿•̀｡):
    1. 程序计数器为什么要记录当前线程要去执行的下一条 JVM 字节码指令的内存地址？
      1. 由于 CPU 需要频繁进行线程上下文切换，切换回来时必须知道从那里继续执行

2.2.3. 程序计数器空间大小

程序计数器的内存空间大小几乎可以忽略不计（因为只需要记录一个内存地址）：

  1. 在 32 bit 的 JVM 中，程序计数器占 4 byte
  2. 在 64 bit 的 JVM 中，程序计数器占 8 byte

2.3.2. 局部变量表

2.3.2.1. 局部变量表概述

局部变量表用于存储方法参数、局部变量

[!NOTE] 注意事项

  1. 局部变量表是线程私有的区域，每个线程都拥有独立的局部变量表
  2. 局部变量表能够存储基本数据类型的元素、引用数据类型的元素（需要注意的是：实际存储的是引用数据类型的元素的引用）、返回地址（即 Java5 的 returnAddress 类型，了解即可）
  3. 静态变量、静态常量、实例变量、实例常量在不同阶段会被赋默认值，然而方法参数、局部变量并不会在某阶段被赋默认值。因此我们在使用方法参数、局部变量之前，必须对其进行手动赋值，否则会导致编译错误（例如你在实例变量中 int i，不手动赋值就可以正常使用。但是你在局部变量中 int i，不手动赋值肯定不可以正常使用）
2.3.2.2. 局部变量表底层实现

局部变量表底层是基于一个一维数组实现的，该数组的基本存储单元是变量槽（Slot）：

  1. 在 32 bit 的 JVM 中，一个 Slot 占用 4 byte 的内存空间
  2. 在 64 bit 的 JVM 中，一个 Slot 占用 8 byte 的内存空间

[!NOTE] 注意事项

  1. 当线程调用某类的构造方法、某实例的实例方法时，指向被构造的实例、调用的实例的内存地址的引用（即 this）就被存储在局部变量表的数组索引为 0 的数组位置中
    1. 需要注意的是：
      1. 这是 Java 中的一个语法糖，也是为什么我们能在实例方法中使用 this 的原因
      2. 由于线程调用某类、某实例的静态方法时，根本不存在 this，也就导致我们不能在静态方法中使用 this
2.3.2.3. 局部变量表空间大小

局部变量表的内存空间大小在编译阶段就已经确定了，在运行时无法更改：

  1. 占用内存空间 32 bit 及以下的数据（例如 byte、short、int、float、char、boolean 数据类型的数据、引用数据类型的数据、返回地址）占用一个 Slot
  2. 占用内存空间 64 bit 的数据（例如 long、double 数据类型的数据）占用两个 Slot

[!NOTE] 注意事项

  1. 在 JVM 中，byte、short、char、boolean 数据类型的数据，实际上是被当作 int 数据类型的数据进行处理的（因为 int 数据类型的范围涵盖它们，而且 JVM 中并没有专门处理 byte、short、char、boolean 数据类型的数据的 JVM 字节码指令（需要注意的是：如果这些数据类型的数据在堆区中（例如是某实例的实例变量），该占用多少内存空间就占用多少内存空间））
  2. Slot 是可以复用的（例如当局部变量 A 的生命周期结束后，如果声明了新的局部变量 B，那么 B 可能会复用 A 的 Slot，从而达到节省内存空间的目的）以如下代码为例，局部变量 b 会复用局部变量 a 的 Slot，因此整个 main 方法的局部变量表最大只需要 2 个 Slot

2.4. 方法返回地址

当线程调用某方法时，JVM 会先把程序计数器中的内容（即当前线程要去执行的下一条 JVM 字节码指令的内存地址）记录在该方法对应的栈帧中的方法返回地址中，然后再开始执行该方法

如果线程执行方法的过程中未抛出异常，那么当线程执行到以下 JVM 字节码指令时，就会将执行的方法对应的栈帧中的方法返回地址中的内容取出来，并记录在程序计数器中（需要注意的是：这就是正常退出（未抛出异常））：
1. ireturn
1. 返回 byte、short、int、char、boolean 数据类型的数据
2. lreturn
1. 返回 long 数据类型的数据
3. freturn
1. 返回 float 数据类型的数据
4. dreturn
1. 返回 double 数据类型的数据
5. areturn
1. 返回引用数据类型的数据
6. return
1. 返回（需要注意的是：适用于无返回值的方法）

但是如果线程执行方法的过程中抛出了异常，那么线程就会查找方法区中该方法对应的类对应的类元数据区中的方法信息中的异常处理器表，这个表中记录了 From、To、Target、Type，线程会拿着行号从 From 和 To 之间检查，如果 Type 匹配，就跳到 Target 处执行（即 catch 块），如果未匹配，就将该栈帧弹出，将异常抛给调用者

[!NOTE] 注意事项

  1. 方法退出分为方法正常退出（发生异常、未发生异常）、方法异常退出（需要注意的是：方法正常退出（发生异常）是在 catch 块中捕获并处理了异常，并且没有抛出新的异常）
  2. 如果我们捕获并处理异常时，抛出一个新的异常，同样会再去查表，但是如果我们没有在 catch 块中嵌套 catch，很可能无法捕获并处理这个新的异常
  3. 在正常退出（发生异常）中，发生异常时，程序的执行流程会立即跳转到对应的 catch 块进行处理。异常被捕获并处理后，不会回到原先抛出异常的位置继续向下执行，而是直接退出

2.5. 一些附加信息

在 JVM 规范中，允许在栈帧中添加一些附加信息（例如调试信息、性能分析信息等），从而达到优化、调试等的目的

2.6. 本地方法栈

本地方法栈主要用于 native 方法

[!NOTE] 注意事项

  1. 在 HotSpot JVM 中，虚拟机栈、本地方法栈被合并为同一个栈
  2. 并非所有类型的 JVM 都支持 native 方法（因为在 JVM 规范中，并没有明确要求实现本地方法栈时使用的语言、实现的方式等。甚至如果不打算支持 native 方法，也可以无需实现本地方法栈）

2.7. 堆区

2.7.1. 堆区组成部分

2.7.2. 堆区概述

堆区是 JVM 管理内存空间的核心区域，也是整个 JVM 中内存空间最大的区域，几乎所有的实例都会在堆区分配内存空间（因为需要考虑逃逸分析）

[!NOTE] Title

  1. 堆区既会发生 GC，也会抛出 OutOfMemoryError 错误：
    1. OutOfMemoryError: Java heap space
      1. 堆区内存空间不足
  2. 堆区是线程共享的区域（但是需要考虑 TLAB），在单个 JVM 中全局唯一

2.7.3. 堆区空间大小

堆区在运行时具有初始的内存空间的大小（需要注意的是：由 -Xms512m 决定），并支持动态扩展内存空间，但是受到最大内存空间的限制（需要注意的是：由 -Xmx2048m 决定），一旦超出该限制，可能会抛出 OOM 错误（因为需要考虑 GC）

2.7.4. 实例在堆区分配内存空间相关流程

2.7.5. TLAB

2.7.5.1. TLAB 概述

想象这样一个问题：JVM 为了快速为实例在堆区分配内存空间，采用了 “指针碰撞” 技术（即使用一个指针指向 Eden 区已经使用的内存空间的边界，在为实例分配内存空间时，只需要检查剩余的内存空间是否足够，然后按需顺序偏移指针，即可完成实例在堆区分配内存空间）。

但是我们知道：堆区是线程共享的区域。那么在多线程并发场景下，就会出现多线程并发对堆区进行修改的情况。传统的解决方案是加锁，但这会成为一个显著的性能瓶颈

为了解决这一问题，JVM 提供了 TLAB。它将实例在线程共享的堆区分配内存空间，变成实例在线程私有的 TLAB 分配内存空间

2.7.5.2. TLAB 相关流程

[!NOTE] 注意事项

  1. 申请新 TLAB 时，是通过 CAS 保证多线程并发安全的（需要注意的是：旧 TLAB 的剩余空间会使用一个 int 类型的数组填充，这部分空间会被 GC）
  2. JVM 为了快速为实例在 TLAB 分配内存空间，采用了 “指针碰撞” 技术（因为 TLAB 是线程私有的，因此不存在多线程并发场景下的安全问题）

2.7.6. 逃逸分析

2.7.6.1. 逃逸分析概述

逃逸分析是指：JIT 在编译 Java 代码时，会分析一个实例在被创建后，是否被外部（即其他线程、调用对象的构造方法的方法之外的方法）引用。如果 JIT 发现一个实例并没有逃逸，就可以对这个实例进行深度的底层优化，从而极大提升性能。

[!NOTE] 注意事项

  1. 从工程实践的角度来看：能使用方法参数、局部变量，就使用方法参数、局部变量
2.7.6.2. 逃逸分析相关分类
  1. 方法逃逸
  2. 线程逃逸
2.7.6.3. 方法逃逸

方法逃逸是指：在某方法中调用某对象的构造方法，创建某实例，但是该实例作为该方法的返回结果返回给了其他方法，导致其他方法可能访问该实例

2.7.6.4. 线程逃逸

线程逃逸是指：某实例被赋值给静态变量、静态常量，导致其他线程可能访问该实例

2.7.6.5. 实例没有逃逸时，JIT 对其进行的优化
  1. 栈上分配
  2. 标量替换
  3. 锁消除
2.7.6.5.1. 栈上分配

我们知道 “几乎所有的实例都会在堆区分配内存空间”，但是堆区中的实例是需要被 GC 的

而栈上分配是指：如果 JIT 发现一个实例并没有逃逸，就可以将实例在堆区分配内存空间，变成实例在栈帧分配内存空间。这样实例就可以随栈帧被弹出而被自动销毁，完全不需要被 GC

[!NOTE] 注意事项

  1. HotSpot JVM 并没有实现真正意义上的栈上分配（因为 HotSpot JVM 主要是通过标量替换实现优化的）
2.7.6.5.2. 标量替换
2.7.6.5.3. 锁销除

锁销除是指：如果 JIT 发现一个实例并没有逃逸，就可以将该实例上加的 synchronized 锁去掉（因为根本不存在多线程并发场景下的安全问题）

2.8. 方法区

2.8.1. 方法区具体细节

[!NOTE] 注意事项

  1. 每一个实例都会持有一个指向其对应的类的类元数据区的引用（需要注意的是：持有的位置是对象头中的 Klass Word）
  2. 在 Java8 之前，静态遍历、静态常量保存在方法区。但是在 Java8 之后，静态变量、静态常量保存在堆区中（因为静态变量、静态常量保存在实例对应的类的类对象中，而类对象一定是在堆区分配内存空间的（因为 JIT 不会对类对象进行逃逸分析，因为无需也不应该对其进行逃逸分析，而且也无法对其进行逃逸分析），所以静态变量、静态常量保存在堆区）
