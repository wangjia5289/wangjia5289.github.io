四、JVM 相关流程

JIT 编译器

JIT 编译器概述

JIT 编译器用于把热点的 JVM 字节码指令编译成 CPU 对应的机器码指令，并对其进行非常激进的优化

7.1.4.3. 逃逸分析相关优化

当 JIT 编译器通过逃逸分析确定一个对象是 NoEscape 的，它就可以进行以下三种非常重要的性能优化

7.1.4.3.1. 栈上分配

在 Java 中，我们通常认为所有对象都是在堆 (Heap) 上分配的。堆是所有线程共享的，需要垃圾回收器 (GC) 来管理内存。

对于完全不逃逸的对象，JIT 编译器可以将其直接分配在当前线程的栈 (Stack) 上，而不是堆上。

栈上的内存在方法执行结束后，随着栈帧的出栈而被自动、立即地销毁，无需 GC 的介入。

7.1.4.3.2. 标量替换

“标量”是指无法再分解的原始数据类型（如 int, long, boolean 等）。相对地，“聚合量”就是可以被分解的对象（如 class）

因为即使对象能在栈上分配，也还是需要分配一整块内存来表示这个对，所以如果 JIT 发现一个对象不逃逸，并且这个对象在后续代码中可以被拆分，那么它就不会创建这个对象实体，而是将这个对象的所有成员变量（标量）打散，当作几个独立的本地变量来使用

所以连对象分配的开销都省了。这是比栈上分配更彻底的优化，而且打散后的标量更容易被优化，比如可以存放在高速的 CPU 寄存器中

7.1.4.3.3. 锁消除

synchronized 关键字会引入重量级的锁（监视器），涉及操作系统的互斥量，开销很大

JIT 编译器通过逃逸分析，发现某个加锁的对象不可能被其他线程访问（因为它根本没有逃逸出当前线程），那么这个锁就是完全多余的、无效的。此时，JIT 就会大胆地将这个 synchronized 锁操作给直接消除掉

这样避免了不必要的同步开销，提升了单线程的执行性能

4.1.2.1. 实例的访存定位概述

实例的访存定位是指：

4.1.3. 直接指针

其优点是：

  1. 访问速度快
    1. 与句柄访问相比，少了一次指针寻址的开销

其缺点是：

  1. 对象移动时成本高
    1. 一个大型应用中，一个对象里可能有成千上万个 reference 引用，一旦发生 GC，这个对象被移动了（例如从 Eden 区移动到 Survivor 区），就需要找到并更新这成千上万的 reference

4.1.4. 句柄访问

其优点是：

  1. 引用稳定
    1. 一个大型应用中，一个对象里可能有成千上万个 reference 引用，一旦发生 GC，这个对象被移动了（例如从 Eden 区移动到 Survivor 区），如果依旧采用直接指针的方式，就需要找到并更新这成千上万的 reference
    2. 但是如果采用句柄访问的方式，我们只需要修改句柄里的这一个引用即可

其缺点是：

  1. 访问速度慢
    1. 与直接指针相比，访问对象实例需要两次指针寻址（reference ➔ 句柄 ➔ 对象实例），比直接指针多了一层间接访问
  2. 额外的空间开销
    1. 与直接指针相比，句柄池本身需要占用一部分对空间

[!NOTE] 注意事项

  1. 猴哥的烦恼箱 (｡•́︿•̀｡)
    1. 为什么要通过句柄寻找对象类型，我们对象头中不是有 Klass Word 吗？
      1. 如果按这种方式处理，访问路径会变成 reference ➔ 句柄 ➔ 对象实例 ➔ 对象类型
      2. 但实际上我们并不关心对象实例，因此完全可以跳过对象实例，直接定位到对象类型

  3. JVM 执行 <init> 块中的 JVM 字节码指令

    4. 执行构造方法（术语是：构造初始化）

5. 方法的执行过程

5.1. 虚方法与非虚方法

5.1.1. 虚方法与非虚方法概述

我们知道，在运行时常量池中存在三种与方法调用相关的符号引用：

  1. CONSTANT_Mehotdref_info
  2. CONSTANT_InterfaceeMethodref_info
  3. CONSTANT_InvokeDynamic_info

假设 Animal 是一个接口，并且有多个实现类，例如 Dog、Cat。当我们调用 Animal.eat() 时，对应的 CONSTANT_InterfaceMethodref_info 到底会链接到 Dog.eat() 还是 Cat.eat() 呢？

同样，假设 Animal 是一个类，并且有多个子类，例如 Dog、Cat。当我们调用 Animal.eat() 时，对应的 CONSTANT_Methodref_info 又会链接到 Dog.eat() 还是 Cat.eat() 呢？

所以我们可以明白，这些符号引用并不一定都能在解析阶段顺利地转化为直接引用。这也是为什么 Java 要区分虚方法和非虚方法

5.1.2. 非虚方法

非虚方法是在编译阶段就能确定调用目标的，其绑定关系唯一且固定。这种绑定方式称为静态绑定（Static Binding），也叫早期绑定（Early Binding）。典型的非虚方法包括：

  1. 静态方法
  2. 构造方法
  3. private/final 的实例方法
  4. 调用父类的方法

我们知道，调用方法时会创建新的栈帧。对于非虚方法，它的执行过程相对简单：先正常建立栈帧，然后按字节码指令中的索引，通过栈帧中的动态链接找到运行时常量池，并定位到该索引对应的方法符号引用。如果该符号引用尚未解析，就会在此时完成解析，并直接定位到方法区中对应的方法描述信息的内存地址。

5.1.3. 虚方法

虚方法是在编译阶段无法确定调用目标的，必须等到运行时再根据对象的实际类型来决定。这种绑定方式称为动态绑定（Dynamic Binding），也叫晚期绑定（Late Binding）。它是 Java 多态的核心机制，除了前面提到的非虚方法之外，其余的方法都属于虚方法

对于虚方法，它的执行过程分为两个核心阶段：

  1. 方法分派
    1. 当调用 animal.eat() 时，JVM 首先会读取 animal 对象头中的 Klass Word
    2. 然后根据 Klass Word 定位到该对象在方法区中的类元数据
    3. 在类元数据中查找其 虚方法表（vtable），找到 eat 方法对应的入口
    4. 由此确定具体要调用的是哪个类的 eat 方法，并据此创建新的栈帧，栈帧中的动态链接指向当前方法所属类的运行时常量池
  2. 方法执行
    1. 正常执行字节码

因此，虚方法的每一次调用，都需要重新确定最终的目标地址

6.3. 程序计数器

6.3.1. 程序计数器概述

程序计数器（PC 寄存器）用于存储当前线程要去执行的下一条 JVM 字节码指令的内存地址

[!NOTE] 注意事项

  1. 程序寄存器是线程私有的内存，每个线程都拥有独立的程序寄存器
  2. JVM 既不会垃圾回收程序计数器，也不会因为程序计数器而抛出 OutOfMemoryError 错误、、StackOverflowError 错误
  3. 程序计数器实际记录的是 JVM 字节码指令的偏移量
  4. 执行引擎负责读取并执行 JVM 字节码指令。当执行引擎读取 JVM 字节码指令时，会将该 JVM 字节码指令的偏移量转换为其内存地址，然后再读取该 JVM 字节码指令。在执行引擎读取到 JVM 字节码指令后，程序计数器就会存储当前线程要去执行的下一条 JVM 字节码指令的内存地址，然后执行引擎才开始执行该 JVM 字节码指令（需要注意的是：这是一个非常重要的细节）
  5. 如果线程执行的是 native 方法，那么程序计数器存储的是 undefined（因为 native 方法本质上是离开了 JVM 的控制范围）

6.3.2. 程序计数器空间大小

我们几乎可以忽略不计程序计数器的内存空间（因为程序计数器只需要记录一个内存地址）：

  1. 在 32 bit 的 JVM 中，程序计数器占用 4 byte 的内存空间
  2. 在 64 bit 的 JVM 中，程序计数器占用 8 byte 的内存空间

6.4. 虚拟机栈

6.4.1. 虚拟机栈概述

[!NOTE] 注意事项

  1. 虚拟机栈是线程私有的内存空间，每个线程都拥有独立的虚拟机栈
  2. JVM 既会垃圾回收虚拟机栈，也会因为虚拟机栈而抛出 OutOfMemoryError 错误、StackOverflowError 错误：
    1. OutOfMemoryError 错误
      1. OutOfMemoryError: unable to create new native thread
        1. 如果虚拟机栈可以动态扩展内存空间，当 JVM 的内存空间已满，无法再分配虚拟机栈的内存空间时，就会抛出 OutOfMemoryError 错误（需要注意的是：HotSpot JVM 并不支持虚拟机栈动态扩展内存空间）
        2. 当创建本地线程，JVM 的内存空间已满，无法再分配该本地线程对应的程序计数器的内存空间、该本地线程对应的虚拟机栈的内存空间时，也会抛出 OutOfMemoryError 错误
    2. StackOverflowError 错误
      1. 当虚拟机栈的内存空间已满，无法再分配线程执行的方法对应的栈帧的内存空间时，JVM 就会抛出 StackOverflowError 错误

6.4.2. 虚拟机栈组成部分

6.4.2.1. 局部变量表
6.4.2.1.1. 局部变量表概述

局部变量表用于存储方法参数、局部变量

[!NOTE] 注意事项

  1. 局部变量表能够存储基本数据类型的元素、引用数据类型的元素（需要注意的是：局部变量表实际存储的是引用数据类型的元素的引用）、返回地址（即在 Java5 之前的 returnAddress 类型，我们知道有这个东西就行）
  2. 静态变量、静态常量、实例变量、实例常量在不同阶段会被赋予默认值。但是方法参数、局部变量并不会被赋予默认值。因此我们在使用方法参数、局部变量之前，必须对其进行手动赋值，否则会导致类的编译错误
6.4.2.1.2. 局部变量表底层实现

局部变量表的底层是：一个一维数组

局部变量表使用的一维数组的基本存储单元是变量槽（Slot）

  1. 在 32 bit 的 JVM 中，一个 Slot 占用 4 byte 的内存空间
  2. 在 64 bit 的 JVM 中，一个 Slot 占用 8 byte 的内存空间

[!NOTE] 注意事项

  1. 当线程调用某类的构造方法时，指向被构造的实例的内存地址的引用（即 this），就被存储在局部变量表的数组索引为 0 的数组位置中（需要注意的是：这是 Java 中的一个语法糖，也是我们为什么能在构造方法中使用 this 的原因）
  2. 当线程调用某实例的实例方法时，指向该方法所属实例的内存地址的引用（即 this），就被存储在局部变量表的数组索引为 0 的数组位置中（需要注意的是：这是 Java 中的一个语法糖，也是为什么我们能在实例方法中使用 this 的原因）
  3. 当线程调用某类的静态方法时，根本就不存在 this，导致我们不能在静态方法中使用 this
6.4.2.1.3. 局部变量表空间大小

在类的编译过程中，JVM 中的解释器就已经确定了局部变量表的内存空间，在运行 Java 程序的过程中无法更改局部变量表的内存空间：

  1. 占用 32 bit 及以下的内存空间的数据（例如 byte、short、int、float、char、boolean 类型的数据、引用数据类型的数据、返回地址）占用一个 Slot
  2. 占用 64 bit 的内存空间的数据（例如 long、double 类型的数据）占用两个 Slot

而在 32 bit 的 JVM 中，一个 Slot 占用 4 byte 的内存空间。在 64 bit 的 JVM 中，一个 Slot 占用 8 byte 的内存空间。所以我们只需知道方法中的方法参数、局部变量占用几个 Slot，就可以知道该方法对应的栈帧中的局部变量表的内存空间

[!NOTE] 注意事项

  1. 在 JVM 中，byte 类型的数据、short 类型的数据、char 类型的数据、boolean 类型的数据被 JVM 当作 int 类型的数据（因为 int 类型数据的范围涵盖它们，而且在 JVM 中其实并没有专门处理它们的 JVM 字节码指令（需要注意的是：如果它们被存储在堆区（例如是实例的实例变量），那么它们该占用多少内存空间，就占用多少内存空间））
  2. Slot 是可以复用的（例如：在局部变量 A 的生命周期结束后，如果我们又声明了新的局部变量 B，那么 B 可能复用 A 的 Slot，从而达到节省内存空间的目的）。以如下代码为例，局部变量 b 会复用局部变量 a 的 Slot。因此整个 main 方法只需要占用两个 Slot，所以在 32 bit 的 JVM 中，main 方法对应的栈帧中的局部变量表占用 8 byte 的内存空间。在 64 bit 的 JVM 中，main 方法对应的栈帧中的局部变量表占用 16 byte 的内存空间
6.4.2.2. 方法返回地址

当线程调用方法时，会先把程序计数器中的内容（当前线程要去执行的下一条 JVM 字节码指令的内存地址）存储在被调用的方法对应的方法返回地址中，然后再开始执行该方法

如果线程在执行方法的过程中未抛出异常，那么当线程执行到以下 JVM 字节码指令时，就会取出该方法对应的栈帧中的方法返回地址中的内容，并存储在程序计数器中（需要注意的是：这就是方法正常退出（未抛出异常））：
1. ireturn
1. 返回 byte 类型的数据、short 类型的数据、int 类型的数据、char 类型的数据、boolean 类型的数据
2. lreturn
1. 返回 long 类型的数据
3. freturn
1. 返回 float 类型的数据
4. dreturn
1. 返回 double 类型的数据
5. areturn
1. 返回引用数据类型的数据
6. return
1. 仅返回（需要注意的是：仅返回适用于无返回值的方法）

但是如果线程在执行方法的过程中抛出了异常，那么线程就会去查该方法对应的异常处理器表（需要注意的是：异常处理器表被存储在方法区）：

  1. 如果异常处理器表中有匹配的异常
    1. 线程就跳转到 Targer 处，然后执行 Target 处的 Java 代码（即线程执行 catch 块。需要注意的是：不会再回到抛出异常的位置，继续执行原来的代码）
  2. 如果异常处理器表中没有匹配的异常
    1. 线程就会弹出该方法对应的栈帧，并继续向外抛出异常（即抛出异常给调用方）

[!NOTE] 注意事项

  1. 方法退出分为方法正常退出（发生异常、未发生异常）、方法异常退出（需要注意的是：方法正常退出（发生异常）是在 catch 块中，捕获并处理了异常，并且没有抛出新的异常）
  2. 当我们捕获并处理异常时，抛出了一个新的异常，那么线程同样会再去查异常处理器表。但是如果我们没有使用嵌套 catch 块，就很可能无法捕获并处理这个新的异常

6.4.2.4. 操作数栈

操作数栈是 JVM 执行引擎的一个工作区，用于保存计算过程中的中间结果，同时作为变量的临时存储空间。JVM 会根据方法中的字节码指令，向操作数栈中写入数据，或从中提取数据并使用后，结果再写入操作数栈

与局部变量表一样，操作数栈的最大深度在编译阶段就已经确定，运行时无法更改（栈顶缓存并不会影响最大深度）

  1. 32 位及以内的数据类型（reference、returnAddress、byte、short、char、int、float、boolean）占用一个栈单位深度
  2. 64 位的数据类型（long、double）占用两个栈单位深度

[!NOTE] 注意事项

  1. byte、short、char 在压入操作栈前，会被自动转换为 int 类型，对应关系主要包括：
    1. byte ➔ int
    2. short ➔ int
    3. char ➔ int
    4. int ➔ int
    5. boolean ➔ boolean
    6. long ➔ long
    7. float ➔ float
    8. double ➔ double
  2. 我们常说 Java 虚拟机的解释引擎是基于栈的执行引擎，这里的 “栈” 指的就是操作数栈

6.5. 堆区

6.5.1. 堆区概述

堆区是 JVM 管理内存空间的核心区域，也是整个 JVM 中内存空间最大的区域，几乎所有的实例都会在堆区被 JVM 分配内存空间（因为需要考虑逃逸分析）

[!NOTE] 注意事项

  1. 堆区是线程共享的内存空间（但是需要考虑 TLAB），在单个 JVM 中全局唯一
  2. JVM 既会对堆区进行 GC，也会因为堆区而抛出 OutOfMemoryError 错误：
    1. OutOfMemoryError: Java heap space
      1. 堆区内存空间不足

6.5.2. 堆区空间大小

堆区在运行时具有初始的内存空间（需要注意的是：堆区的初始的内存空间由 -Xms512m 决定），并且 JVM 支持动态扩展、动态缩减堆区的内存空间，但是受到堆区的最大的内存空间的限制（需要注意的是：堆区的最大的内存空间由 -Xmx2048m 决定）。一旦堆区的内存空间超出堆区的最大的内存空间的限制，JVM 就可能会抛出 OutOfMemoryError 错误（因为需要考虑 GC）

6.5.3. 堆区组成部分

6.5.4. 实例被 JVM 在堆区分配内存空间

  1. 在 JVM 执行完 GC 之后，JVM 会根据堆区的内存空间的空闲比例，动态扩展、动态缩减堆区的内存空间

6.5.5. TLAB

6.5.5.1. TLAB 概述

想象这样一个问题：JVM 为了快速为实例在堆区分配内存空间，采用了 “指针碰撞” 技术（即使用一个指针指向 Eden 区已经使用的内存空间的边界，当 JVM 为实例分配内存空间时，只需要检查剩余的内存空间是否足够，然后按需顺序偏移指针，即可完成为实例在堆区分配内存空间）

但是我们知道 “堆区是线程共享的内存空间”，在多线程并发场景下，就会出现多线程并发对堆区进行修改的情况。传统的解决方案是加锁，但这会成为一个显著的性能瓶颈

为了解决这一问题，JVM 提供了 TLAB。TLAB 将实例在线程共享的堆区被 JVM 分配内存空间，变成了实例在线程私有的 TLAB 被 JVM 分配内存空间

6.5.5.2. TLAB 相关流程

[!NOTE] 注意事项

  1. 当线程申请新 TLAB 时，JVM 是通过 CAS 保证多线程并发安全的（需要注意的是：旧 TLAB 的剩余空间会使用一个 int 类型的数组填充，这部分空间会被 GC）
  2. JVM 为了快速为实例在 TLAB 分配内存空间，采用了 “指针碰撞” 技术（因为 TLAB 是线程私有的，因此不存在多线程并发场景下的安全问题）

6.5.6. 逃逸分析

6.5.6.1. 逃逸分析概述

逃逸分析是指：JIT 在编译 Java 代码时，会分析一个实例在被创建后，是否被外部（即其他线程、调用对象的构造方法的方法之外的方法）引用。如果 JIT 发现一个实例并没有逃逸，就可以对这个实例进行深度的底层优化，从而极大提升性能。

[!NOTE] 注意事项

  1. 从工程实践的角度来看：能使用方法参数、局部变量，就使用方法参数、局部变量
6.5.6.2. 逃逸分析相关分类
  1. 方法逃逸
  2. 线程逃逸
6.5.6.3. 方法逃逸

方法逃逸是指：在某方法中调用某对象的构造方法，创建某实例，但是该实例作为该方法的返回结果返回给了其他方法，导致其他方法可能访问该实例

6.5.6.4. 线程逃逸

线程逃逸是指：某实例被赋值给静态变量、静态常量，导致其他线程可能访问该实例

6.5.6.5. 实例没有逃逸时，JIT 对其进行的优化
  1. 栈上分配
  2. 标量替换
  3. 锁消除
6.5.6.5.1. 栈上分配

我们知道 “几乎所有的实例都会在堆区分配内存空间”，但是堆区中的实例是需要被 GC 的

而栈上分配是指：如果 JIT 发现一个实例并没有逃逸，就可以将实例在堆区分配内存空间，变成实例在栈帧分配内存空间。这样实例就可以随栈帧被弹出而被自动销毁，完全不需要被 GC

[!NOTE] 注意事项

  1. HotSpot JVM 并没有实现真正意义上的栈上分配（因为 HotSpot JVM 主要是通过标量替换实现优化的）
6.5.6.5.2. 标量替换
6.5.6.5.3. 锁销除

锁销除是指：如果 JIT 发现一个实例并没有逃逸，就可以将该实例上加的 synchronized 锁去掉（因为根本不存在多线程并发场景下的安全问题）

6.6. 方法区

6.6.2. 方法区概述

[!NOTE] 注意事项

  1. 方法区是线程共享的内存空间，在单个 JVM 中全局唯一
  2. JVM 既会垃圾回收方法区，，也会因为方法区而抛出 OutOfMemoryError 错误：
    1. OutOfMemoryError: Metaspace
      1. 元空间的内存空间不足
    2. OutOfMemoryError: PermGen space
      1. 永久代的内存空间不足
  1. 每一个实例都会存储一个指向其对应的类的类元数据区的内存地址（需要注意的是：存储的位置是对象头中的 Klass Word）
  2. 在 Java8 之前，静态变量、静态常量存储在方法区。但是在 Java8 及之后，静态变量、静态常量存储在堆区（因为静态变量、静态常量存储在类对象中，而类对象一定是在堆区被 JVM 分配内存空间的（因为 JIT 不会对类对象进行逃逸分析，因为无需也不应该对类对象进行逃逸分析，而且也无法对类对象进行逃逸分析），所以静态变量、静态常量存储在堆区）
  3. 在 Java8 之前的永久代，在 Java8 及以后的元空间，都是对 JVM 规范中的方法区的实现。永久代与元空间的区别在于：
    1. 永久代的内存空间是堆区的内存空间的部分，内存空间往往较小（例如几十 MB），很容易导致 JVM 抛出 OutOfMemoryError 错误
    2. 而元空间的内存空间是本地内存空间的部分，内存空间甚至可以动态扩展至整个物理内存空间的上限

7.1.1.1. 栈顶缓存技术

栈顶缓存，C++ 解释器做不到栈顶缓存，但是模板解释器可以做到

JVM 采用的是基于栈的架构，即运算主要依赖 操作数栈（Operand Stack） 而非寄存器。这种设计带来了良好的跨平台性，但也使得大多数计算操作（如加法、乘法）都必须先将操作数压入操作数栈，再由字节码指令将其弹出、计算并将结果压回等一系列繁琐的流程

例如一次简单的加法可能需要经历 load → push → push → iadd → store 的多个步骤，而在寄存器架构中可能只需一条指令即可完成。

另外，每次入栈和出栈操作，本质上都是对 JVM 内存结构的访问，而内存访问的速度远不及 CPU 寄存器，更不用说 JVM 内存了

为了解决这一问题，HotSpot JVM 的设计者提出了栈顶缓存（ToS，Top-of-Stack Caching）技术，将操作数栈顶部的若干个元素缓存在 CPU 寄存器中，以减少对操作数栈内存的频繁访问，从而提升执行引擎的整体执行效率。

在解释执行阶段，字节码逐条执行，大部分值都直接读写内存中的操作数栈，寄存器的使用非常有限（仅用于短暂调度）。然而，当某个方法成为热点，触发 JIT 编译后，JIT 编译器会在生成本地机器码的过程中启用栈顶缓存技术，将栈顶的值尽可能缓存在寄存器中。

然而，由于硬件本身的限制，寄存器数量十分有限。以常见的 x86 架构为例，通用寄存器仅有 8 个（eax、ebx、ecx、edx、esi、edi、ebp、esp），即便是现代的 x86-64 架构也只有 16 个。这些寄存器还需要被用于局部变量、参数传递等其他任务，因此必须通过非常严格的寄存器分配与抢占策略进行管理。这也正是栈顶缓存技术只被应用于热点方法，而非所有方法的原因。并且寄存器能够缓存的数据非常有限，但通常只缓存 2-4 个槽位就已能显著提升性能。

需要注意的是，只有在必要的场景下，寄存器中的数据才会被同步回栈内存（即先存入 CPU 寄存器，必要时才写回内存中的操作数栈）：

  1. 调用方法前
    1. 如果要调用某个方法，而该方法需要从栈顶获取参数，就必须先将寄存器中的值写回栈内存，确保方法调用机制能按照字节码协议正确读取参数
  2. 操作数栈溢出时
    1. 当需要缓存的栈顶数据超过寄存器容量时，旧数据必须写回栈中，以腾出寄存器空间
  3. 数据生命周期结束前的检查
    1. 如果某个值在之后不会再使用，为了保持字节码语义的正确性（如异常处理或 GC 根节点扫描），可能会提前将其写回栈中
  4. 调试 / 异常机制介入时
    1. 当调试器需要访问栈上值，或在发生异常抛出、栈回溯等情况下，JVM 需还原执行现场，也必须将寄存器中的值同步回操作数栈
