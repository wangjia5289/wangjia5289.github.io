2.2. 本地方法栈

本地方法栈主要用于 native 方法

[!NOTE] 注意事项

  1. 在 HotSpot JVM 中，虚拟机栈、本地方法栈被合并为同一个栈
  2. 并非所有类型的 JVM 都支持 native 方法（因为在 JVM 规范中，如果不打算让 JVM 支持 native 方法，那么该 JVM 就可以不实现本地方法栈）

2.3. 程序计数器

2.3.1. 程序计数器概述

程序计数器（PC 寄存器）是用于存储当前线程要去执行的下一条 JVM 字节码指令的内存地址（需要注意的是：实际记录的是这条 JVM 字节码指令的偏移量）

[!NOTE] 注意事项

  1. 程序寄存器是线程私有的内存，每个线程都拥有独立的程序寄存器
  2. JVM 既不会对程序计数器进行 GC，也不会因为程序计数器而抛出 OutOfMemoryError 错误、、StackOverflowError 错误
  3. 执行引擎负责读取并执行 JVM 字节码指令。当执行引擎读取 JVM 字节码指令时，会将该 JVM 字节码指令的偏移量转换为其内存地址，然后再读取该 JVM 字节码指令。在执行引擎读取到 JVM 字节码指令后，程序计数器就会存储当前线程要去执行的下一条 JVM 字节码指令的内存地址，然后执行引擎才开始执行该 JVM 字节码指令（需要注意的是：这是一个非常重要的细节）
  4. 如果线程执行的是 native 方法，那么程序计数器存储的是 undefined（因为 native 方法本质上是离开了 JVM 的控制）

2.3.2. 程序计数器空间大小

程序计数器的内存空间几乎可以忽略不计（因为只需要记录一个内存地址）：

  1. 在 32 bit 的 JVM 中，程序计数器占 4 byte
  2. 在 64 bit 的 JVM 中，程序计数器占 8 byte

2.3.3. 程序计数器组成部分

2.4. 虚拟机栈

2.4.1. 虚拟机栈概述

[!NOTE] 注意事项

  1. 虚拟机栈是线程私有的内存空间，每个线程都拥有独立的虚拟机栈
  2. JVM 既会对虚拟机栈进行 GC，也会因为虚拟机栈而抛出 OutOfMemoryError 错误、StackOverflowError 错误：
    1. OutOfMemoryError 错误
      1. OutOfMemoryError: unable to create new native thread
        1. 如果虚拟机栈可以动态扩展内存空间，当 JVM 的内存空间已满，无法再为虚拟机栈分配内存空间时，就会抛出 OutOfMemoryError 错误（需要注意的是：HotSpot JVM 并不支持虚拟机栈动态扩展内存空间）
        2. 当创建本地线程，JVM 的内存空间已满，无法再为该本地线程对应的程序计数器、虚拟机栈分配内存空间时，也会抛出 OutOfMemoryError 错误
    2. StackOverflowError 错误
      1. 当虚拟机栈的内存空间已满，无法再为线程执行的方法对应的栈帧分配内存空间时，就会抛出 StackOverflowError 错误

2.4.2. 虚拟机栈组成部分

2.4.3. 局部变量表

2.4.3.1. 局部变量表概述

局部变量表用于存储方法参数、局部变量

[!NOTE] 注意事项

  1. 局部变量表能够存储基本数据类型的元素、引用数据类型的元素（需要注意的是：实际存储的是引用数据类型的元素的引用）、返回地址（即 Java5 之前的 returnAddress 类型，了解即可）
  2. 静态变量、静态常量、实例变量、实例常量在不同阶段会被赋默认值，然而方法参数、局部变量并不会在某阶段被赋默认值。因此我们在使用方法参数、局部变量之前，必须对其进行手动赋值，否则会导致类的编译错误（例如你在实例变量中 int i，不手动赋值就可以正常使用。但是你在局部变量中 int i，不手动赋值肯定不可以正常使用）
2.4.3.2. 局部变量表底层实现

局部变量表底层是基于一个一维数组实现的，该数组的基本存储单元是变量槽（Slot）：

  1. 在 32 bit 的 JVM 中，一个 Slot 占用 4 byte 的内存空间
  2. 在 64 bit 的 JVM 中，一个 Slot 占用 8 byte 的内存空间

[!NOTE] 注意事项

  1. 当线程调用某类的构造方法时，指向被构造的实例的内存地址的引用（即 this）就被存储在局部变量表的数组索引为 0 的数组位置中（需要注意的是：这是 Java 中的一个语法糖，也是我们为什么能在构造方法中使用 this 的原因）
  2. 当线程调用某实例的实例方法时，指向该方法所属实例的内存地址的引用（即 this）就被存储在局部变量表的数组索引为 0 的数组位置中
    1. 需要注意的是：
      1. 这是 Java 中的一个语法糖，也是为什么我们能在实例方法中使用 this 的原因
      2. 由于线程调用某类、某实例的静态方法时，根本不存在 this，也就导致我们不能在静态方法中使用 this
2.4.3.3. 局部变量表空间大小

局部变量表的内存空间在类的编译阶段就已经确定了，在运行时无法更改：

  1. 占用 32 bit 及以下的内存空间的数据（例如 byte、short、int、float、char、boolean 类型的数据、引用数据类型的数据、返回地址）占用一个 Slot
  2. 占用 64 bit 的内存空间的数据（例如 long、double 类型的数据）占用两个 Slot

而在 32 bit 的 JVM 中，一个 Slot 占用 4 byte 的内存空间。在 64 bit 的 JVM 中，一个 Slot 占用 8 byte 的内存空间。所以我们只需要判断一个方法中的方法参数、局部变量占用几个 Slot，就可以判断出该方法对应的栈帧中的局部变量表的内存空间

[!NOTE] 注意事项

  1. 在 JVM 中，byte、short、char、boolean 类型的数据，实际上是被当作 int 类型的数据进行处理的（因为 int 类型的范围涵盖它们，而且 JVM 中并没有专门处理 byte、short、char、boolean 类型的数据的 JVM 字节码指令（需要注意的是：如果这些类型的数据存储在堆区（例如是某实例的实例变量），那么这些数据该占用多少内存空间就占用多少内存空间））
  2. Slot 是可以复用的（例如当局部变量 A 的生命周期结束后，如果声明了新的局部变量 B，那么 B 可能复用 A 的 Slot，从而达到节省内存空间的目的）。以如下代码为例，局部变量 b 会复用局部变量 a 的 Slot，因此整个 main 方法只需要占用两个 Slot，所以在 32 bit 的 JVM 中，main 方法对应的栈帧中的局部变量表占用 8 byte 的内存空间。在 64 bit 的 JVM 中，main 方法对应的栈帧中的局部变量表占用 16 byte 的内存空间

2.4.4. 方法返回地址

当线程调用某方法时，JVM 会先把程序计数器中的内容（即当前线程要去执行的下一条 JVM 字节码指令的内存地址）记录在被调用的方法对应的方法返回地址中，然后再开始执行被调用的方法

如果线程在执行方法的过程中未抛出异常，那么当线程执行到以下 JVM 字节码指令时，就会将被执行的方法对应的方法返回地址中的内容取出来，并记录在程序计数器中（需要注意的是：这就是正常退出（未抛出异常））：
1. ireturn
1. 返回 byte、short、int、char、boolean 类型的数据
2. lreturn
1. 返回 long 类型的数据
3. freturn
1. 返回 float 类型的数据
4. dreturn
1. 返回 double 类型的数据
5. areturn
1. 返回引用数据类型的数据
6. return
1. 返回（需要注意的是：适用于无返回值的方法）

但是如果线程在执行方法的过程中抛出了异常，那么线程就会去查被执行的方法对应的异常处理器表（需要注意的是：异常处理器表位于方法区）：

  1. 如果异常处理器表中有匹配的异常，线程就跳转到 Targer 处继续执行（即线程执行 catch 块。需要注意的是：不会回到抛出异常的位置继续执行）
  2. 如果异常处理器表中没有匹配的异常，线程就会将被执行的方法对应的栈帧弹出，并继续向外抛出异常（即抛出异常给调用方）

[!NOTE] 注意事项

  1. 方法退出分为方法正常退出（发生异常、未发生异常）、方法异常退出（需要注意的是：方法正常退出（发生异常）是在 catch 块中捕获并处理了异常，并且没有抛出新的异常）
  2. 如果我们捕获并处理异常时，抛出了一个新的异常，那么线程同样会再去查异常处理器表，但是如果我们没有在 catch 块中嵌套 catch，很可能无法捕获并处理这个新的异常

2.5. 堆区

2.5.1. 堆区组成部分

2.5.2. 堆区概述

堆区是 JVM 管理内存空间的核心区域，也是整个 JVM 中内存空间最大的区域，几乎所有的实例都会在堆区被 JVM 分配内存空间（因为需要考虑逃逸分析）

[!NOTE] 注意事项

  1. 堆区是线程共享的内存空间（但是需要考虑 TLAB），在单个 JVM 中全局唯一
  2. JVM 既会对堆区进行 GC，也会因为堆区而抛出 OutOfMemoryError 错误：
    1. OutOfMemoryError: Java heap space
      1. 堆区内存空间不足

2.5.3. 堆区空间大小

堆区在运行时具有初始的内存空间（需要注意的是：堆区的初始的内存空间由 -Xms512m 决定），并且 JVM 支持动态扩展、动态缩减堆区的内存空间，但是受到堆区的最大的内存空间的限制（需要注意的是：堆区的最大的内存空间由 -Xmx2048m 决定）。一旦堆区的内存空间超出堆区的最大的内存空间的限制，JVM 就可能会抛出 OutOfMemoryError 错误（因为需要考虑 GC）

2.5.4. 实例被 JVM 在堆区分配内存空间 🌈

[!NOTE] 注意事项

  1. 在 JVM 执行完 GC 之后，JVM 会根据堆区的内存空间的空闲比例，动态扩展、动态缩减堆区的内存空间

2.5.5. TLAB

2.5.5.1. TLAB 概述

想象这样一个问题：JVM 为了快速为实例在堆区分配内存空间，采用了 “指针碰撞” 技术（即使用一个指针指向 Eden 区已经使用的内存空间的边界，当 JVM 为实例分配内存空间时，只需要检查剩余的内存空间是否足够，然后按需顺序偏移指针，即可完成为实例在堆区分配内存空间）

但是我们知道 “堆区是线程共享的内存空间”，在多线程并发场景下，就会出现多线程并发对堆区进行修改的情况。传统的解决方案是加锁，但这会成为一个显著的性能瓶颈

为了解决这一问题，JVM 提供了 TLAB。TLAB 将实例在线程共享的堆区被 JVM 分配内存空间，变成了实例在线程私有的 TLAB 被 JVM 分配内存空间

2.5.5.2. TLAB 相关流程

[!NOTE] 注意事项

  1. 当线程申请新 TLAB 时，JVM 是通过 CAS 保证多线程并发安全的（需要注意的是：旧 TLAB 的剩余空间会使用一个 int 类型的数组填充，这部分空间会被 GC）
  2. JVM 为了快速为实例在 TLAB 分配内存空间，采用了 “指针碰撞” 技术（因为 TLAB 是线程私有的，因此不存在多线程并发场景下的安全问题）

2.5.6. 逃逸分析

2.5.6.1. 逃逸分析概述

逃逸分析是指：JIT 在编译 Java 代码时，会分析一个实例在被创建后，是否被外部（即其他线程、调用对象的构造方法的方法之外的方法）引用。如果 JIT 发现一个实例并没有逃逸，就可以对这个实例进行深度的底层优化，从而极大提升性能。

[!NOTE] 注意事项

  1. 从工程实践的角度来看：能使用方法参数、局部变量，就使用方法参数、局部变量
2.5.6.2. 逃逸分析相关分类
  1. 方法逃逸
  2. 线程逃逸
2.5.6.3. 方法逃逸

方法逃逸是指：在某方法中调用某对象的构造方法，创建某实例，但是该实例作为该方法的返回结果返回给了其他方法，导致其他方法可能访问该实例

2.5.6.4. 线程逃逸

线程逃逸是指：某实例被赋值给静态变量、静态常量，导致其他线程可能访问该实例

2.5.6.5. 实例没有逃逸时，JIT 对其进行的优化
  1. 栈上分配
  2. 标量替换
  3. 锁消除
2.5.6.5.1. 栈上分配

我们知道 “几乎所有的实例都会在堆区分配内存空间”，但是堆区中的实例是需要被 GC 的

而栈上分配是指：如果 JIT 发现一个实例并没有逃逸，就可以将实例在堆区分配内存空间，变成实例在栈帧分配内存空间。这样实例就可以随栈帧被弹出而被自动销毁，完全不需要被 GC

[!NOTE] 注意事项

  1. HotSpot JVM 并没有实现真正意义上的栈上分配（因为 HotSpot JVM 主要是通过标量替换实现优化的）
2.5.6.5.2. 标量替换
2.5.6.5.3. 锁销除

锁销除是指：如果 JIT 发现一个实例并没有逃逸，就可以将该实例上加的 synchronized 锁去掉（因为根本不存在多线程并发场景下的安全问题）

2.6. 方法区

2.6.1. 方法区组成部分

2.6.2. 方法区概述

[!NOTE] 注意事项

  1. 方法区是线程共享的内存空间，在单个 JVM 中全局唯一
  2. JVM 既会对方法区进行 GC，也会因为方法区而抛出 OutOfMemoryError 错误：
    1. OutOfMemoryError: Metaspace
      1. 元空间内存不足
    2. OutOfMemoryError: PermGen space
      1. 永久代内存不足
  3. 每一个实例都会存储一个指向其对应的类的类元数据区的内存地址（需要注意的是：存储的位置是对象头中的 Klass Word）
  4. 在 Java8 之前，静态变量、静态常量存储在方法区。但是在 Java8 及之后，静态变量、静态常量存储在堆区（因为静态变量、静态常量存储在类对象中，而类对象一定是在堆区被 JVM 分配内存空间的（因为 JIT 不会对类对象进行逃逸分析，因为无需也不应该对类对象进行逃逸分析，而且也无法对类对象进行逃逸分析），所以静态变量、静态常量存储在堆区）
  5. 在 Java8 之前的永久代，在 Java8 及以后的元空间，都是对 JVM 规范中的方法区的实现。永久代与元空间的区别在于：
    1. 永久代的内存空间是堆区的内存空间的部分，内存空间往往较小（例如几十 MB），很容易导致 JVM 抛出 OutOfMemoryError 错误
    2. 而元空间的内存空间是本地内存空间的部分，内存空间甚至可以动态扩展至整个物理内存空间的上限

2.6.3. 类型信息

类型信息主要包括：

  1. 类的全限定名
  2. 类继承的类（即父类）的全限定名
  3. 类实现的接口的全限定名列表（因为一个类可以实现多个接口）
  4. Java 修饰符
    1. 访问修饰符
      1. public
      2. protected
      3. package-private
      4. private
    2. 非访问修饰符
      1. 类级别的非访问修饰符
        1. final
        2. static
        3. strictfp
        4. abstract
        5. sealed
        6. non-sealed
  5. 类在类的加载阶段涉及到的加载器

static：需修正。在 Java 中，顶级类（Top-level Class）不能使用 static 修饰。只有内部类（Nested Class）可以使用 static。

类元信息是所有引用类型的元数据，主要包括：

  1. 类的全限定名
    1. 例如 java/lang/String
    2. 注意事项
      1. 当我们日常编写时，类的全限定名是用点号，例如 java.lang.String
      2. 但是在 JVM 内部、字节码文件（.class 文件）以及 JNI 描述符中，类的全限定名是用斜杠，例如 java/lang/String
  2. 父类的全限定名
    1. 注意事项
      1. 是直接父类的全限定类名
      2. 除了 java.lang.Object，所有类都应该有父类
  3. 实现接口的全限定名列表
    1. 因为一个类可能实现多个接口，所以用列表进行记录
  4. 类的修饰符
    1. 访问修饰符
      1. public
      2. protected
      3. package-private（默认）
      4. private
    2. 非访问修饰符
      1. final
      2. static
      3. abstract
    3. 注意事项
      1. 这些信息在 .class 文件中都包含在 access_flags 字段
      2. 除此之外，还会记录类型本身的种类标识符，用于 JVM 判断该类到底是普通类、接口、枚举类，还是注解类型
        1. class
        2. interface
        3. enum
        4. @interface
      3. 虽然我们通常认为 enum 是一种特殊的 class，@interface 是一种特殊的 interface，但本质上它们都是处于同一级别的关键字，并在字节码层面通过不同的访问标志加以区分
  5. 类加载器引用
    1. 加载该类的类加载器的引用

[!NOTE] 注意事项

  1. 无论是类、接口还是枚举、注解，在 JVM 的底层实现和运行层面，它们最终都被表示为 Class，因此都会在类的元数据信息中占有一席之地。

3. 垃圾回收器

3.1. 垃圾回收器前置基础

3.1.1. 垃圾回收相关策略 🌈

  1. 串行垃圾回收策略
    1. 在同一时间段内，垃圾回收器仅允许一个线程执行垃圾回收。在线程执行垃圾回收的过程中，垃圾回收器会 “暂停” 所有执行 Java 代码的线程，直到该线程执行完垃圾回收（术语是：STW）
  2. 并行垃圾回收策略
    1. 在同一时间段内，垃圾回收器使用多个线程同时执行垃圾回收。在线程执行垃圾回收的过程中，垃圾回收器会执行 STW
  3. 并发垃圾回收策略
    1. 在线程执行垃圾回收的过程中，垃圾回收器允许同时运行其他的执行 Java 代码的线程、执行垃圾回收的线程
    2. 需要注意的是：
      1. 如果 CPU 的核数不足，执行垃圾回收的线程、其他的执行 Java 代码的线程可能会以交替的方式运行，这种情况才符合传统意义上的并发
      2. 即便并发垃圾回收器采用了并发垃圾回收策略，也并不代表 Java 程序不会垃圾回收器执行 STW 的情况，只不过这个情况没有被体现在并发垃圾回收策略中

[!NOTE] 注意事项

  1. 迄今为止，并不存在任何一种垃圾回收器能够实现 Java 程序不会出现垃圾回收器执行 STW 的情况

垃圾标记算法概述

我们知道 “几乎所有的实例都会被 JVM 分配堆区的内存空间”，而 JVM 中的垃圾回收器在清理堆区之前，必须区分该堆区中哪些实例是 “存活的”，哪些实例是 “死亡” 的

垃圾标记算法相关分类

  2. 可达性分析算法

垃圾清除相关算法

  1. 标记-清除算法

  3. 标记-整理算法

3.1.4.1.1. 标记-清除算法概述

标记-清除算法是垃圾清除算法中最基本、最古老的一种，也是许多垃圾清除算法的基础

标记-清除算法相关流程

3.1.2. 垃圾回收相关事件 🌈

3.1.2.1. 新生代回收 🌈
3.1.2.1.1. 新生代回收概述 🌈

新生代回收（Minor GC、Young GC）是指：垃圾回收器垃圾回收堆区中的新生代的内存空间

3.1.2.1.2. 新生代回收触发条件 🌈
  1. 当堆区中的 Eden 区的剩余的内存空间不足，JVM 不能直接为新的实例分配堆区中的 Eden 区的内存空间时，就会触发垃圾回收器执行 Minor GC
3.1.2.2. 老年代回收 🌈

老年代回收（Major GC）是指：垃圾回收器垃圾回收堆区中的老年代的内存空间

[!NOTE] 注意事项

  1. Major GC 这个词很容易引起混淆。因为在不同的垃圾回收器中，Major GC 有不同的含义：
    1. 在 CMS GC 中，Major GC 是指：垃圾回收器垃圾回收堆区中的老年代的内存空间
    2. 在其他的垃圾回收器中，在垃圾回收器执行 Major GC 的过程中，该垃圾回收器通常也会执行 Minor GC。因此，我们通常认为 Major GC ≈ Full GC
3.1.2.3. 整堆回收 🌈

整堆回收（Full GC）是指：垃圾回收器垃圾回收堆区的内存空间、方法区的内存空间

[!NOTE] 注意事项

  1. 垃圾回收器执行Full GC 的开销非常大，一个优秀的 Java 程序应该尽量避免触发垃圾回收器执行 Full GC
3.1.2.4. 混合回收 🌈

混合回收是 G1 GC 独有的概念，同时也是该垃圾回收器的精髓。混合回收（Mixed GC）是指：垃圾回收器垃圾回收堆区中的新生代的 Region 的内存空间、部分堆区中的老年代的 Region 的内存空间

3.2. 垃圾回收器相关分类 🌈

  1. 以垃圾回收器垃圾回收的内存区域分类：
    1. 整堆垃圾回收器

      2. G1 GC
    2. 新生代垃圾回收器
      1. Serial GC
      2. ParNew GC
      3. Parallel Scavenge GC
    3. 老年代垃圾回收器
      1. CMS GC
      2. Serial Old GC
      3. Parallel Old GC
  2. 以垃圾回收器使用的垃圾回收策略分类：
    1. 串行垃圾回收器
      1. Serial GC
      2. Serial Old GC
    2. 并行垃圾回收器
      1. ParNew GC
      2. Parallel Old GC
      3. Parallel Scavenge GC
    3. 并发垃圾回收器
      1. ZGC
      2. G1 GC
      3. CMS GC

3.3.1. Serial GC + Serial Old GC 🌈

3.3.1.1. Serial GC + Serial Old GC 概述 🌈

Serial GC + Serial Old GC 支持 Minor GC、Full GC：

  1. Serial GC 使用单线程，采用串行垃圾回收策略，使用标记-复制算法执行 Minor GC
  2. Serial Old GC 使用单线程，采用串行垃圾回收策略，使用标记-整理算法执行 Full GC
3.3.1.2. Serial GC + Serial Old GC 相关流程 🌈
3.3.1.2.1. Minor GC 流程 🌈

当堆区中的 Eden 区的剩余的内存空间不足，JVM 不能直接为新的实例分配堆区中的 Eden 区的内存空间时，就会触发 Serial GC 执行 Minor GC:

  1. Serial GC 首先会执行 STW
  2. 然后 Serial GC 使用单线程，采用串行垃圾回收策略，使用标记-复制算法执行 Minor GC
  3. 在 Serial GC 执行完 Minor GC 后，Serial GC 会恢复运行被 “暂停” 的线程
3.3.1.2.2. Full GC 流程 🌈

触发 Serial Old GC 执行 Full GC 的条件：

  1. 当堆区中的老年代的剩余的内存空间不足，垃圾回收器在执行 Minor GC 的过程中，堆区中的 Survivor 0 区、Survivor 1 区中的实例晋升到堆区中的老年代失败时，就会触发 Serial Old GC 执行 Full GC
  2. 当堆区中的老年代的剩余的内存空间不足，JVM 不能直接为新的实例分配堆区中的老年代的内存空间时，就会触发 Serial Old GC 执行 Full GC
  3. 垃圾回收器在执行 Minor GC 之前，JVM 会先检查堆区中的老年代的剩余的、最大的、连续的内存空间，是否大于堆区中的新生代中的所有实例的内存空间之和（因为如果大于，就说明垃圾回收器执行 Minor GC 是绝对安全的）。然后 JVM 会再检查堆区中的老年代的剩余的、最大的、连续的内存空间，是否大于历次从堆区中的 Survivor 0 区、Survivor 1 区晋升到堆区中的老年代的实例的内存空间之和的平均值（因为垃圾回收器执行 Full GC 的开销非常大，所以 JVM 是能不让垃圾回收器执行 Full GC，就不让垃圾回收器执行 Full GC）。当 JVM 发现两者的条件不满足时，就会触发 Serial Old GC 执行 Full GC（术语是：空间分配担保失败）
  4. 调用 System.gc
    1. 当线程调用 System.gc 时，JVM 会建议垃圾回收器执行 Full GC（需要注意的是：只是建议，不保证一定会执行）
  5. 当方法区的内存空间不足时

Serial Old GC 执行 Full GC 的流程：

  1. Serial Old GC 首先会执行 STW
  2. 然后 Serial Old GC 使用单线程，采用串行垃圾回收策略，使用标记-整理算法执行 Full GC
  3. 在 Serial Old GC 执行完 Full GC 后，Serial Old GC 会恢复运行被 “暂停” 的线程

3.3.2. Parallel Scavenge GC + Parallel Old GC 🌈

3.3.2.1. Parallel Scavenge GC + Parallel Old GC 概述 🌈

Parallel Scavenge GC + Parallel Old GC 是 HotSpot JVM 在 Java8 的默认的垃圾回收器，支持 Minor GC、Full GC：

  1. Parallel Scavenge GC 使用多线程，采用并行垃圾回收策略，使用标记-复制算法执行 Minor GC
  2. Parallel Old GC 使用多线程，采用并行垃圾回收策略，使用标记-整理算法执行 Full GC

[!NOTE] 注意事项

  1. Parallel Scavenge GC + Parallel Old GC 也被称之为 “吞吐量优先的垃圾回收器组合”
  2. Parallel Scavenge GC + Parallel Old GC 具备自适应调节的能力。我们只需要配置少量 JVM 参数，Parallel Scavenge GC 就会根据运行 Java 程序时的状况，自动调整部分 JVM 参数，从而达到贴近我们配置的 JVM 参数的目的：
    1. -XX:MaxGCPauseMillis
      1. 用于配置 Parallel Scavenge GC + Parallel Old GC 在执行垃圾回收的过程中，执行 STW 的被允许的最长时间
    2. -XX:GCTimeRatio
      1. 用于配置 Parallel Scavenge GC + Parallel Old GC 执行垃圾回收的时间，占运行 Java 程序的总时间的比例
    3. -XX:+UseAdaptiveSizePolicy
      1. 用于开启 Parallel Scavenge GC + Parallel Old GC 的自适应调节的功能
3.3.2.2. Parallel Scavenge GC + Parallel Old GC 相关流程 🌈

Parallel Scavenge GC + Parallel Old GC 的流程相似于 Serial GC + Serial Old GC 的流程，只不过 Parallel Scavenge GC + Parallel Old GC 是使用多线程执行垃圾回收，并且具备自适应的调节的能力

3.3.3. ParNew GC + CMS GC 🌈

3.3.3.1. ParNew GC + CMS GC 概述 🌈

ParNew GC + CMS GC 支持 Minor GC、Major GC、Full GC：

  1. ParNew GC 使用多线程，采用并行垃圾回收策略，使用标记-复制算法执行 Minor GC
  2. CMS GC 使用多线程，采用并行垃圾回收策略、并发垃圾回收策略，使用标记-清除算法执行 Major GC
  3. Serial Old GC 使用单线程，采用串行垃圾回收策略，使用标记-整理算法执行 Full GC（需要注意的是：Serial Old GC 是 CMS GC 的后备垃圾回收器）

[!NOTE] 注意事项

  1. 虽然 CMS GC 是一个老年代垃圾回收器。但是 CMS GC 无法搭配在 HotSpot JVM 在 Java4 及以后就存在的 Parallel Scavenge GC。因此，ParNew GC 诞生的使命就是作为 CMS GC 配套的新生代垃圾回收器
  2. 由于 CMS GC 的后备垃圾回收器是 Serial Old GC，这是 G1 GC 取代 CMS GC 的核心原因之一
  3. 猴哥的烦恼箱 (｡•́︿•̀｡)：
    1. 既然使用标记-清除算法会产生内存空间碎片，那么为什么 CMS GC 在执行 Major GC 时，不使用标记-整理算法？
      1. 因为标记-整理算法需要移动存活的实例，而移动存活的实例会改变该实例的内存地址
      2. 所以如果 CMS GC 在运行并发清除阶段时，使用标记-整理算法，就必须执行 STW，否则会导致引用关系错乱
3.3.3.2. ParNew GC + CMS GC 相关流程 🌈
3.3.3.2.1. Minor GC 流程 🌈

ParNew GC 执行 Minor GC 的流程相似于 Parallel Scavenge GC 执行 Minor GC 的流程，只不过 ParNew GC 不具备自适应调节的能力

3.3.3.2.2. Major GC 流程 🌈

当堆区中的剩余的老年代的内存空间的使用率达到某阈值时，就会触发 CMS GC 执行 Major GC：

  1. 初始标记阶段（Initial Mark Phase）
    1. 在CMS GC 进行初始标记阶段的过程中，CMS GC 使用多线程，采用并行垃圾回收策略，标记出 GC Roots 直接可达的实例（需要注意的是：不做深度遍历)
    2. 在 CMS GC 进行初始标记阶段的过程中，CMS GC 会执行 STW。但是由于 CMS GC 需要标记的实例的数量较少，所以 CMS GC 执行 STW 的时间较短
  2. 并发标记阶段（Concurrent Mark Phase）
    1. 在 CMS GC 进行并发标记阶段的过程中，CMS GC 使用多线程，采用并发垃圾回收策略，遍历整个实例图，标记出所有 GC Roots 可达的实例
    2. CMS GC 进行并发标记阶段会耗费较长时间。但是在 CMS GC 进行并发标记阶段的过程中，CMS GC 不会执行 STW
  3. 重新标记阶段（Remark Phase）
    1. 在 CMS GC 进行重新标记阶段的过程中，CMS GC 使用多线程，采用并行垃圾回收策略，修正在 CMS GC 进行并发标记阶段的过程中，由于其他的线程继续运行，而导致标记不准确的部分实例
    2. 在 CMS GC 进行重新标记阶段的过程中，CMS GC 会执行 STW（需要注意的是：在 CMS GC 进行重新标记阶段的过程中，执行 STW 的时间，会比在 CMS GC 进行初始标记阶段的过程中，执行 STW 的时间要长）
  4. 并发清除阶段（Concurrent Sweep Phase）
    1. 在 CMS GC 进行并发清除阶段的过程中，CMS GC 使用多线程，采用并发垃圾回收策略，使用标记-清理算法执行 Major GC
    2. 在 CMS GC 进行并发清除阶段的过程中，CMS GC 不会执行 STW
  5. 重置线程阶段（Reset Thread Phase）
    1. 在 CMS GC 进行重置线程阶段的过程中，CMS GC 使用多线程，采用并发垃圾回收策略，“恢复” 在本次在 CMS GC 执行 Major GC 的过程中，使用到的数据结构、实例被标记的状态，为下次 CMS GC 执行 Major GC 做准备
    2. 在 CMS GC 进行重置线程阶段的过程中，CMS GC 不会执行 STW
3.3.3.2.3. Full GC 流程 🌈

想象这样一个问题：虽然 CMS GC 的策略是 “主动预防“。但是在运行 Java 程序的过程中，仍然可能出现一些极端的情况（例如：CMS GC 执行 Major GC 垃圾回收堆区中的老年代中的实例的速度，赶不上堆区中的 Survivor 0 区、Survivor 1 区中的实例晋升到堆区中的老年代的速度）

为了解决这一问题，CMS GC 会使用 CMS GC 的后备垃圾回收器（即 Serial Old GC）执行 Full GC

3.3.4. G1 GC 🌈

3.3.4.1. G1 GC 概述 🌈

G1 GC 是 HotSpot JVM 在 Java9 及以后的默认的垃圾回收器，支持 Minor GC、Mixed GC、Full GC：

  1. G1 GC 使用多线程，采用并发垃圾回收策略，使用标记-复制算法执行 Minor GC
  2. G1 GC 使用多线程，采用并行垃圾回收策略、并发垃圾回收策略，使用标记-清除算法、标记-复制算法执行 Mixed GC（需要注意的是：G1 GC 使用标记-清除算法垃圾回收的是 Humongous 区）
  3. G1 GC 使用单线程或多线程（需要注意的是：看 HotSpot JVM 的 Java 版本），采用并发垃圾回收策略，使用标记-整理算法执行 FUll GC

在之前的垃圾回收器中，JVM 将堆区的内存空间划分新生代、老年代。但是在 G1 GC 中，JVM 将堆区的内存空间划分为多个内存空间相等的 Region

除此之外，G1 GC 还引入一种新的内存区域（即 Humongous 区）。多个连续的 Region 组成一个 Humongous 区，专门用于存储超大实例。如果实例需要的内存空间超过一个 Region 的 50% 的内存空间，JVM 就会为该实例分配 Humongous 区的内存空间（需要注意的是：我们通常把 Humongous 区看作是堆区中的老年代中的一部分）

那么问题来了：既然 G1 GC 执行垃圾回收是按照 Region 为单位，那么 G1 GC 该如何判断其他 Region 中的实例，是否引用 G1 GC 要垃圾回收的 Region 中的实例？

为了解决这一问题，G1 GC 引入了记忆集（Rset、Remembered Set） 。每个 Region 都维护了一个 RSet，专门用于记录引用该 Region 中的实例的其他 Region 中的实例

[!NOTE] 注意事项

  1. JVM 默认根据堆区的最大的内存空间，将堆区的内存空间划分为 2048 个内存空间相等的 Region，每个 Region 的内存空间为 1MB - 32MB（例如：堆区的最大内存空间为 32GB，那么每个 Region 的内存空间为 16MB。所以在刚运行 Java 程序，堆区的内存空间为 4GB 时，Region 只有 256 个）
    1. 需要注意的是：
      1. Region 的内存空间必须为 $2^n$ MB
      2. 我们可以通过配置 JVM 参数，手动指定 Region 的内存空间（需要注意的是：在这种情况下，Region 的数量不再固定为 2048 个）
  2. G1 GC 具备自适应调节的能力。我们只需要配置少量 JVM 参数，G1 GC 就会根据运行 Java 程序时的状况，自动调整部分 JVM 参数，从而达到贴近我们配置的 JVM 参数的目的：
    1. -XX:MaxGCPauseMillis
      1. 用于配置 Parallel Scavenge GC + Parallel Old GC 在执行垃圾回收的过程中，执行 STW 的被允许的最长时间（需要注意的是：对 G1 GC 执行 Full GC 不起作用）
3.3.4.2. G1 GC 相关流程 🌈
3.3.4.2.1. Minor GC 流程 🌈

当堆区中的 Eden 区的内存空间不足，JVM 不能直接为新的实例分配堆区中的 Eden 区的内存空间时，就会触发 G1 GC 执行 Minor GC（需要注意的是：在 G1 GC 中，Minor GC 被称之为 “Yong-Only Phase”）：

  1. G1 GC 首先会执行 STW
  2. 然后 G1 GC 会将所有堆区中的新生代的 Region 存储到回收集合（Cset、Collection Set）
  3. 接着 G1 GC 会使用多线程，采用并行垃圾回收策略，使用标记-复制算法执行 MINOR GC
  4. 在 G1 GC 执行完 Minor GC 后，G1 GC 会恢复运行被 “暂停” 的线程
3.3.4.2.2. Mixed GC 流程 🌈

当堆区中的老年代的内存空间的使用率达到某阈值时，就会触发 G1 GC 执行 Mixed GC（需要注意的是：在 G1 GC 中，Mixed GC 被称之为 “Space-Reclamation Phase”）：

  1. 初始标记阶段（Initial Mark Phase）
    1. 在 G1 GC 进行初始标记阶段的过程中，G1 GC 使用多线程，采用并行垃圾回收策略，标记出 GC Roots 直接可达的实例（需要注意的是：不做深度遍历）
    2. 在 G1 GC 进行初始标记阶段的过程中，G1 GC 会执行 STW。但是由于 G1 GC 需要标记的实例的数量较少，所以 G1 GC 执行 STW 的时间较短
  2. 并发标记阶段（Concurrent Mark Phase）
    1. 在 G1 GC 进行并发标记阶段的过程中，G1 GC 使用多线程，采用并发垃圾回收策略，遍历整个实例图，标记出所有 GC Roots 可达的实例
    2. G1 GC 进行并发标记阶段会耗费较长时间。但是在 G1 GC 进行并发标记阶段的过程中，G1 GC 不会执行 STW
  3. 重新标记阶段（Remark Phase）
    1. 在 G1 GC 进行重新标记阶段的过程中，G1 GC 使用多线程，采用并行垃圾回收策略，修正在 G1 GC 进行并发标记阶段的过程中，由于其他的线程继续运行，而导致标记不准确的部分实例
    2. 在 G1 GC 进行重新标记阶段的过程中，G1 GC 会执行 STW（需要注意的是：在 G1 GC 进行重新标记阶段的过程中，执行 STW 的时间，会比在 G1 GC 进行初始标记阶段的过程中，执行 STW 的时间要长）
  4. 筛选清理阶段（Cleanup Phase）
    1. 在 G1 GC 进行筛选清理阶段的过程中，G1 GC 会将所有堆区中的新生代的 Region 存储代 Cset。然后 G1 GC 会排序所有 Region 的 “回收价值”，并根据我们配置的 JVM 参数，将部分高 “回收价值” 的堆区中的老年代的 Region 存储到 Cset
    2. 在 G1 GC 进行筛选清理阶段，G1 GC 会执行 STW
  5. 混合回收阶段（Mixed GC Phase）
    1. 在 G1 GC 进行混合回收阶段的过程中，G1 GC 使用多线程，采用并行垃圾回收策略，使用标记-复制算法执行 Mixed GC（需要注意的是：实际会垃圾回收所有堆区中的新生代的 Region 和部分高 “回收价值” 的堆区中的老年代的 Region ）
    2. 在 G1 GC 进行混合回收阶段的过程中，G1 GC 会执行 STW
3.3.4.2.3. Full GC 流程 🌈

虽然 G1 GC 的策略也是 “主动预防”。但是在运行 Java 程序的过程中，也可能出现一些极端情况。为了解决这一问题，G1 GC 会执行 Full GC：

  1. HotSpot JVM 在 Java 10 之前，是使用单线程，采用串行垃圾回收策略，使用标记-整理算法执行 Full GC
  2. HotSpot JVM 在 Java10 及以后，是使用多线程，采用并行垃圾回收策略，使用标记-整理算法执行 Full GC

触发 G1 GC 执行 Full GC 的条件相似于触发 Serial Old GC 执行 Full GC 的条件，只不过 G1 GC 并不存在传统意义上的 “空间担保分配”（因为它是基于 Region 的）。但是当堆区中的空闲的 Region 的数量不足，G1 GC 在执行 Minor GC、Major GC 的过程中，将 Region 中的实例复制到另一个 Region 失败时，就会触发 G1 GC 执行 Full GC（需要注意的是：这本质上也是一种 “空间担保分配”）

3.3.5. 对象的 finalization 机制

Object 类提供了 void finalize() 方法，若某个类对其进行了重写，那么在 GC 该对象之前，会先调用 finalize()，允许对象在销毁前执行一些自定义逻辑

然而，在调用 finalize() 的时候，可能导致对象 “复活”（即重新与引用链上的某个对象建立关联）。因此，JVM 在判断对象是否可回收时，通常需要经历两次标记过程：

  1. 第一次标记
    1. GC 通过可达性分析，发现某些对象与 GC Roots 之间已不存在引用链，判定它们为 “可回收候选对象”
  2. 第二次标记
    1. 如果对象没有重写 finalize() 方法，或其 finalize() 方法已经被调用过了
      1. 该对象会被直接判定为不可复活，继续进入回收流程
    2. 如果对象重写了 finalize() 方法，并且尚未被执行过
      1. GC 会将该对象插入到 F-Queue 队列
      2. 随后，一个由虚拟机自动创建的、低优先级的 Finalizer 线程会异步触发 finalize() 方法的执行
      3. 在下一个 GC 周期中，当进行新一轮的可达性分析时，GC 会再次检查那些曾经在 F-Queue 中、并且其 finalize() 方法已经被执行完毕的对象
        1. 如果该对象重新与引用链上的某个对象建立了联系，那么它将被移出即将回收的集合，实现 “复活”
        2. 如果该对象依然没有与引用链上的某个对象建立了联系，那么它就会被最终判定为垃圾，在下一次垃圾回收时被真正地回收

需要注意的是，从 Java9 开始，Object 类的 void finalize() 方法已经被正式弃用了
