2.2.1. 字节码解释器

2.2.1.1. 字节码解释器概述

字节码解释器（C++ 解释器）是指：JVM 的设计团队使用 C++ 语言编写的解释器

2.2.1.2. 字节码解释器底层原理

字节码解释器本质上就是一个 while 循环，该 while 循环中嵌套了一个巨大的 switch-case 语句

当字节码解释器解释执行 JVM 字节码指令时，其实就是该解释器进行一次 while 循环，然后根据 switch-case 语句调用一次该 JVM 字节码指令对应的 C++ 方法

2.2.1.3. 字节码解释器利弊分析
2.2.1.3.1. 优势
  1. 移植性高
    1. 由于字节码解释器是 JVM 的设计团队使用 C++ 语言编写的，所以只要 C++ 语言适配了的系统（即该系统拥有兼容的 C++ 编译器、C++ 运行时环境），我们就可以在该系统上使用字节码解释器
2.2.1.3.2. 劣势
    1. 我们知道：当字节码解释器解释执行 JVM 字节码指令时，其实就是该解释器进行一次 while 循环，然后根据 switch-case 语句调用一次该 JVM 字节码指令对应的 C++ 方法
    2. 所以，每当字节码解释器执行一条 JVM 字节码指令时，该解释器就会进行一次（while 循环 && switch-case 语句的判断 && C++ 方法的执行 && 控制流的跳转）
    3. 但是，我们一行简单的 Java 代码（例如：a = b + c）就会被 javac 编译器编译成多条 JVM 字节码指令
    4. 所以，JVM 哪怕只是执行一行简单的 Java 代码，该解释器都会重复的执行上述的流程
    5. 然而，只有当该解释器执行 C++ 方法时，该解释器才是在真正的执行实际的业务
    6. 所以，当 JVM 使用字节码解释器解释执行 JVM 字节码指令时，该解释器会造成极大的算力的浪费（术语是：管理成本远远大于生产成本）
    1. 由于字节码解释器是 JVM 的设计团队使用 C++ 语言编写的解释器
    2. 所以，字节码解释器其实是被 C++ 编译器编译成 CPU 对应的机器码指令的
    3. 我们知道：如果我们使用 C++ 语言编写一个程序，该编译器是可以在编译该程序时深度的优化该程序的（例如：该编译器可以将部分的数据存储在寄存器中。因为该编译器拥有该程序的全局视角，清楚的知道（方法的调用链 && 变量的生命周期 && ...））
    4. 但是，当该编译器在编译该解释器时，该编译器拥有的该解释器的全局视角会被该解释器的特性显著的削弱（因为该解释器本质上就是一个 while 循环 + switch-case 语句。在该 while 循环外，字节码解释器中的 C++ 代码并不存在可供 C++ 编译器分析的复杂的方法的调用链。而在该 while 循环中，尽管 switch-case 语句中的各个分支中包含了大量的 C++ 方法，但是由于该编译器并不知道在 JVM 运行该解释器的过程中，该解释器要按照什么顺序来执行 C++ 方法（因为该顺序要根据 JVM 字节码指令的顺序，而该 JVM 字节码指令的顺序又要根据我们编写 Java 代码））
    5. 所以，该解释器不知道我们下一步要去执行哪一个 C++ 方法，更不知道该解释器正在执行的 C++ 方法中，哪些数据会被之后要去执行的 C++ 方法继续使用
    6. 所以 C++ 编译器不得不把该解释器正在执行的 C++ 方法中的数据写回内存空间中（需要注意的是：该数据其实是被该解释器写回到该解释器正在执行的 C++ 方法所属的 Java 方法对应的栈帧中的操作数栈的内存空间中）
    7. 所以，当 JVM 使用字节码解释器解释执行 JVM 字节码指令时，会产生大量的读写内存空间的行为
    1. 现代的 CPU 极其的依赖于复杂的（指令流水线 && 分支预测）技术来提升执行的速度
    2. 但是，字节码解释器中的 switch-case 语句对于 CPU 来说，其实就是一个间接分支
    3. 所以，CPU 的分支预测其实是很难预测 switch-case 语句要去跳转的目标地址的
    4. 可是，一旦 CPU 的分支预测失败，CPU 就必须清空已经加载到流水线中的指令，并重新加载正确的指令

2.2.2. 模板解释器

2.2.2.1. 模板解释器概述

模板解释器是指：JVM 的设计团队使用汇编语言编写的解释器

2.2.2.2. 模板解释器底层原理

模板解释器的底层原理是指：在我们启动 JVM 的过程中，模板解释器会为每一条 JVM 字节码指令生成 CPU 对应的机器码指令

2.2.2.3. 模板解释器相关流程

当模板解释器解释执行 JVM 字节码指令时，其实就是该解释器读取当前要去执行的 JVM 字节码指令，然后直接跳转到该 JVM 字节码指令被该解释器编译后的 CPU 对应的机器码指令并执行

在该解释器执行完该 CPU 对应的机器码指令之后，该解释器会再次读取下一条要去执行的 JVM 字节码指令，然后再跳转到该 JVM 字节码指令被该解释器编译后的 CPU 对应的机器码指令并执行

[!NOTE] 注意事项

  1. 由于模板解释器是在 CPU 对应的机器码指令的层面直接进行控制流的跳转，所以该解释器进行跳转的效率非常高

4. 垃圾回收器

4.1. 垃圾回收器前置基础

4.1.1. 垃圾回收概述

垃圾回收器（GC、Garbage Collection）用于在 Java 程序运行的过程中，清理堆区、方法区不再使用的资源，从而达到释放内存空间的目的：

  1. 清理堆区不再使用的资源
    1. 清理不再被引用的实例（需要注意的是：堆区是 GC 工作的主要内存区域）
  2. 清理方法区不再使用的资源

GC（Garbage Collection） 是在程序运行过程中，自动回收堆区和方法区中不再使用的内存资源：

  1. 堆区垃圾回收
    1. 回收那些已经没有任何引用指向的对象，是 JVM 垃圾回收的主要工作区域
  2. 方法区垃圾回收
    1. 常量池的回收
    2. 类的卸载
      1. 类的卸载比较严格，当满足以下三个条件时，一个类才有可能被卸载：
        1. 该类的所有实例都已被回收
        2. 加载该类的 ClassLoader 已经被回收
        3. 该类对应的 java.lang.Class 对象没有在任何地方被引用
      2. 然而多数情况下，一个类的 ClassLoader 和 Class 对象都会被长期引用

[!NOTE] 注意事项

  1. 从回收频率上看，其规律主要包括：
    1. 频繁收集新生代
    2. 较少收集老年代
    3. 基本不动永久代（方法区）

4.1.2. 垃圾回收相关策略

  1. 串行垃圾回收策略
    1. 当垃圾回收器执行垃圾回收时，会暂停其他的所有的执行 Java 代码的线程（术语是：STW），并仅使用单个线程执行垃圾回收
  2. 并行垃圾回收策略
    1. 当垃圾回收器执行垃圾回收时，会执行 STW，并使用多个线程并行的执行垃圾回收
  3. 并发垃圾回收策略
    1. 当垃圾回收器执行垃圾回收时，不会执行 STW，并允许执行垃圾回收的线程，与其他的执行 Java 代码的线程同时运行
    5. 垃圾回收器执行垃圾回收的过程中，不会执行 STW
    6. 当 CPU 的核数不足时，垃圾回收器执行垃圾回收的线程、其他的在 JVM 中执行 Java 代码的线程，可能会出现交替运行的情况（需要注意的是：交替运行的情况才符合传统意义上的并发）
    7. 即便并发垃圾回收器采用了并发垃圾回收策略，也并不代表在运行 Java 程序的过程中，垃圾回收器不会执行 STW。但是这个情况没有被体现在并发垃圾回收策略中

[!NOTE] 注意事项

  1. 迄今为止，市面上并不存在任何一种垃圾回收器，能够实现在运行 Java 程序的过程中，垃圾回收器不会执行 STW

4.1.3. 垃圾回收相关事件

4.1.3.1. 新生代回收
4.1.3.1.1. 新生代回收概述

新生代回收（Minor GC、Young GC）是指：垃圾回收器垃圾回收（运行时数据区 ➔ 堆区 ➔ 新生代）的内存空间

4.1.3.1.2. 新生代回收触发条件
  1. 当（运行时数据区 ➔ 堆区 ➔ Eden 区）的剩余的内存空间不足，JVM 不能直接为新的实例分配 Eden 区的内存空间时，就触发让垃圾回收器执行 Minor GC
4.1.3.2. 老年代回收

老年代回收（Major GC）是指：垃圾回收器垃圾回收（运行时数据区 ➔ 堆区 ➔ 老年代）的内存空间

[!NOTE] 注意事项

  1. Major GC 这个单词很容易引起混淆，因为该单词在不同的垃圾回收器中有不同的含义：
    1. 在 CMS GC 中，Major GC 是指：垃圾回收器垃圾回收（运行时数据区 ➔ 堆区 ➔ 老年代）的内存空间
    2. 但是，在其他的垃圾回收器中，当它们执行 Minor GC 时，通常也会执行 Minor GC，因此在这种情况下我们通常认为 Major GC ≈ Full GC
4.1.3.3. 整堆回收

整堆回收（Full GC）是指：垃圾回收器垃圾回收（运行时数据区 ➔ 堆区 && 运行时数据区 ➔ 方法区）的内存空间

[!NOTE] 注意事项

  1. 垃圾回收器执行 Full GC 的开销非常大，所以一个优秀的 Java 程序应该尽量避免触发垃圾回收器执行 Full GC
4.1.3.4. 混合回收

混合回收（Mixed GC）是指：垃圾回收器垃圾回收（所有运行时数据区 ➔ 堆区 ➔ 新生代 && 部分运行时数据区 ➔ 堆区 ➔ 老年代）的 Region 的内存空间

[!NOTE] 注意事项

  1. Mixed GC 是 G1 GC 特有的概念，同时也是 G1 GC 的精髓

4.1.4. 垃圾标记相关算法

4.1.4.1. 垃圾标记算法概述

在垃圾回收器执行垃圾回收之前，必须区分实例是否已经 “死亡”

4.1.4.2. 垃圾标记算法相关分类
  1. 引用计数算法
  2. 可达性分析算法
4.1.4.3. 引用计数算法
4.1.4.3.1. 引用计数算法概述

引用计数算法是指：每个实例维护一个 int 类型的属性（术语是：计数器），用于记录该实例被引用的次数（需要注意的是：当计数器为 0 时，说明实例已经“死亡”）

4.1.4.3.2. 引用计数算法优缺点

1. 优点

  1. JVM 实现引用计数算法相对简单
  2. 垃圾回收器识别已经 “死亡” 的实例相对容易
  3. 垃圾回收器垃圾回收已经 “死亡” 的实例相对 “无需等待”（例如：当线程执行方法时，可以 “顺带角” 的看一下实例是否已经 “死亡”。如果实例已经死亡，可以 “顺带角” 的垃圾回收该实例）

2. 缺点

  1. 由于需要频繁更新计数器，所以增加了时间上的开销
  2. 由于每个实例需要维护一个 int 类型的属性，所以增加了内存空间上的开销
  4. 由于无法处理循环引用，所以可能会导致内存泄漏（需要注意的是：这是引用计数算法的致命缺陷，也是 HotSpot JVM，并没有采用该算法的原因）

4.1.4.4. 可达性分析算法

可达性分析算法（根搜索算法、追踪性垃圾收集算法）是以根对象集合（GC Roots，一组始终处于活跃状态的引用）为起始点，自上而下地搜索与这些根对象相关联的目标对象是否可达,内存中的所有存活对象，都会被根对象集合直接或间接连接

搜索过程中所经过的路径被称为引用链（Reference Chain），如果某个目标对象在内存中没有任何引用链与 GC Roots 相连，那么它就是不可达的，也就意味着该对象已死亡

[!NOTE] 注意事项

  1. 如果要使用可达性分析算法判断内存是否可回收，分析工作必须在一个能够保证内存状态一致性的快照中进行
  2. 如果无法满足这一点，分析结果的准确性就无法保证，这也是导致 GC 进行时必须执行 “Stop The World” 的一个重要原因。
  3. 即便是号称几乎不会停顿的 CMS 收集器，在枚举根节点阶段，也必须暂停所有应用线程（主线程、工作线程、守护线程）

4.1.5.1. 标记-清除算法
4.1.5.1.1. 标记-清除算法概述

标记-清除算法是垃圾清除算法中最基本、最古老的一种，也是许多垃圾清除算法的基础

4.1.5.1.2. 标记-清除算法相关流程
4.1.5.1.3. 标记-清除算法

标记-清除算法是垃圾回收中最基本和最古老的一种算法，也是许多现代垃圾回收算法的基础，其相关流程主要包括：

  1. 标记阶段
    1. 垃圾回收器从一组被称为 GC Roots 的对象出发，遍历所有从 GC Roots 可达的对象，并标记这些对象为 “存活” （通常通过在对象头中设置一个标记位 mark bit 来实现）
    2. 这个遍历过程通常采用递归的深度优先搜索（DFS）或广度优先搜索（BFS）方式，沿着对象引用链不断向下，直到所有可达对象都被标记完毕
  2. 清除阶段
    1. 垃圾回收器会遍历整个堆内存
    2. 对于未被标记的对象（即不可达对象），它们的内存会被回收，并把这些地址加入到空闲内存列表中，供后续的新对象分配使用
    3. 对于已标记的对象（即存活对象），它们的标记位会在此阶段被清除，为下一次垃圾回收做准备

其优点是：

  1. 实现简单
    1. 相比于标记-复制算法、标记-整理算法，标记-清除算法的基本逻辑相对直观，容易实现
  2. 能够处理循环引用

其缺点是：

  1. 内存碎片化
    1. 回收后的内存空间不连续，产生大量的内存碎片
    2. 需要额外维护一个空闲列表用于记录空闲地址，增加内存开销
    3. 当后续需要分配较大对象时，即使总的空闲内存足够，也可能因为没有足够大的连续空间而导致分配失败

[!NOTE] 注意事项

  1. 无论采用哪种垃圾回收算法，也无论使用哪种垃圾回收器，都不可避免地会存在 STW（Stop The World），即暂停所有应用线程以执行垃圾回收操作
  2. 一个优秀的垃圾回收器，必须在吞吐量与响应速度之间取得良好平衡，既能高效回收内存，又不会严重影响程序的响应性能
4.1.5.1.4. 标记-复制算法

标记-复制算法的主要流程包括：

  1. 标记阶段
    1. 垃圾回收器从一组被称为 GC Roots 的对象出发，遍历所有从 GC Roots 可达的对象，并标记这些对象为 “存活” （通常通过在对象头中设置一个标记位 mark bit 来实现）
    2. 这个遍历过程通常采用递归的深度优先搜索（DFS）或广度优先搜索（BFS）方式，沿着对象引用链不断向下，直到所有可达对象都被标记完毕
  2. 复制阶段
    1. 垃圾回收器会遍历整个 From 空间
    2. 对于已标记的对象（既存活对象），它们会被复制到 To 空间，并按顺序紧凑排列，从而自然地消除了内存碎片。并且在复制的同时更新所有引用，确保指向已移动对象的指针保持正确
    3. 对于未被标记的对象（既不可达对象），它们仍然留在 From 空间，在复制完成后，它们的内存会被回收，并把这些地址加入到空闲内存列表中，供后续的新对象分配使用

下一次垃圾回收时，From 和 To 空间将交换角色，继续重复同样的回收流程。总结来说，复制算法以牺牲空间换取了时间，并彻底解决了内存碎片问题

其优点是：

  1. 实现简单：
    1. 相比于复杂的标记-整理算法，标记-复制算法的基本逻辑相对直观，容易实现
  2. 消除内存碎片化
    1. 相比于标记-清除算法，标记-复制算法消除了内存碎片化
  3. 能够处理循环引用

其缺点是：

  1. 内存空间利用率低：
    1. 相比于标记-清除算法、标记整理算法，标记-复制算法需要将可用的内存空间一分为二，每次只使用一半。
  2. 不适用于存活对象占多数的场景
    1. 需要移动的存活对象越少，其开销就越低
    2. 当存活对象较多时，复制的开销会显著增加

4.1.5.1.5. 标记-整理算法（标记压缩算法）

标记-整理算法是垃圾回收领域中一种重要的算法，它融合了标记-清除算法的优点，同时解决了其所带来的内存碎片问题。因此，它常被视为标记-清除算法的升级版，即 “标记-清除 plus”。

标记-整理算法的主要流程包括：

  1. 标记阶段
    1. 垃圾回收器从一组被称为 GC Roots 的对象出发，遍历所有从 GC Roots 可达的对象，并标记这些对象为 “存活” （通常通过在对象头中设置一个标记位 mark bit 来实现）
    2. 这个遍历过程通常采用递归的深度优先搜索（DFS）或广度优先搜索（BFS）方式，沿着对象引用链不断向下，直到所有可达对象都被标记完毕
  2. 整理阶段
    1. 垃圾回收器会遍历整个堆内存
    2. 对于未被标记的对象（即不可达对象），它们的内存会被回收，并把这些地址加入到空闲内存列表中，供后续的新对象分配使用
    3. 对于已标记的对象（即存活对象），它们会被按顺序移动到堆的一段（通常是起始地址）。并且在复制的同时更新所有引用，确保指向已移动对象的指针保持正确

其优点是：

  1. 消除内存碎片化
    1. 相比于标记-清除算法，标记-整理算法消除了内存碎片化
  2. 内存利用率高：
    1. 相比于标记-复制算法，标记-整理算法可以充分利用整个堆空间进行对象分配
  3. 能够处理循环引用

其缺点是：

  1. 实现复杂：
    1. 相比于标记-清除、标记-复制算法，标记-整理算法的实现复杂性更高

4.3.1. Serial GC + Serial Old GC

4.3.1.1. Serial GC + Serial Old GC 支持的垃圾回收相关事件

Serial GC + Serial Old GC 支持 Minor GC、Full GC：

  1. Serial GC 使用单线程，采用串行垃圾回收策略，使用标记-复制算法执行 Minor GC
  2. Serial Old GC 使用单线程，采用串行垃圾回收策略，使用标记-整理算法执行 Full GC
4.3.1.2. Serial GC + Serial Old GC 相关流程
4.3.1.2.1. Serial GC 执行 Minor GC 的过程

当（运行时数据区 ➔ 堆区 ➔ Eden 区）的剩余的内存空间不足，JVM 不能直接为新的实例分配 Eden 区的内存空间时，就会触发 Serial GC 执行 Minor GC

  1. 首先，Serial GC 会执行 STW
  2. 然后，Serial GC 使用单线程，采用串行垃圾回收策略，使用标记-复制算法执行 Minor GC
  3. 在 Serial GC 执行完 Minor GC 之后，Serial GC 会恢复被暂停的执行 Java 代码的线程
4.3.1.2.2. Serial Old GC 执行 Full GC 的过程

触发 Serial Old GC 执行 Full GC 的条件有：

1.

  1. 当堆区中的老年代的剩余的内存空间不足，垃圾回收器在执行 Minor GC 的过程中，堆区中的 Survivor 0 区、Survivor 1 区中的实例晋升到堆区中的老年代失败时，就会触发 Serial Old GC 执行 Full GC
  2. 当堆区中的老年代的剩余的内存空间不足，JVM 不能直接为新的实例分配堆区中的老年代的内存空间时，就会触发 Serial Old GC 执行 Full GC
  3. 垃圾回收器在执行 Minor GC 之前，JVM 会先检查堆区中的老年代的剩余的、最大的、连续的内存空间，是否大于堆区中的新生代中的所有实例的内存空间之和（因为如果大于，就说明垃圾回收器执行 Minor GC 是绝对安全的）。然后 JVM 会再检查堆区中的老年代的剩余的、最大的、连续的内存空间，是否大于历次从堆区中的 Survivor 0 区、Survivor 1 区晋升到堆区中的老年代的实例的内存空间之和的平均值（因为垃圾回收器执行 Full GC 的开销非常大，所以 JVM 是能不触发垃圾回收器执行 Full GC，就不触发垃圾回收器执行 Full GC）。当 JVM 发现两者的条件不满足时，就会触发 Serial Old GC 执行 Full GC（术语是：空间分配担保失败）
  4. 调用 System.gc
    1. 当线程调用 System.gc 时，JVM 会建议垃圾回收器执行 Full GC（需要注意的是：只是建议，不保证一定会执行）
  5. 当方法区的内存空间不足时

Serial Old GC 执行 Full GC 的流程：

  1. Serial Old GC 首先会执行 STW
  2. 然后 Serial Old GC 使用单线程，采用串行垃圾回收策略，使用标记-整理算法执行 Full GC
  3. 在 Serial Old GC 执行完 Full GC 后，Serial Old GC 会恢复运行被 “暂停” 的线程

4.3.2. Parallel Scavenge GC + Parallel Old GC

4.3.2.1. Parallel Scavenge GC + Parallel Old GC 概述

Parallel Scavenge GC + Parallel Old GC 是 HotSpot JVM 在 Java8 的默认的垃圾回收器，支持 Minor GC、Full GC：

  1. Parallel Scavenge GC 使用多线程，采用并行垃圾回收策略，使用标记-复制算法执行 Minor GC
  2. Parallel Old GC 使用多线程，采用并行垃圾回收策略，使用标记-整理算法执行 Full GC

[!NOTE] 注意事项

  1. Parallel Scavenge GC + Parallel Old GC 也被称之为 “吞吐量优先的垃圾回收器组合”
  2. Parallel Scavenge GC + Parallel Old GC 具备自适应调节的能力。我们只需要配置少量 JVM 参数，Parallel Scavenge GC 就会根据运行 Java 程序时的状况，自动调整部分 JVM 参数，从而达到贴近我们配置的 JVM 参数的目的：
    1. -XX:MaxGCPauseMillis
      1. 用于配置 Parallel Scavenge GC + Parallel Old GC 在执行垃圾回收的过程中，执行 STW 的被允许的最长时间
    2. -XX:GCTimeRatio
      1. 用于配置 Parallel Scavenge GC + Parallel Old GC 执行垃圾回收的时间，占运行 Java 程序的总时间的比例
    3. -XX:+UseAdaptiveSizePolicy
      1. 用于开启 Parallel Scavenge GC + Parallel Old GC 的自适应调节的功能
4.3.2.2. Parallel Scavenge GC + Parallel Old GC 相关流程

Parallel Scavenge GC + Parallel Old GC 的流程相似于 Serial GC + Serial Old GC 的流程，只不过 Parallel Scavenge GC + Parallel Old GC 是使用多线程执行垃圾回收，并且具备自适应的调节的能力

4.3.3. ParNew GC + CMS GC

4.3.3.1. ParNew GC + CMS GC 概述

ParNew GC + CMS GC 支持 Minor GC、Major GC、Full GC：

  1. ParNew GC 使用多线程，采用并行垃圾回收策略，使用标记-复制算法执行 Minor GC
  2. CMS GC 使用多线程，采用并行垃圾回收策略、并发垃圾回收策略，使用标记-清除算法执行 Major GC
  3. Serial Old GC 使用单线程，采用串行垃圾回收策略，使用标记-整理算法执行 Full GC（需要注意的是：Serial Old GC 是 CMS GC 的后备垃圾回收器）

[!NOTE] 注意事项

  1. 虽然 CMS GC 是一个老年代垃圾回收器。但是 CMS GC 无法搭配在 HotSpot JVM 在 Java4 及以后就存在的 Parallel Scavenge GC。因此，ParNew GC 诞生的使命就是作为 CMS GC 配套的新生代垃圾回收器
  2. 由于 CMS GC 的后备垃圾回收器是 Serial Old GC，这是 G1 GC 取代 CMS GC 的核心原因之一
  3. 猴哥的烦恼箱 (｡•́︿•̀｡)：
    1. 既然使用标记-清除算法会产生内存空间碎片，那么为什么 CMS GC 在执行 Major GC 时，不使用标记-整理算法？
      1. 因为标记-整理算法需要移动存活的实例，而移动存活的实例会改变该实例的内存地址
      2. 所以如果 CMS GC 在运行并发清除阶段时，使用标记-整理算法，就必须执行 STW，否则会导致引用关系错乱
4.3.3.2. ParNew GC + CMS GC 相关流程
4.3.3.2.1. Minor GC 流程

ParNew GC 执行 Minor GC 的流程相似于 Parallel Scavenge GC 执行 Minor GC 的流程，只不过 ParNew GC 不具备自适应调节的能力

4.3.3.2.2. Major GC 流程

当堆区中的剩余的老年代的内存空间的使用率达到某阈值时，就会触发 CMS GC 执行 Major GC：

  1. 初始标记阶段（Initial Mark Phase）
    1. 在CMS GC 进行初始标记阶段的过程中，CMS GC 使用多线程，采用并行垃圾回收策略，标记出 GC Roots 直接可达的实例（需要注意的是：不做深度遍历)
    2. 在 CMS GC 进行初始标记阶段的过程中，CMS GC 会执行 STW。但是由于 CMS GC 需要标记的实例的数量较少，所以 CMS GC 执行 STW 的时间较短
  2. 并发标记阶段（Concurrent Mark Phase）
    1. 在 CMS GC 进行并发标记阶段的过程中，CMS GC 使用多线程，采用并发垃圾回收策略，遍历整个实例图，标记出所有 GC Roots 可达的实例
    2. CMS GC 进行并发标记阶段会耗费较长时间。但是在 CMS GC 进行并发标记阶段的过程中，CMS GC 不会执行 STW
  3. 重新标记阶段（Remark Phase）
    1. 在 CMS GC 进行重新标记阶段的过程中，CMS GC 使用多线程，采用并行垃圾回收策略，修正在 CMS GC 进行并发标记阶段的过程中，由于其他的线程继续运行，而导致标记不准确的部分实例
    2. 在 CMS GC 进行重新标记阶段的过程中，CMS GC 会执行 STW（需要注意的是：在 CMS GC 进行重新标记阶段的过程中，执行 STW 的时间，会比在 CMS GC 进行初始标记阶段的过程中，执行 STW 的时间要长）
  4. 并发清除阶段（Concurrent Sweep Phase）
    1. 在 CMS GC 进行并发清除阶段的过程中，CMS GC 使用多线程，采用并发垃圾回收策略，使用标记-清理算法执行 Major GC
    2. 在 CMS GC 进行并发清除阶段的过程中，CMS GC 不会执行 STW
  5. 重置线程阶段（Reset Thread Phase）
    1. 在 CMS GC 进行重置线程阶段的过程中，CMS GC 使用多线程，采用并发垃圾回收策略，“恢复” 在本次在 CMS GC 执行 Major GC 的过程中，使用到的数据结构、实例被标记的状态，为下次 CMS GC 执行 Major GC 做准备
    2. 在 CMS GC 进行重置线程阶段的过程中，CMS GC 不会执行 STW
4.3.3.2.3. Full GC 流程

想象这样一个问题：虽然 CMS GC 的策略是 “主动预防“。但是在运行 Java 程序的过程中，仍然可能出现一些极端的情况（例如：CMS GC 执行 Major GC 垃圾回收堆区中的老年代中的实例的速度，赶不上堆区中的 Survivor 0 区、Survivor 1 区中的实例晋升到堆区中的老年代的速度）

为了解决这一问题，CMS GC 会使用 CMS GC 的后备垃圾回收器（即 Serial Old GC）执行 Full GC

4.3.4. G1 GC

4.3.4.1. G1 GC 概述

G1 GC 是 HotSpot JVM 在 Java9 及以后的默认的垃圾回收器，支持 Minor GC、Mixed GC、Full GC：

  1. G1 GC 使用多线程，采用并发垃圾回收策略，使用标记-复制算法执行 Minor GC
  2. G1 GC 使用多线程，采用并行垃圾回收策略、并发垃圾回收策略，使用标记-清除算法、标记-复制算法执行 Mixed GC（需要注意的是：G1 GC 使用标记-清除算法垃圾回收的是 Humongous 区）
  3. G1 GC 使用单线程或多线程（需要注意的是：看 HotSpot JVM 的 Java 版本），采用并发垃圾回收策略，使用标记-整理算法执行 FUll GC

在之前的垃圾回收器中，JVM 将堆区的内存空间划分新生代、老年代。但是在 G1 GC 中，JVM 将堆区的内存空间划分为多个内存空间相等的 Region

除此之外，G1 GC 还引入一种新的内存区域（即 Humongous 区）。多个连续的 Region 组成一个 Humongous 区，专门用于存储超大实例。如果实例需要的内存空间超过一个 Region 的 50% 的内存空间，JVM 就会为该实例分配 Humongous 区的内存空间（需要注意的是：我们通常把 Humongous 区看作是堆区中的老年代中的一部分）

那么问题来了：既然 G1 GC 执行垃圾回收是按照 Region 为单位，那么 G1 GC 该如何判断其他 Region 中的实例，是否引用 G1 GC 要垃圾回收的 Region 中的实例？

为了解决这一问题，G1 GC 引入了记忆集（Rset、Remembered Set） 。每个 Region 都维护了一个 RSet，专门用于记录引用该 Region 中的实例的其他 Region 中的实例

[!NOTE] 注意事项

  1. JVM 默认根据堆区的最大的内存空间，将堆区的内存空间划分为 2048 个内存空间相等的 Region，每个 Region 的内存空间为 1MB - 32MB（例如：堆区的最大内存空间为 32GB，那么每个 Region 的内存空间为 16MB。所以在刚运行 Java 程序，堆区的内存空间为 4GB 时，Region 只有 256 个）
    1. 需要注意的是：
      1. Region 的内存空间必须为 $2^n$ MB
      2. 我们可以通过配置 JVM 参数，手动指定 Region 的内存空间（需要注意的是：在这种情况下，Region 的数量不再固定为 2048 个）
  2. G1 GC 具备自适应调节的能力。我们只需要配置少量 JVM 参数，G1 GC 就会根据运行 Java 程序时的状况，自动调整部分 JVM 参数，从而达到贴近我们配置的 JVM 参数的目的：
    1. -XX:MaxGCPauseMillis
      1. 用于配置 Parallel Scavenge GC + Parallel Old GC 在执行垃圾回收的过程中，执行 STW 的被允许的最长时间（需要注意的是：对 G1 GC 执行 Full GC 不起作用）
4.3.4.2. G1 GC 相关流程
4.3.4.2.1. Minor GC 流程

当堆区中的 Eden 区的内存空间不足，JVM 不能直接为新的实例分配堆区中的 Eden 区的内存空间时，就会触发 G1 GC 执行 Minor GC（需要注意的是：在 G1 GC 中，Minor GC 被称之为 “Yong-Only Phase”）：

  1. G1 GC 首先会执行 STW
  2. 然后 G1 GC 会将所有堆区中的新生代的 Region 存储到回收集合（Cset、Collection Set）
  3. 接着 G1 GC 会使用多线程，采用并行垃圾回收策略，使用标记-复制算法执行 MINOR GC
  4. 在 G1 GC 执行完 Minor GC 后，G1 GC 会恢复运行被 “暂停” 的线程
4.3.4.2.2. Mixed GC 流程

当堆区中的老年代的内存空间的使用率达到阈值时，就会触发 G1 GC 执行 Mixed GC（需要注意的是：在 G1 GC 中，Mixed GC 被称之为 “Space-Reclamation Phase”）：

  1. 初始标记阶段（Initial Mark Phase）
    1. 在 G1 GC 进行初始标记阶段的过程中，G1 GC 使用多线程，采用并行垃圾回收策略，标记出 GC Roots 直接可达的实例（需要注意的是：不做深度遍历）
    2. 在 G1 GC 进行初始标记阶段的过程中，G1 GC 会执行 STW。但是由于 G1 GC 需要标记的实例的数量较少，所以 G1 GC 执行 STW 的时间较短
  2. 并发标记阶段（Concurrent Mark Phase）
    1. 在 G1 GC 进行并发标记阶段的过程中，G1 GC 使用多线程，采用并发垃圾回收策略，遍历整个实例图，标记出所有 GC Roots 可达的实例
    2. G1 GC 进行并发标记阶段会耗费较长时间。但是在 G1 GC 进行并发标记阶段的过程中，G1 GC 不会执行 STW
  3. 重新标记阶段（Remark Phase）
    1. 在 G1 GC 进行重新标记阶段的过程中，G1 GC 使用多线程，采用并行垃圾回收策略，修正在 G1 GC 进行并发标记阶段的过程中，由于其他的线程继续运行，而导致标记不准确的部分实例
    2. 在 G1 GC 进行重新标记阶段的过程中，G1 GC 会执行 STW（需要注意的是：在 G1 GC 进行重新标记阶段的过程中，执行 STW 的时间，会比在 G1 GC 进行初始标记阶段的过程中，执行 STW 的时间要长）
  4. 筛选清理阶段（Cleanup Phase）
    1. 在 G1 GC 进行筛选清理阶段的过程中，G1 GC 会将所有堆区中的新生代的 Region 存储代 Cset。然后 G1 GC 会排序所有 Region 的 “回收价值”，并根据我们配置的 JVM 参数，将部分高 “回收价值” 的堆区中的老年代的 Region 存储到 Cset
    2. 在 G1 GC 进行筛选清理阶段，G1 GC 会执行 STW
  5. 混合回收阶段（Mixed GC Phase）
    1. 在 G1 GC 进行混合回收阶段的过程中，G1 GC 使用多线程，采用并行垃圾回收策略，使用标记-复制算法执行 Mixed GC（需要注意的是：实际会垃圾回收所有堆区中的新生代的 Region 和部分高 “回收价值” 的堆区中的老年代的 Region ）
    2. 在 G1 GC 进行混合回收阶段的过程中，G1 GC 会执行 STW
4.3.4.2.3. Full GC 流程

虽然 G1 GC 的策略也是 “主动预防”。但是在运行 Java 程序的过程中，也可能出现一些极端情况。为了解决这一问题，G1 GC 会执行 Full GC：

  1. HotSpot JVM 在 Java 10 之前，G1 GC 使用单线程，采用串行垃圾回收策略，使用标记-整理算法执行 Full GC
  2. HotSpot JVM 在 Java10 及以后，G1 GC 使用多线程，采用并行垃圾回收策略，使用标记-整理算法执行 Full GC

触发 G1 GC 执行 Full GC 的条件相似于触发 Serial Old GC 执行 Full GC 的条件，只不过 G1 GC 并不存在传统意义上的 “空间担保分配”（因为它是基于 Region 的）。但是当堆区中的空闲的 Region 的数量不足，G1 GC 在执行 Minor GC、Major GC 的过程中，将 Region 中的实例复制到另一个 Region 失败时，就会触发 G1 GC 执行 Full GC（需要注意的是：这本质上也是一种 “空间担保分配”）

4.3.5. 对象的 finalization 机制

Object 类提供了 void finalize() 方法，若某个类对其进行了重写，那么在 GC 该对象之前，会先调用 finalize()，允许对象在销毁前执行一些自定义逻辑

然而，在调用 finalize() 的时候，可能导致对象 “复活”（即重新与引用链上的某个对象建立关联）。因此，JVM 在判断对象是否可回收时，通常需要经历两次标记过程：

  1. 第一次标记
    1. GC 通过可达性分析，发现某些对象与 GC Roots 之间已不存在引用链，判定它们为 “可回收候选对象”
  2. 第二次标记
    1. 如果对象没有重写 finalize() 方法，或其 finalize() 方法已经被调用过了
      1. 该对象会被直接判定为不可复活，继续进入回收流程
    2. 如果对象重写了 finalize() 方法，并且尚未被执行过
      1. GC 会将该对象插入到 F-Queue 队列
      2. 随后，一个由虚拟机自动创建的、低优先级的 Finalizer 线程会异步触发 finalize() 方法的执行
      3. 在下一个 GC 周期中，当进行新一轮的可达性分析时，GC 会再次检查那些曾经在 F-Queue 中、并且其 finalize() 方法已经被执行完毕的对象
        1. 如果该对象重新与引用链上的某个对象建立了联系，那么它将被移出即将回收的集合，实现 “复活”
        2. 如果该对象依然没有与引用链上的某个对象建立了联系，那么它就会被最终判定为垃圾，在下一次垃圾回收时被真正地回收

需要注意的是，从 Java9 开始，Object 类的 void finalize() 方法已经被正式弃用了

        2. 这是因为子类继承了父类的成员，需要确保父类已经就绪
      5. 使用 ClassLoader.loadClass() 时

        2. 这是更底层、更灵活的加载方式，通常在自定义类加载器时使用

  • 当虚拟机启动时，用户需要指定一个要执行的主类（包含 main() 方法的那个类），虚拟机会先加载并初始化这个主类。

    1. 这个对象作为方法区这个类的各种数据的访问入口

2.2. 验证阶段

验证阶段是连接阶段的第一个子阶段，在这一阶段，JVM 的主要任务主要包括：

  1. 文件格式验证
    1. 检查字节码是否符合 Class 文件格式的规范，是否以魔数 0xCAFEBABE 开头。
    2. 确保主版本号和副版本号在 JVM 支持的范围内。
    3. 检查文件结构，如常量池、字段表、方法表等是否符合规范。
  2. 元数据验证
    1. 这个类是否有父类
      1. 除了 java.lang.Object，所有类都应该有父类
    2. 这个类的父类是否是 final 的（final 类不允许被继承）
    3. 这个类是否实现了它所声明实现的所有接口的方法
    4. 类中的字段和方法是否与父类或接口的规范保持一致
    5. 确保类、字段和方法访问权限的正确性（例如，不能覆盖一个 final 方法）
  3. 字节码验证
    1. 控制流分析
      1. 确保方法体内的控制流是合法的，例如，不会出现跳转到方法体外的指令，或者指令序列是连贯的
    2. 类型检查
      1. 确保在操作数栈上的操作是合法的，例如，不会将一个整数类型的值赋值给一个引用类型的变量，或者对一个非对象类型进行方法调用
    3. 栈帧验证
      1. 验证在方法执行过程中，栈帧（Stack Frame）的操作是正确的，例如，操作数栈的深度不会溢出，局部变量表的索引是合法的
    4. 方法调用验证
      1. 确保方法调用时参数的数量和类型都与目标方法的签名匹配
    5. 异常处理验证
      1. 确保异常处理器（exception handler）是合法的，并且能够正确处理抛出的异常
  4. 符号引用验证
    1. 符号引用中描述的全限定名是否能找到对应的类、字段或方法。
    2. 符号引用中描述的类、字段或方法是否具有访问权限（例如，是否尝试访问一个 private 成员）
    3. 引用的字段或方法是否与被引用的类兼容（例如，对一个接口调用其未定义的方法）

[!NOTE] 注意事项

  1. 实际上，文件格式验证发生在加载阶段之前，而符号引用验证则是在解析阶段进行的。

[!NOTE] 注意事项

  1. JVM 验证的并不是 .class 字节码文件本身，而是将其加载并解析后生成的结构化类元信息，因此验证过程发生在加载过程之后
  2. 一个类被加载后，既会在方法区（元空间）中构建对应的运行时数据结构，同时还会在内存中生成一个 java.lang.Class 实例，该对象内部包含指向元信息的引用，我们可以通过其提供的方法访问和获取该类的详细信息。
  3. 类对象被垃圾回收是类的方法区中的被垃圾回收的条件之一

4. 分层编译的过程

分层编译是 HotSpot JVM 特有的概念，用于融合（Client 模式 && Server 模式）的特性

[!NOTE] 注意事项-1

  1. HotSpot JVM 在 Java8 及以后，默认启用分层编译

我们可以将分层编译的过程分为：

  1. Level 0 阶段
  2. Level 1 阶段
  3. Level 2 阶段
  4. Level 3 阶段
  5. Level 4 阶段

[!NOTE] 注意事项-2

  1. Level 0 阶段中的 Java 代码被 JVM 使用解释器解释执行（需要注意的是：其实是该 Java 代码被 javac 编译器编译后的 JVM 字节码指令被 JVM 使用解释器解释执行）
  2. Level 1 阶段 - Level 3 阶段中的 Java 代码被 JVM 使用 C1 编译器编译成 CPU 对应的机器码指令
  3. Level 4 阶段中的 Java 代码被 JVM 使用 C2 编译器编译成 CPU 对应的机器码指令

4.1. Level 0 阶段

在 Level 0 阶段中，JVM 会使用解释器解释执行该阶段中的 Java 代码。与此同时，JVM 会通过（方法调用计数器、回边计数器）来判断哪些（方法 || 方法中的循环体）是热点的 Java 代码：

  1. 方法调用计数器
    1. 用于存储方法被调用的次数
  2. 回边计数器
    1. 用于存储方法中的所有的循环体被调用的次数之和

当方法对应的（方法调用计数器 && 回边计数器）存储的值之和达到阈值时（需要注意的是：该阈值较低，可能为 2000 - 4000），JVM 就会判断该（方法 || 方法中的循环体）有点热度，需要晋升到 Level 1 阶段 - Level 3 阶段

然后 JVM 会比较该方法对应的回边计数器存储的值是否 ≥ OSR 阈值，判断究竟是将该方法晋升，还是将该方法中的循环体及该循环体往后的所有的 Java 代码晋升

接着 JVM 会根据（要晋升的 Java 代码的结构、C2 编译器的任务队列是否空闲），判断究竟将该 Java 代码晋升到 Level 1 阶段 - Level 3 阶段中的哪个阶段

[!NOTE] 注意事项

  1. 在 Java 中，每一个方法都对应一个（方法调用计数器 && 回边计数器）（需要注意的是：这两个计数器被存储在该方法对应的类的类元数据中的方法信息中）
  2. （<init> 方法、<clinit> 方法）也有可能成为热点的 Java 代码（需要注意的是：<init> 方法非常容易成为热点的 Java 代码，但是 <clinit> 方法几乎不可能成为热点的 Java 代码）
  3. 在 Java 中，其实并不存在不在方法中的 Java 代码（例如：你声明实例变量时的 int i = 5，其实也是在 <init> 方法中）
  4. 根据二八原则，可能只有 20% 的 Java 代码能够成为热点的 Java 代码，而 80% 的 Java 代码其实并不能成为热点的 Java 代码

4.2. Level 1 阶段

当 JVM 根据要晋升的 Java 代码的结构，判定该 Java 代码的优化空间极小，即便 C2 编译器介入也提升不了多少性能，反而会浪费算力资源时（例如：Getter 方法、Setter 方法），就会将要晋升的 Java 代码晋升到 Level 1 阶段

[!NOTE] 注意事项

  1. Level 1 阶段中的 Java 代码将会永远留在 Level 1 阶段，不会再晋升到 Level 4 阶段
  2. Level 1 阶段中的 Java 代码不涉及到任何的性能监控的代码（因为 Level 1 阶段中的 Java 代码不会再晋升到 Level 4 阶段，所以不需要进行性能监控）
  3. Level 1 阶段中的 Java 代码，在 Level 1 阶段 - Level 4 阶段中的所有的 Java 代码中，是执行最快的 Java 代码（因为不需要进行性能监控）

在 Level 1 阶段中，JVM 会使用 C1 编译器将该阶段中的 Java 代码编译成 CPU 对应的机器码指令，并对其进行一系列的性能优化：

  1. 方法内联
  2. 常量折叠
  3. 复写传播
  4. 冗余代码消除

[!NOTE] 注意事项

  1. C2 编译器也进行了这四个性能优化，但是 C2 编译器比 C1 编译器做的更激进、更彻底

4.2.1. 方法内联

方法内联是指：C2 编译器将被调用方调用的方法的 Java 代码，直接复制到调用方的 Java 代码中

[!NOTE] 注意事项

  1. 当 C1 编译器在对 Level 1 阶段 - Level 3 阶段中的 Java 代码进行方法内联时，不能处理多态（因为 C1 编译器并不知道，我们究竟要使用哪一个（接口的具体实现类 || 抽象类的具体子类））
  2. 但是当 C2 编译器在对 Level 4 阶段中的 Java 代码进行方法内联时，可以处理多态（因为 Level 4 阶段中的 Java 代码，已经在 Level 3 阶段中收集到了足够的信息，C2 编译器知道我们最常用的究竟是哪一个（接口的具体实现类 || 抽象类的具体子类），于是就假设我们还会用这个，将我们假设的类的方法的 Java 代码复制到调用方的 Java 代码中。需要注意的是：这是一个赌狗行为，如果在执行 Java 程序的过程中，出现了其他情况，也就是我们赌输了，那么这个 Java 代码将会重新回到 Level 0 阶段）

4.2.2. 常量折叠

[!NOTE] 注意事项

  1. 当 C1 编译器在对 Level 1 阶段 - Level 3 阶段中的 Java 代码进行常量折叠时，不能穿透控制流（即 （if-else if-else 语句 || switch-case 语句 || while 循环语句 || 或 for 循环语句）??对吗）
    1. 需要注意的是：
      1. 如果控制流是静态的，例如你 int a = 5，int b = 10，然后 if（a>b）这种是直接能穿透的
      2. 但是如果你控制流的条件取决于入参，而入参 C1 编译器是不确定的，所以就不能穿透控制流
  2. 但是当 C2 编译器在对 Level 4 阶段中的 Java 代码进行常量折叠时，可以穿透控制流（因为 Level 4 阶段中的 Java 代码，已经在 Level 3 阶段中收集到了足够的信息，C2 编译器知道我们最常用的一个分支是什么，于是就假设我们还会用这个，将这个分支进行常量折叠。需要注意的是：这是一个赌狗行为，如果在执行 Java 程序的过程中，出现了其他情况，也就是我们赌输了，那么这个 Java 代码将会重新回到 Level 0 阶段？？？在这里对吗？）

4.2.3. 复写传播

复写传播是指：如果我们把变量 x 赋值给了 y（需要注意的是：必须是 y = x），而且中间没有改动（需要注意的是：如果 （y || x）记录的是基本数据类型的数据，那么 y 不能再被赋值，x 不能再被赋值。如果是引用数据类型的数据，只要两者的内存地址不变，就都是可以的），那么后面凡是用到 y 的地方，C1 编译器都会直接把它替换成 x

4.2.4. 冗余代码消除

冗余代码消除是指：消除（在后文中无用 && 不可达路径中）的 Java 代码

[!NOTE] 注意事项

  1. 当 C1 编译器在对 Level 1 阶段 - Level 3 阶段中的 Java 代码进行冗余代码消除时，不敢消除创建对象的 Java 代码
  2. 但是 C2 编译器可以对 Level 4 阶段中的 Java 代码使用逃逸分析 + 标量替换，从而可能会消除一部分创建对象的 Java 代码（因为需要实例并没有逃逸）

4.3. Level 2 阶段

当 JVM 根据要晋升的 Java 代码的结构，判定该 Java 代码的优化空间极大，未来可以让 C2 编译器介入，但是 C2 编译器的任务队列已满时，就会将要晋升的 Java 代码晋升到 Level 2 阶段

在 Level 2 阶段中，JVM 同样会使用 C1 编译器将 Java 代码编译成 CPU 对应的机器码指令，并对其进行一系列的性能优化

C1 编译器在 Level 2 阶段对 Java 代码进行的性能优化，与 C1 编译器在 Level 1 阶段对 Java 代码进行的性能优化一致。但是 Level 2 阶段的 Java 代码涉及到少量的性能监控的代码（需要注意的是：只是通过（方法调用器 && 回边计数器）统计（方法被调用的次数 && 方法中的所有循环体被调用的次数之和））

当 C2 编译器的任务队列缓解时，JVM 就会将 Level 2 阶段中的 Java 代码晋升到 Level 3 阶段

[!NOTE] 注意事项

  1. 由于 Level 2 中的 Java 代码涉及到少量的性能监控的代码，所以执行速度比 Level 1 阶段中的 Java 代码要慢，但是比 Level 3 阶段中的 Java 代码要快

4.4. Level 3 阶段

当 JVM 根据晋升的 Java 代码的结构，判定该 Java 代码的优化空间极大，未来可以让 C2 编译器介入，而且 C2 编译器的任务队列空闲时，就会将要晋升的 Java 代码晋升到 Level 3 阶段（需要注意的是：不要忘了还有 Level 2 阶段中的 Java 代码可以晋升到 Level 3 阶段）

在 Level 3 阶段中，JVM 同样会使用 C1 编译器将 Java 代码编译成 CPU 对应的机器码指令，并对其进行一系列的性能优化

C1 编译器在 Level 3 阶段对 Java 代码进行的性能优化，与 C1 编译器在 Level 1 阶段对 Java 代码进行的性能优化一致，但是 Level 3 阶段的 Java 代码涉及到全量的性能监控的代码

当方法对应的（方法调用计数器 && 回边计数器）存储的值之和达到阈值时，JVM 就会将 Level 3 阶段中的 Java 代码晋升到 Level 4 阶段

[!NOTE] 注意事项

  1. Level 3 阶段中的 Java 代码是 Level 1 阶段 - Level 4 阶段中的所有的 Java 代码中，执行最慢的 Java 代码（因为涉及到全量的性能监控的代码）
  2. 虽然 Level 3 阶段中的 Java 代码执行的最慢，但是能最快收集到 C2 编译器需要的信息

分支预测、类型分析、MDO

4.5. Level 4 阶段

在 Level 4 阶段中，JVM 会使用 C2 编译器将该阶段中的 Java 代码编译成 CPU 对应的机器码指令，并对其进行一系列的性能优化：

  1. 逃逸分析
  2. 循环展开
  3. 方法内联
  4. 常量折叠
  5. 复写传播
  6. 循环向量化
  7. 冗余代码消除
  8. 数组边界检查消除

[!NOTE] 注意事项

  1. 虽然 C1 编译器也在做（方法内联 && 常量折叠 && 复写传播 && 冗余代码消除），但是 C2 编译器做的更彻底，更激进
  2. 这里我们只着重学习一下逃逸分析

4.5.1. 逃逸分析

4.5.1.1. 逃逸分析概述

逃逸分析是指：JIT 在编译 Java 代码时，会分析一个实例在被创建后，是否被外部（即其他线程、调用对象的构造方法的那个方法之外的其他方法）引用。如果 JIT 发现一个实例并没有逃逸，就可以对这个实例进行深度的底层优化，从而极大提升性能。

[!NOTE] 注意事项

  1. 从工程实践的角度来看：能使用方法参数和局部变量，就使用方法参数和局部变量
4.5.1.2. 逃逸分析常用分类

JIT 编译器会将对象的逃逸分析的分类主要包括：

  1. 全局逃逸
    1. 一个对象的引用“逃逸”出了方法或者线程。比如，对象的引用被赋值给了类的静态变量、成员变量，或者作为方法的返回值。这是最高级别的逃逸
    2. 这种状态下，几乎无法进行任何优化，对象必须在堆上分配
  2. 参数逃逸
    1. 一个对象的引用在方法内创建，但被作为参数传递给了另一个方法
    2. 这是一种部分逃逸。编译器可以进行一些分析，如果被调用的方法没有对这个对象进行“全局逃逸”的操作，那么仍然有优化的空间
  3. 不逃逸
    1. 一个对象的引用完全限制在方法内部，没有以任何形式传递到方法外部
    2. 这是最理想的状态，JIT 编译器可以对这种对象进行大刀阔斧的优化
