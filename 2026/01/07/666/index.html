四、JVM 相关流程

1. 类编译的过程（🐂）

类编译的过程（前端编译）是指：javac 编译器把 .java 文件编译成 .class 文件（即把 Java 代码编译成 JVM 字节码指令）

在类编译的过程中，javac 编译器会把语法糖转换为该语法糖对应的逻辑，并对 Java 代码进行严格的语义检查（例如：检查某类是否有父类），然后再把 Java 代码编译成 JVM 字节码指令

[!NOTE] 注意事项

  1. 我们称类编译的过程为前端编译，称分层编译的过程为后端编译
  2. 详见笔记：Java 语法糖

2. 类加载的过程

2.2. 验证阶段（🐂）

在验证阶段中，JVM 会进行：

  1. 格式检查
    1. 检查 .class 文件中的开头是否为 OxCAFEBABE
    2. 检查 .class 文件中的（主版本号 && 副版本号）是否在该 JVM 支持的范围中
    3. 检查 .class 文件中的结构是否（完整 && 符合 JVM 规范）（需要注意的是：JVM 会着重检查常量池）
  2. 语义检查
  3. 字节码验证
  4. 符号引用验证

[!NOTE] 注意事项

  1. 其实 JVM 是交叉进行（加载阶段 && 验证阶段）的。JVM 会先进行（验证阶段 ➔ 格式检查），再进行加载阶段，然后再进行（验证阶段 ➔ 其他的（检查 && 验证））

准备阶段

2.3. 解析阶段

在解析阶段中，JVM 会将（类的类元数据区 ➔ 运行时常量池 ➔ 符号引用）解析成直接引用（即将符号形式的文本信息转换为可被 JVM 直接定位的内存地址）

JVM 规范规定的符号引用主要包括：

  1. CONSTANT_Class_info
    1. 在解析阶段之前，存储类的全限定名
    2. 在解析阶段之后，存储一个指向该类的类元数据区的内存地址
  2. CONSTANT_Fieldref_info
    1. 在解析阶段之前，存储（类的全限定名 && 字段的（名词 && 描述符））
    2. 在解析阶段之后，存储一个指向（该类的类元数据区 ➔ 该字段对于的字段信息）的内存地址
  3. CONSTANT_Methodref_info
    1. 在解析阶段之前，存储（类的全限定名 && 方法的（名字 && 描述符））
    2. 在解析阶段之后，存储一个指向（该类的类元数据区 ➔ 该方法对于的方法信息）的内存地址
  4. CONSTANT_InterfaceMethodref_info
    1. CONSTANT_InterfaceMethodref_info 于 CONSTANT_Methodref_info 相似，只不过 CONSTANT_InterfaceMethodref_info 针对的是接口方法
  5. CONSTANT_MethodType_info
  6. CONSTANT_MethodHandle_info
  7. CONSTANT_InvokeDynamic_info

[!NOTE] 注意事项

  1. （CONSTANT_MethodType_info && CONSTANT_MethodHandle_info && CONSTANT_InvokeDynamic_info）是在 Java 7 中，Java 语言的设计团队为了引入 invokedynamic JVM 字节码指令，修改了 JVM 规范，将这三个符号引用引入了进来

[!NOTE] 注意事项
2. 不同 JVM 对解析阶段的处理方式可能存在差异：
1. 一些 JVM 实现会在解析阶段，遍历运行时常量池中的所有符号引用，并尝试将其立即解析为直接引用
2. 也有一些 JVM 实现会在解析阶段，选择只解析其中关键或高频使用的符号引用，比如 java.lang.Object 这类核心类
3. 我们的 HotSpot JVM 采用的是延迟解析的策略
1. 在解析阶段，HotSpot 会遍历并识别出所有类和接口的符号引用（CONSTANT_Class_info）
2. 对于像 java.lang.Object 这样的基础核心类，JVM 会尝试解析它
3. 而对于大多数的符号引用，通常采用按需解析，即在首次使用时才触发解析
4. 例如在初始化阶段执行 clinit 方法时，或在运行 init、main 方法，甚至其他方法的字节码时，如果某个符号引用还未被解析，就会在那一刻解析
3. 所以符号引用将在 解析阶段 或者 程序运行时 被解析为直接引用，并不一定在解析阶段就被解析成直接引用

2.3.1. CONSTANT_Class_info

在解析阶段之前，CONSTANT_Class_info 存储了一个 CONSTANT_Utf8_info 的常量池索引，而该 CONSTANT_Utf8_info 又存储了类的全限定名。我们可以简单理解为：在解析阶段之前，CONSTANT_Class_info 存储了类的全限定名

在解析阶段之后，CONSTANT_Class_info 存储了一个指向该类的类元数据区的内存地址

2.3.2. CONSTANT_Utf8_info

CONSTANT_Utf8_info 用于存储在 Java 程序中的各种文本信息（例如：类名）

在解析阶段之后，CONSTANT_Utf8_info 并不会被 JVM 解析成直接引用（因为 CONSTANT_Utf8_info 本质上只是为 Java 程序提供各种文本信息的符号引用

2.3.3. CONSTANT_NameAndType_info

在解析阶段之前，CONSTANT_NameAndType_info 存储了两个 CONSTANT_Utf8_info 的常量池索引，而这两个 CONSTANT_Utf8_info 又分别存储了（方法 || 字段）的（名字 && 描述符）。我们可以简单理解为：在解析阶段之前，CONSTANT_NameAndType_info 存储了（方法 || 字段）的（名字 && 描述符）

在解析阶段之后，CONSTANT_NameAndType_info 并不会被 JVM 解析成直接引用（因为 CONSTANT_NameAndType_info 本质上只是为（CONSTANT_Fieldref_info && CONSTANT_Methodref_info && CONSTANT_InterfaceMethodref_info）提供描述信息的符号引用）

2.3.4. CONSTANT_Fieldref_info

在解析阶段之前，CONSTANT_Fieldref_info 存储了一个（CONSTANT_Class_info && CONSTANT_NameAndType_info）的常量池索引，而该 CONSTANT_Class_info 存储了类的全限定名，该 CONSTANT_NameAndType_info 存储了字段的（名字 && 描述符）

在解析阶段之后，CONSTANT_Fieldref_info 存储了一个指向（类的类元数据区 ➔ 该字段对应的字段信息）的内存地址

2.3.5. CONSTANT_Methodref_info

在解析阶段之前，CONSTANT_Fieldref_info 存储了一个（CONSTANT_Class_info && CONSTANT_NameAndType_info）的常量池索引，而 CONSTANT_Class_info 存储类的全限定名，CONSTANT_NameAndType_info 存储方法的（名字 && 描述符）

在解析阶段之后，CONSTANT_Fieldref_info 存储了一个指向（类的类元数据区 ➔ 该方法对应的方法信息）的内存地址

2.3.6. CONSTANT_String_info

在解析阶段之前，CONSTANT_String_info 存储了一个 CONSTANT_Utf8_info 的常量池索引，而该 CONSTANT_Utf8_info 又存储了一个根据字面量构建 String 实例的构造方法的字面量

在解析阶段之后，CONSTANT_String_info 存储了一个指向使用该构造方法构造的 String 实例的内存地址

[!NOTE] 注意事项

  1. 对于静态常量的解析阶段，就算说，很多东西嘛，顺序肯定是不一样的
  3. 在解析阶段之后，CONSTANT_String_info 存储的是一个堆区的内存地址，而其他的存储的是方法区的内存地址

2.3.7. CONSTANT_InterfaceMethodref_info

CONSTANT_InterfaceMethodref_info 与 CONSTANT_Methodref_info 类似，只不过 CONSTANT_InterfaceMethodref_info 指向的是接口方法的描述信息

2.3.8. CONSTANT_Integer_info

CONSTANT_Integer_info 用于存储整数常量。虽然它的名字里虽然带有 Integer，但与 Integer 对象本身没有任何关系，只是单纯地用来存储一个整数常量。至于 Integer 对象，它和其他对象一样，依然遵循普通对象的处理方式。

我们知道，小整数常量（-1 - 5）会通过 iconst_n 字节码指令直接压入操作数栈，而不需要像前面提到的那样，在执行字节码时先去运行时常量池查找、必要时再解析，然后再取值。简单来说，这些常量可以直接获得，而无需额外解析或查找步骤。同样地，取值范围在 -128 - 127 的整数会使用 bipush n 指令，范围在 -32768 ~ 32767 的整数会使用 sipush n 指令

然而，如果超出这个范围，就必须通过常量池来获取。但需要注意的是，CONSTANT_Integer_info 常量并不像其他符号引用那样需要解析，它本身就直接存储了数值，因此字节码在执行时只需从常量池中取出即可，无需再进行解析过程

与此类似的几种主要包括：

  1. CONSTANT_Integer_info
    1. 整数常量
  2. CONSTANT_Float_info
    1. 浮点数常量
  3. CONSTANT_Long_info
    1. 长整数常量
  4. CONSTANT_Double_info
    1. 双精度浮点数常量

这些符号引用都无需经过解析，因为它们本身就直接包含了数值

[!NOTE] 注意实习

  1. boolean、byte、short、char 都是使用 iconst_n、bipush n、sipush n 压入操作数栈的
    1. boolean
      1. 0 表示 false
      2. 1 表示 true
    2. char
      1. 不要忘记，我们存储 char 是根据 Unicode 字符集存储的，所以 ‘a’ ➔ 97

初始化阶段

需要注意的是，类的加载过程并不一定会进入初始化阶段，例如下面几种情况就不会触发初始化，这被称为 “被动引用”：

  1. 通过子类引用父类的静态字段
    1. 例如，System.out.println(SubClass.parentStaticField);
    2. 这只会初始化父类，而不会初始化子类
  2. 通过数组定义来引用类
    1. 例如，MyClass［］ mcArray = new MyClass［10］;
    2. 这并不会触发 MyClass 类的初始化，它只是创建了一个 MyClass 类型的数组对象
  3. 引用类的常量
    1. 例如，System.out.println(MyClass.MY_CONSTANT);
    2. 如果常量在编译阶段就被放入了调用类的常量池中（例如被 final 修饰的基本类型或字符串常量），那么本质上并没有直接引用定义常量的类，因此不会触发该类的初始化

但是 JVM 规范中规定了，在以下几种情况中，必须进行初始化：

  1. 遇到 new、getstatic、putstatic 或 invokestatic 字节码指令时

    1. 使用 new 关键字实例化对象
    2. 访问类的静态字段时
      1. 被 final 修饰的静态字段，如果其值已在编译期确定，则不会触发初始化
    3. 访问类的静态方法时

  2. 使用 java.lang.reflect 包的方法对类型进行反射调用时

    1. 例如，使用 Class.forName(“com.example.MyClass”) 或 MyClass.class.newInstance()

  3. 初始化一个类的子类时

    1. 如果一个类还没有初始化，它的任何一个子类在初始化之前，父类都必须先初始化。

  4. JVM 启动时指定的主类

    1. 包含 main() 方法的那个类，JVM 在启动时会先初始化这个类

  5. JDK 7 引入的动态语言支持

    1. 当一个 java.lang.invoke.MethodHandle 实例的解析结果是 REF_getStatic、REF_putStatic、REF_invokeStatic、REF_newInvokeSpecial 四种类型的方法句柄时，并且这个方法句柄对应的类没有初始化时，需要先触发其初始化

  6. 遇到 new、getstatic、putstatic 或 invokestatic 这四条字节码指令时，如果类没有进行过初始化，则需要先触发其初始化。这四条指令分别对应：

    • 使用 new 关键字实例化对象。
    • 读取一个类的静态字段（被 final 修饰，已在编译期把结果放入常量池的静态字段除外）。
    • 设置一个类的静态字段。
    • 调用一个类的静态方法。

  7. 使用 java.lang.reflect 包的方法对类进行反射调用时，如果类没有进行过初始化，则需要先触发其初始化。

  8. 当初始化一个类时，如果其父类还没有被初始化，则需要先初始化其父类。

  9. 当虚拟机启动时，用户需要指定一个要执行的主类（包含 main() 方法的那个类），虚拟机会先加载并初始化这个主类。

  10. 当使用 JDK 7 的动态语言支持时，如果一个 java.lang.invoke.MethodHandle 实例最后的解析结果是 REF_getStatic、REF_putStatic、REF_invokeStatic、REF_newInvokeSpecial 四种类型的方法句柄，并且这个方法句柄对应的类没有进行过初始化，则需要先触发其初始化。

  11. 当一个接口中定义了 JDK 8 新加入的默认方法（default 方法）时，如果有这个接口的实现类发生了初始化，那该接口要在其之前被初始化。

在初始化阶段，JVM 会进行：

  1. 执行 <clinit> 方法
    1. 赋予真实值给该类的静态变量（术语是：显式初始化）
    2. 执行静态代码块（即 static {...}）
