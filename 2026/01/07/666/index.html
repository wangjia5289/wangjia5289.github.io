三、JVM 架构体系

1. 运行时数据区

1.1. 运行时数据区组成部分（🐂）

1.2. 本地方法栈（🐂）

本地方法栈（Native Method Stacks）与虚拟机栈相似，只不过本地方法栈针对的是 native 方法

[!NOTE] 注意事项

  1. 在 HotSpot JVM 中，（虚拟机栈 && 本地方法栈）被合并为同一个栈
  2. 根据 JVM 规范：如果我们不打算让 JVM 支持 native 方法，那么就可以不实现本地方法栈

1.3. 程序计数器（🐂）

1.3.1. 程序计数器概述（🐂）

程序计数器（PC 寄存器、Program Counter Register）用于存储线程要去执行的下一条 JVM 字节码指令在（线程正在执行的方法的 JVM 字节码指令数组 ➔ 偏移量）

[!NOTE] 注意事项

  1. 如果线程正在执行的方法是 native 方法，那么程序计数器存储的是 undefined（因为 native 方法本质上是离开了 JVM 的控制范围）
  2. JVM 既不会垃圾回收程序计数器，也不会因为程序计数器而抛出（OutOfMemoryError 错误、StackOverflowError 错误）
  3. 执行引擎负责读取并执行 JVM 字节码指令
    1. 当执行引擎读取 JVM 字节码指令时，首先会将 JVM 字节码指令在（线程正在执行的方法的字节码数组 ➔ 偏移量）转换为真实的内存地址，然后再去读取 JVM 字节码指令
    2. 在执行引擎读取到 JVM 字节码指令之后，程序计数器就会存储线程要去执行的下一条 JVM 字节码指令在（线程正在执行的方法的 JVM 字节码指令数组 ➔ 偏移量），然后执行引擎才开始执行 JVM 字节码指令（需要注意的是：这是一个非常重要的细节）

1.3.2. 程序计数器空间大小（🐂）

程序计数器占用的内存空间非常小：

  1. 在 32 bit 的 JVM 中，程序计数器仅占用 4 byte 的内存空间
  2. 在 64 bit 的 JVM 中，程序计数器仅占用 8 byte 的内存空间

1.3.3. 程序计数器组成部分（🐂）

1.3.4. 程序计数器多线程并发安全性（🐂）

程序计数器是线程私有的，每个线程都拥有独立的程序计数器，所以不存在多线程并发安全问题

1.4. 虚拟机栈

1.4.1. 虚拟机栈概述（🐂）

虚拟机栈（JVM Stacks）与本地方法栈相似，只不过虚拟机栈针对的是 Java 方法

[!NOTE] 注意事项

  1. JVM 既会垃圾回收虚拟机栈，也会因为虚拟机栈而抛出（OutOfMemoryError 错误、StackOverflowError 错误）：
    1. OutOfMemoryError 错误
      1. OutOfMemoryError: unable to create new native thread
        1. 如果虚拟机栈可以动态的扩展内存空间，那么当 JVM 为虚拟机栈分配内存空间时，如果 JVM 的内存空间不足，就会抛出 OutOfMemoryError 错误（需要注意的是：HotSpot JVM 并不支持虚拟机栈动态的扩展内存空间）
        2. 如果 JVM 正在申请操作系统创建本地线程，那么当 JVM 为本地线程对应的（程序计数器、虚拟机栈）分配内存空间时，如果 JVM 的内存空间不足，就会抛出 OutOfMemoryError 错误
    2. StackOverflowError 错误
      1. 当 JVM 为线程要去执行的方法分配栈帧的内存空间时，如果虚拟机栈的内存空间不足，那么就会抛出 StackOverflowError 错误

1.4.2. 虚拟机栈组成部分

1.4.2.1. 局部变量表
1.4.2.1.1. 局部变量表概述

局部变量表用于存储（方法 ➔ （方法参数、局部变量））

[!NOTE] 注意事项

  1. 局部变量表能够存储基本数据类型的数据、引用数据类型的数据（需要注意的是：实际上局部变量表存储的是引用数据类型的数据的引用）、返回地址（即在 Java5 之前的 returnAddress 类型）
  2. （静态变量、静态常量、实例变量、实例常量）在不同的阶段会被 JVM 赋予默认值。但是方法参数、局部变量并不会被 JVM 赋予默认值。因此我们在使用（方法参数、局部变量）之前，必须对其进行手动赋值，否则在类编译的过程中就会出错（因为 javac 编译器会对 Java 代码进行严格的语义检查）
1.4.2.1.2. 局部变量表底层实现

局部变量表的底层是：一个一维数组

局部变量表使用的一维数组的基本存储单元是变量槽（Slot）

  1. 在 32 bit 的 JVM 中，一个 Slot 占用 4 byte 的内存空间
  2. 在 64 bit 的 JVM 中，一个 Slot 占用 8 byte 的内存空间

[!NOTE] 注意事项

  1. 当线程调用某类的构造方法时，指向被构造的实例的内存地址的引用（即 this），就被存储在局部变量表的数组索引为 0 的数组位置中（需要注意的是：这是 Java 中的一个语法糖，也是我们为什么能在构造方法中使用 this 的原因）
  2. 当线程调用某实例的实例方法时，指向该方法所属实例的内存地址的引用（即 this），就被存储在局部变量表的数组索引为 0 的数组位置中（需要注意的是：这是 Java 中的一个语法糖，也是为什么我们能在实例方法中使用 this 的原因）
  3. 当线程调用某类的静态方法时，根本就不存在 this，导致我们不能在静态方法中使用 this
1.4.2.1.3. 局部变量表空间大小

在类的编译过程中，JVM 中的解释器就已经确定了局部变量表的内存空间，在运行 Java 程序的过程中无法更改局部变量表的内存空间：

  1. 占用 32 bit 及以下的内存空间的数据（例如 byte、short、int、float、char、boolean 类型的数据、引用数据类型的数据、返回地址）占用一个 Slot
  2. 占用 64 bit 的内存空间的数据（例如 long、double 类型的数据）占用两个 Slot

而在 32 bit 的 JVM 中，一个 Slot 占用 4 byte 的内存空间。在 64 bit 的 JVM 中，一个 Slot 占用 8 byte 的内存空间。所以我们只需知道方法中的方法参数、局部变量占用几个 Slot，就可以知道该方法对应的栈帧中的局部变量表的内存空间

[!NOTE] 注意事项

  1. 在 JVM 中，byte 类型的数据、short 类型的数据、char 类型的数据、boolean 类型的数据被 JVM 当作 int 类型的数据（因为 int 类型数据的范围涵盖它们，而且在 JVM 中其实并没有专门处理它们的 JVM 字节码指令（需要注意的是：如果它们被存储在堆区（例如是实例的实例变量），那么它们该占用多少内存空间，就占用多少内存空间））
  2. Slot 是可以复用的（例如：在局部变量 A 的生命周期结束后，如果我们又声明了新的局部变量 B，那么 B 可能复用 A 的 Slot，从而达到节省内存空间的目的）。以如下代码为例，局部变量 b 会复用局部变量 a 的 Slot。因此整个 main 方法只需要占用两个 Slot，所以在 32 bit 的 JVM 中，main 方法对应的栈帧中的局部变量表占用 8 byte 的内存空间。在 64 bit 的 JVM 中，main 方法对应的栈帧中的局部变量表占用 16 byte 的内存空间
1.4.2.2. 方法的返回地址

当线程调用方法时，首先会把程序计数器中的内容存储在（方法对应的栈帧 ➔ 方法的返回地址）中，然后再执行方法

如果线程在执行方法的过程中未抛出异常，那么当执行到以下 JVM 字节码指令时，就会将方法的返回地址中的内存存储在程序计数器中：

1. 返回 （byte、short、int、char、boolean）类型的数据
2. lreturn
1. 返回 long 类型的数据
3. freturn
1. 返回 float 类型的数据
4. dreturn
1. 返回 double 类型的数据
5. areturn
1. 返回引用数据类型的数据
6. return
1. 仅返回（需要注意的是：仅返回适用于无返回值的方法）

但是如果线程在执行方法的过程中抛出了异常，那么线程就会去查该方法对应的异常处理器表（需要注意的是：异常处理器表被存储在方法区）：

  1. 如果异常处理器表中有匹配的异常
    1. 线程就跳转到 Targer 处，然后执行 Target 处的 Java 代码（即线程执行 catch 块。需要注意的是：不会再回到抛出异常的位置，继续执行原来的代码）
  2. 如果异常处理器表中没有匹配的异常
    1. 线程就会弹出该方法对应的栈帧，并继续向外抛出异常（即抛出异常给调用方）

[!NOTE] 注意事项

  1. 方法退出分为方法正常退出（发生异常、未发生异常）、方法异常退出（需要注意的是：方法正常退出（发生异常）是在 catch 块中，捕获并处理了异常，并且没有抛出新的异常）
  2. 当我们捕获并处理异常时，抛出了一个新的异常，那么线程同样会再去查异常处理器表。但是如果我们没有使用嵌套 catch 块，就很可能无法捕获并处理这个新的异常

1.4.2.4. 操作数栈

操作数栈是 JVM 执行引擎的一个工作区，用于保存计算过程中的中间结果，同时作为变量的临时存储空间。JVM 会根据方法中的字节码指令，向操作数栈中写入数据，或从中提取数据并使用后，结果再写入操作数栈

与局部变量表一样，操作数栈的最大深度在编译阶段就已经确定，运行时无法更改（栈顶缓存并不会影响最大深度）

  1. 32 位及以内的数据类型（reference、returnAddress、byte、short、char、int、float、boolean）占用一个栈单位深度
  2. 64 位的数据类型（long、double）占用两个栈单位深度

[!NOTE] 注意事项

  1. byte、short、char 在压入操作栈前，会被自动转换为 int 类型，对应关系主要包括：
    1. byte ➔ int
    2. short ➔ int
    3. char ➔ int
    4. int ➔ int
    5. boolean ➔ boolean
    6. long ➔ long
    7. float ➔ float
    8. double ➔ double
  2. 我们常说 Java 虚拟机的解释引擎是基于栈的执行引擎，这里的 “栈” 指的就是操作数栈

1.5. 堆区

1.5.1. 堆区概述（🐂）

[!NOTE] 注意事项

  1. JVM 为几乎所有的实例在堆区分配内存空间（因为需要考虑 C2 编译器进行的逃逸分析）
  2. JVM 既会使用垃圾回收器垃圾回收堆区，也会因为堆区而抛出 OutOfMemoryError 错误：
    1. OutOfMemoryError: Java heap space
      1. 堆区的内存空间不足

1.5.2. 堆区空间大小

堆区具有初始的内存空间（需要注意的是：由 -Xms512m 决定），并且支持动态（扩展 || 缩减）内存空间，但是受到最大的内存空间的限制（需要注意的是：由 -Xmx2048m 决定）

1.5.3. 堆区组成部分（🐂）

1.5.3.1. TLAB 区

想象这样一个问题：JVM 为了快速为实例在堆区分配内存空间，采用了 “指针碰撞” 技术（即使用一个指针指向 Eden 区已经使用的内存空间的边界，当 JVM 为实例分配内存空间时，只需要检查剩余的内存空间是否足够，然后按需顺序偏移指针，即可完成为实例在堆区分配内存空间）

但是我们知道 “堆区是线程共享的内存空间”，在多线程并发场景下，就会出现多线程并发对堆区进行修改的情况。传统的解决方案是加锁，但这会成为一个显著的性能瓶颈

为了解决这一问题，JVM 提供了 TLAB。TLAB 将实例在线程共享的堆区被 JVM 分配内存空间，变成了实例在线程私有的 TLAB 被 JVM 分配内存空间

1.5.4. 堆区多线程并发安全性（🐂）

堆区是线程共享的，所以存在多线程并发安全问题。但是（堆区 ➔ TLAB 区）是线程私有的，每个线程都拥有独立的 TLAB 区，所以不存在多线程并发安全问题

1.5.5. JVM 为实例在堆区分配内存空间相关流程

[!NOTE] 注意事项

  1. 在 JVM 执行完 GC 之后，JVM 会根据堆区的内存空间的空闲比例，动态扩展、动态缩减堆区的内存空间

1.5.6. JVM 为实例在堆区中的 TLAB 区分配内存空间相关流程

[!NOTE] 注意事项

  1. 当线程申请新 TLAB 时，JVM 是通过 CAS 保证多线程并发安全的（需要注意的是：旧 TLAB 的剩余空间会使用一个 int 类型的数组填充，这部分空间会被 GC）
  2. JVM 为了快速为实例在 TLAB 分配内存空间，采用了 “指针碰撞” 技术（因为 TLAB 是线程私有的，因此不存在多线程并发场景下的安全问题）

1.6. 方法区

1.6.1. 方法区概述

[!NOTE] 注意事项

  1. JVM 既会使用垃圾回收器垃圾回收方法区，也会因为方法区而抛出 OutOfMemoryError 错误：
    1. OutOfMemoryError: Metaspace
      1. 元空间的内存空间不足
    2. OutOfMemoryError: PermGen space
      1. 永久代的内存空间不足

方法区是用来存储已经被 JVM 加载的各种类的元数据信息、类的静态变量，以及即时编译器（JIT）编译后生成的代码缓存

方法区在运行时具有初始空间大小，并支持动态扩展，但受最大空间限制，一旦超出该限制，可能会抛 OOM

JDK1.7 之前的永久代、JDK1.8 之后的元空间，都是对 JVM 规范中的方法区的实现，其最大的区别在于：元空间不再使用 JVM 虚拟机设置的堆内存，而是改为使用本地内存，因此更少发生 OOM

  1. 永久代（PermGen） 是 JVM 内存的一部分，默认值往往较小（比如几十 MB），而且很容易超过上限，从而导致 OOM
  2. 元空间（Metaspace） 使用的是 本地内存（Native memory），大小可以远远超过 JVM 堆的限制，甚至可以动态扩展至整个物理内存的上限
  1. 每一个实例都会存储一个指向其对应的类的类元数据区的内存地址（需要注意的是：存储的位置是对象头中的 Klass Word）
  2. 在 Java8 之前，静态变量、静态常量存储在方法区。但是在 Java8 及之后，静态变量、静态常量存储在堆区（因为静态变量、静态常量存储在类对象中，而类对象一定是在堆区被 JVM 分配内存空间的（因为 JIT 不会对类对象进行逃逸分析，因为无需也不应该对类对象进行逃逸分析，而且也无法对类对象进行逃逸分析），所以静态变量、静态常量存储在堆区）
  3. 在 Java8 之前的永久代，在 Java8 及以后的元空间，都是对 JVM 规范中的方法区的实现。永久代与元空间的区别在于：
    1. 永久代的内存空间是堆区的内存空间的部分，内存空间往往较小（例如几十 MB），很容易导致 JVM 抛出 OutOfMemoryError 错误
    2. 而元空间的内存空间是本地内存空间的部分，内存空间甚至可以动态扩展至整个物理内存空间的上限

1.6.2. 虚拟机栈多线程并发安全性（🐂）

虚拟机栈是线程私有的，每个线程都拥有独立的虚拟机栈，所以不存在多线程并发安全问题

2. 解释器

2.1. 解释器概述

解释器（Interpreter）用于解释执行 JVM 字节码指令

2.2. 解释器相关分类

  1. 字节码解释器
  2. 模板解释器

2.2.1. 字节码解释器

2.2.1.1. 字节码解释器概述

字节码解释器（C++ 解释器）是指：JVM 的设计团队使用 C++ 语言编写的解释器

2.2.1.2. 字节码解释器底层原理

字节码解释器本质上就是一个 while 循环，该 while 循环中嵌套了一个巨大的 switch-case 语句

当字节码解释器解释执行 JVM 字节码指令时，其实就是该解释器进行一次 while 循环，然后根据 switch-case 语句调用一次该 JVM 字节码指令对应的 C++ 方法

2.2.1.3. 字节码解释器利弊分析
2.2.1.3.1. 优势
  1. 移植性高
    1. 由于字节码解释器是 JVM 的设计团队使用 C++ 语言编写的，所以只要 C++ 语言适配了的系统（即该系统拥有兼容的 C++ 编译器、C++ 运行时环境），我们就可以在该系统上使用字节码解释器
2.2.1.3.2. 劣势
    1. 我们知道：当字节码解释器解释执行 JVM 字节码指令时，其实就是该解释器进行一次 while 循环，然后根据 switch-case 语句调用一次该 JVM 字节码指令对应的 C++ 方法
    2. 所以，每当字节码解释器执行一条 JVM 字节码指令时，该解释器就会进行一次（while 循环 && switch-case 语句的判断 && C++ 方法的执行 && 控制流的跳转）
    3. 但是，我们一行简单的 Java 代码（例如：a = b + c）就会被 javac 编译器编译成多条 JVM 字节码指令
    4. 所以，JVM 哪怕只是执行一行简单的 Java 代码，该解释器都会重复的执行上述的流程
    5. 然而，只有当该解释器执行 C++ 方法时，该解释器才是在真正的执行实际的业务
    6. 所以，当 JVM 使用字节码解释器解释执行 JVM 字节码指令时，该解释器会造成极大的算力的浪费（术语是：管理成本远远大于生产成本）
    1. 由于字节码解释器是 JVM 的设计团队使用 C++ 语言编写的解释器
    2. 所以，字节码解释器其实是被 C++ 编译器编译成 CPU 对应的机器码指令的
    3. 我们知道：如果我们使用 C++ 语言编写一个程序，该编译器是可以在编译该程序时深度的优化该程序的（例如：该编译器可以将部分的数据存储在寄存器中。因为该编译器拥有该程序的全局视角，清楚的知道（方法的调用链 && 变量的生命周期 && ...））
    4. 但是，当该编译器在编译该解释器时，该编译器拥有的该解释器的全局视角会被该解释器的特性显著的削弱（因为该解释器本质上就是一个 while 循环 + switch-case 语句。在该 while 循环外，字节码解释器中的 C++ 代码并不存在可供 C++ 编译器分析的复杂的方法的调用链。而在该 while 循环中，尽管 switch-case 语句中的各个分支中包含了大量的 C++ 方法，但是由于该编译器并不知道在 JVM 运行该解释器的过程中，该解释器要按照什么顺序来执行 C++ 方法（因为该顺序要根据 JVM 字节码指令的顺序，而该 JVM 字节码指令的顺序又要根据我们编写 Java 代码））
    5. 所以，该解释器不知道我们下一步要去执行哪一个 C++ 方法，更不知道该解释器正在执行的 C++ 方法中，哪些数据会被之后要去执行的 C++ 方法继续使用
    6. 所以 C++ 编译器不得不把该解释器正在执行的 C++ 方法中的数据写回内存空间中（需要注意的是：该数据其实是被该解释器写回到该解释器正在执行的 C++ 方法所属的 Java 方法对应的栈帧中的操作数栈的内存空间中）
    7. 所以，当 JVM 使用字节码解释器解释执行 JVM 字节码指令时，会产生大量的读写内存空间的行为
    1. 现代的 CPU 极其的依赖于复杂的（指令流水线 && 分支预测）技术来提升执行的速度
    2. 但是，字节码解释器中的 switch-case 语句对于 CPU 来说，其实就是一个间接分支
    3. 所以，CPU 的分支预测其实是很难预测 switch-case 语句要去跳转的目标地址的
    4. 可是，一旦 CPU 的分支预测失败，CPU 就必须清空已经加载到流水线中的指令，并重新加载正确的指令
2.2.1.4. 字节码解释器相关产品
  1. Zero 解释器

2.2.2. 模板解释器

2.2.2.1. 模板解释器概述

模板解释器是指：JVM 的设计团队使用汇编语言编写的解释器

2.2.2.2. 模板解释器底层原理

模板解释器的底层原理是指：在我们启动 JVM 的过程中，模板解释器会为每一条 JVM 字节码指令生成 CPU 对应的机器码指令

2.2.2.3. 模板解释器相关流程

当模板解释器解释执行 JVM 字节码指令时，其实就是该解释器读取当前要去执行的 JVM 字节码指令，然后直接跳转到该 JVM 字节码指令被该解释器编译后的 CPU 对应的机器码指令并执行

在该解释器执行完该 CPU 对应的机器码指令之后，该解释器会再次读取下一条要去执行的 JVM 字节码指令，然后再跳转到该 JVM 字节码指令被该解释器编译后的 CPU 对应的机器码指令并执行

[!NOTE] 注意事项

  1. 由于模板解释器是在 CPU 对应的机器码指令的层面直接进行控制流的跳转，所以该解释器进行跳转的效率非常高

3. JIT 编译器

3.1. JIT 编译器概述

JIT 编译器（JIT Compiler）用于将热点的 Java 代码被 javac 编译器编译后的 JVM 字节码指令编译成 CPU 对应的机器码指令（需要注意的是：在后文中，我们简称为热点的 Java 代码），并在该代码的基础上进行了一系列的性能的优化

[!NOTE] 注意事项

  1. 代码缓存（Code Cache）负责存储 JIT 编译器编译热点的 Java 代码后生成的 CPU 对应的机器码指令

1. 类编译的过程（🐂）

类编译的过程（前端编译）是指：javac 编译器把 .java 文件编译成 .class 文件（即 javac 编译器把 Java 代码编译成 JVM 字节码指令）
